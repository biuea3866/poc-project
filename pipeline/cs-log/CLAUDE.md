# CS 문의 대응 파이프라인

고객 또는 팀원의 문의/버그 리포트를 분석하여 원인과 해결 방안을 도출한다.

## 공통 가이드 참조
> 아래 공통 규칙을 따릅니다. 파이프라인 특화 규칙은 이 문서에서 정의합니다.

- [문체/용어 규칙](../common/output-style.md)

## 실행 원칙

- **추측 금지**: 반드시 관련 코드를 읽고 로그/데이터 근거로 분석한다.
- **결론 선행**: 원인과 조치 방안을 먼저, 상세 분석은 뒤에.
- **코드 참조**: `파일경로#메서드명` 또는 `파일경로:라인번호` 형식.
- **고객 응대 초안 포함**: 분석 완료 후 고객에게 보낼 답변 초안을 작성한다.

## 입력 유형

| 유형 | 예시 |
|------|------|
| 고객 문의 텍스트 | "카드 변경했는데 미결제 메일이 계속 와요" |
| 에러 로그 | payment-server 결제 실패 로그 |
| Jira/Slack 문의 | 티켓 번호 또는 메시지 내용 |

## 출력 경로

```
.analysis/cs-log/
├── CLAUDE.md                                    # 이 파일
├── README.md                                    # CS 기록 인덱스
└── {날짜}_{고객사}_{요약}/
    └── analysis.md                              # 분석 기록
```

## 출력 어조 <!-- 상세 규칙은 공통 가이드 참조 -->

- 해요체(~합니다). 핵심부터. 한 문장에 하나의 정보.
- 고객 응대 초안은 정중하고 명확하게.
- 내부 분석은 구체적으로 (코드 참조, SQL 쿼리 포함).

## Step 1: 문의 분류 및 키워드 추출

문의 내용을 읽고 아래를 파악한다:

### 문의 유형

| 유형 | 설명 | 긴급도 |
|------|------|--------|
| **결제/플랜** | 결제 실패, 카드 변경, 플랜 만료, 환불 | 높음 |
| **계정/권한** | 로그인 불가, 권한 오류, 멤버 관리 | 높음 |
| **기능 버그** | 특정 기능이 예상대로 동작하지 않음 | 중간 |
| **데이터 확인** | 특정 데이터의 상태/값 확인 요청 | 중간 |
| **기능 질문** | 사용법, 설정 방법 문의 | 낮음 |
| **장애 대응** | 서비스 전체/부분 장애 | 긴급 |

### 핵심 정보 추출

- 고객사명, 담당자 이메일
- 영향받는 도메인 (결제, 공고, 지원자, 평가, 면접 등)
- 에러 메시지 (있는 경우)
- 재현 조건 (workspaceId, openingId 등)

## Step 3: 취합 및 분석 기록 작성

팀원 4명의 결과를 취합하여 분석 기록을 작성한다.

### 분석 기록 구조

```markdown
# CS: {고객사} — {문의 요약}

> 접수일: {날짜}
> 상태: 조사 중 / DB 확인 필요 / 해결 완료
> 심각도: Critical / Major / Minor

---

## 1. 문의 내용
- **고객사**: {이름}
- **담당자**: {이메일}
- **내용**: {문의 원문}

## 2. 로그 분석
{관련 로그가 있으면 정리. 없으면 "로그 미제공"}

## 3. 원인 분석
{코드 추적 결과 기반 원인. 시나리오별로 분류.}

## 4. 확인 필요 사항
{DB 조회 쿼리, 로그 검색 키워드 등 추가 확인이 필요한 것.}

## 5. 결제 흐름 코드 경로
{관련 코드 경로 표로 정리.}

## 6. 대응 방안
{상태별 조치 방법 + 고객 응대 초안.}
```
