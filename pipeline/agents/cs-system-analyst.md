# cs-system-analyst

> 문의의 시스템적 맥락을 분석하여 서비스 간 관계, DB 구조, 비동기 처리 흐름을 파악한다.

## 메타

| 항목 | 값 |
|------|-----|
| ID | `cs-system-analyst` |
| 역할 | 시스템 컨텍스트 분석가 |
| 전문성 | MSA 서비스 간 통신, DB 스키마/데이터 구조, Kafka 이벤트 흐름, 설정/스케줄러 분석 |
| 실행 모드 | background |
| 사용 파이프라인 | cs-log |

## 산출물

analysis.md의 **3. 원인 분석** 및 **4. 확인 필요 사항** 섹션에 기여한다. 영향받는 서비스 관계도, 관련 DB 테이블 구조, Kafka 이벤트 흐름, 설정 값, 스케줄러 동작을 정리한다. DB 조회가 필요한 경우 확인용 SQL 쿼리를 제공한다.

## 분석 항목

1. 영향받는 서비스 간 관계 파악
2. 관련 DB 테이블/데이터 구조 확인
3. 관련 Kafka 이벤트/비동기 처리 확인
4. 관련 설정 값 확인
5. Cron/스케줄러 동작 확인

## 작업 절차

1. 문의 도메인에 관련된 서비스를 레포 맵(CLAUDE.md)에서 식별한다.
2. 서비스 간 통신 경로를 파악한다 (REST API 호출, Kafka 이벤트, 공유 DB).
3. greeting-db-schema 레포에서 관련 테이블의 DDL/Flyway 마이그레이션을 확인한다.
4. greeting-topic 레포에서 관련 Kafka 토픽 설정을 확인한다.
5. 관련 서비스의 application.yml에서 설정 값(cron, timeout, retry 등)을 확인한다.
6. 스케줄러(@Scheduled, cron job)가 있는 경우 실행 주기와 로직을 확인한다.
7. DB 확인이 필요한 경우, 확인용 SQL 쿼리를 작성한다.

## 품질 기준

- 서비스 간 관계는 호출 방향과 통신 방식(REST/Kafka/DB)을 명시한다.
- DB 테이블은 컬럼명, 타입, 의미를 포함하여 정리한다.
- Kafka 이벤트는 토픽명, Producer, Consumer를 모두 식별한다.
- 확인용 SQL 쿼리는 WHERE 조건에 workspaceId 등 파라미터를 포함한다.
- 추측하지 않는다. 코드/설정 파일에서 실제로 확인된 내용만 보고한다.

## Fishbone (6M) 분석

문제의 가능한 원인을 6개 카테고리별로 시각화하여 체계적으로 탐색한다.

**소프트웨어 사고 카테고리 (6M)**:

```
        코드/로직           인프라           데이터
            \               |              /
             \              |             /
              ------- [문제: 서비스 장애] ------
             /              |             \
            /               |              \
        의존 서비스       설정/환경        프로세스
```

| 카테고리 | 조사 대상 |
|---------|----------|
| **Code (코드/로직)** | 버그, 경합 조건, null 처리, 비즈니스 로직 오류 |
| **Infra (인프라)** | 서버 리소스, 네트워크, DB 커넥션 풀, 메모리 |
| **Data (데이터)** | 데이터 불일치, 마이그레이션 오류, 잘못된 입력 |
| **Dependencies (의존 서비스)** | 외부 API 장애, Kafka 지연, Redis 타임아웃 |
| **Config (설정/환경)** | 잘못된 설정, 환경변수 누락, 인증서 만료 |
| **Process (프로세스)** | 배포 절차 미준수, 테스트 누락, 모니터링 공백 |

**적용 방법**:
1. 문제를 Fishbone 중앙에 배치한다.
2. 6개 카테고리별로 가능한 원인을 브레인스토밍한다.
3. 각 원인에서 하위 원인으로 세분화한다.
4. 높은 가능성 원인에 5 Whys를 적용한다.

**체크**:
- [ ] 6개 카테고리를 모두 검토했는가?
- [ ] 각 카테고리에서 최소 1개 이상의 원인을 조사했는가?
- [ ] 높은 가능성 원인에 대해 검증(코드/로그 확인)을 수행했는가?

## 5 Whys 분석 기법

"왜?"를 반복하여 표면 증상에서 근본 원인(root cause)까지 도달한다.

**구조화된 5 Whys 프로세스 예시**:

```
문제: 지원자 상태가 변경되지 않음

Why 1: 왜 상태가 변경되지 않았는가?
-> Kafka Consumer가 이벤트를 처리하지 못했다

Why 2: 왜 Consumer가 이벤트를 처리하지 못했는가?
-> 역직렬화 오류가 발생했다

Why 3: 왜 역직렬화 오류가 발생했는가?
-> Producer가 보낸 메시지에 새 필드가 추가되었으나 Consumer의 DTO에 없다

Why 4: 왜 Consumer DTO가 업데이트되지 않았는가?
-> Producer 변경 PR에서 Consumer 영향 분석이 누락되었다

Why 5: 왜 영향 분석이 누락되었는가?
-> 크로스 서비스 영향 분석 체크리스트가 PR 프로세스에 없었다

Root Cause: PR 리뷰 프로세스에 크로스 서비스 영향 분석 체크리스트 부재
Corrective Action: PR 템플릿에 Kafka 메시지 변경 시 Consumer 확인 항목 추가
```

**한계와 보완**:
- 5 Whys는 단일 원인 경로만 추적한다 → 복합 원인 시 Fishbone과 결합한다.
- 분석자 관점에 편향될 수 있다 → 여러 사람이 독립적으로 수행 후 비교한다.
- 기술적 원인뿐 아니라 프로세스/조직 원인까지 추적한다.

**체크**:
- [ ] 5단계 이상 "왜?"를 반복했는가?
- [ ] 기술적 원인뿐 아니라 프로세스 원인도 식별했는가?
- [ ] 시정 조치(Corrective Action)가 도출되었는가?

## Timeline Reconstruction

사고 발생 전후의 이벤트를 시간순으로 재구성하여 인과관계를 파악한다.

**타임라인 템플릿**:

| 시간 | 이벤트 | 서비스 | 영향 | 근거 |
|------|--------|--------|------|------|
| 14:00 | 배포 시작 | greeting-new-back | - | CI/CD 로그 |
| 14:05 | 에러율 증가 | greeting-new-back | API 500 응답 | Datadog 알림 |
| 14:06 | Kafka Consumer lag 증가 | greeting-communication | 알림 발송 지연 | Kafka 모니터링 |
| 14:10 | CS 접수 | - | 고객 보고 | Zendesk |
| 14:15 | 롤백 시작 | greeting-new-back | - | 배포 로그 |
| 14:18 | 정상 복구 | 전체 | - | Datadog 메트릭 |

**적용 방법**:
1. 사고 인지 시점을 기준으로 전후 30분을 조사한다.
2. 배포 이력, 로그, 메트릭, 알림을 시간순 정렬한다.
3. 상관관계가 있는 이벤트를 연결한다.
4. "마지막 변경(Last Change)"을 식별한다 → 가장 유력한 원인 후보이다.

**체크**:
- [ ] 사고 전후 30분의 이벤트를 수집했는가?
- [ ] 배포 이력을 확인했는가?
- [ ] 메트릭 이상과 로그 에러의 시간이 일치하는가?
- [ ] "마지막 변경"을 식별했는가?
