## 분석 항목

1. **FE 변경 범위**: 영향받는 화면, 컴포넌트, API 호출 변경을 식별한다.
2. **BE 변경 범위**: 영향받는 서비스, API, DB 스키마, 이벤트를 식별한다.
3. **인프라 변경**: K8s, Gateway, Ingress, 환경변수, 신규 서비스 배포 필요 여부를 확인한다.
4. **디자인 산출물 필요 여부**: Figma/디자인 작업이 선행되어야 하는 항목을 식별한다.
5. **QA 시나리오**: 테스트해야 할 핵심 시나리오를 개략적으로 도출한다.
6. **타 팀 협업 필요 여부**: FE, 인프라, 디자인, PM 등 타 팀과 협업이 필요한 항목을 정리한다.

### 3점 PERT 추정

각 작업 항목의 공수를 낙관/기대/비관 3가지 시나리오로 추정하여 불확실성을 반영한다.

**공식:**

```
PERT 기대값  = (O + 4M + P) / 6
표준편차(σ)  = (P - O) / 6
신뢰구간 95% = 기대값 +- 2σ
```

- **O** (Optimistic): 최선 시나리오. 모든 것이 순조로울 때.
- **M** (Most Likely): 가장 현실적인 시나리오. 과거 경험 기반.
- **P** (Pessimistic): 최악 시나리오. 예상치 못한 장애물 발생 시.

**적용 예시:**

| 기능 | O(일) | M(일) | P(일) | PERT(일) | σ | 95% 범위 |
|------|-------|-------|-------|---------|---|---------|
| 지원자 상태 전이 API | 2 | 3 | 7 | 3.5 | 0.83 | 1.8~5.2 |
| Kafka Consumer 구현 | 3 | 5 | 12 | 5.8 | 1.50 | 2.8~8.8 |
| OpenSearch 인덱싱 | 5 | 8 | 15 | 8.7 | 1.67 | 5.3~12.0 |

**적용 방법:**
1. PRD의 각 기능을 작업 항목으로 분해한다 (WBS)
2. 각 항목에 O, M, P 세 가지 추정값을 산정한다
3. PERT 공식으로 기대값과 표준편차를 계산한다
4. 전체 합산 시 Root-Sum-Square로 총 불확실성을 계산한다

### COSMIC 기능 점수

기능의 규모를 데이터 이동(Entry/Exit/Read/Write) 횟수로 객관적 정량화한다.

| 유형 | 설명 | 매핑 |
|------|------|------|
| **Entry (E)** | 외부에서 시스템으로 데이터 진입 | API Request Body, Kafka Message 수신 |
| **Exit (X)** | 시스템에서 외부로 데이터 전달 | API Response Body, Kafka Message 발행, 알림 발송 |
| **Read (R)** | 영속 스토어에서 데이터 읽기 | DB SELECT, Redis GET, S3 GET |
| **Write (W)** | 영속 스토어에 데이터 쓰기 | DB INSERT/UPDATE/DELETE, Redis SET |

**규모 산정 절차:**
1. PRD 기능을 Functional Process 단위로 분해한다 (= 하나의 트리거에 대한 반응)
2. 각 Functional Process에서 E/X/R/W 이동 횟수를 카운트한다
3. 총 CFP (COSMIC Function Point) = 모든 이동의 합계
4. 과거 프로젝트의 CFP당 소요시간을 참고하여 공수를 추정한다

**예시:**
```
기능: "지원자 상태 변경"
- Entry: API Request (지원자 ID, 목표 상태) → 1 CFP
- Read: 지원자 현재 상태 조회 → 1 CFP
- Read: 상태 전이 규칙 확인 → 1 CFP
- Write: 지원자 상태 업데이트 → 1 CFP
- Write: 상태 변경 이력 기록 → 1 CFP
- Exit: API Response (변경 결과) → 1 CFP
- Exit: Kafka 이벤트 발행 (상태 변경) → 1 CFP
- Exit: 알림 발송 (이메일/알림톡) → 1 CFP
총: 8 CFP
```

### WBS 5단계 분해

PRD 전체를 5단계로 분해하여 100% 커버리지를 보장한다. 100% Rule: WBS는 모든 작업을 100% 포함해야 한다.

| 레벨 | 대상 | 크기 기준 |
|------|------|----------|
| 1 | PRD 전체 | Epic |
| 2 | 주요 기능 영역 | Feature |
| 3 | 개별 기능 | User Story |
| 4 | 작업 항목 | Task (1~3일) |
| 5 | 세부 작업 | Sub-task (0.5~1일) |

**Greeting 플랫폼 WBS 체크리스트:**
- [ ] DB 스키마 변경 (Flyway 마이그레이션)
- [ ] Entity / Repository 구현
- [ ] OutputPort / Adapter 구현
- [ ] Service / Facade 구현
- [ ] Controller + Swagger 문서
- [ ] Kafka Producer / Consumer
- [ ] 단위 테스트 (Kotest BehaviorSpec)
- [ ] 통합 테스트 (TestContainers)
- [ ] API Gateway 라우팅 설정
- [ ] 환경 설정 (application.yml, Terraform)
- [ ] 코드 리뷰 + detekt 통과
- [ ] QA 지원 (테스트 시나리오 제공)

### 스토리 포인트 교정 앵커 (Greeting 기준)

팀 내 일관된 추정을 위해 Greeting 프로젝트 기준 작업을 앵커로 설정한다.

| SP | 기준 작업 (Greeting 기준) | 복잡도 |
|----|------------------------|--------|
| 1 | 단순 필드 추가 (Entity + Migration) | 변경 파일 1~2개 |
| 2 | 단순 CRUD API 1개 (Controller~Repository) | 변경 파일 3~5개 |
| 3 | 비즈니스 로직이 있는 API (상태 전이 등) | 변경 파일 5~8개 |
| 5 | 새로운 도메인 + 여러 API + 테스트 | 변경 파일 10~15개 |
| 8 | 서비스 간 연동 (Kafka + API) | 변경 서비스 2개 이상 |
| 13 | 대규모 기능 (분할 필요 신호) | Epic급, 반드시 분할 |

## 작업 절차

1. PRD 기능별로 영향받는 파트(FE/BE/인프라/디자인)를 매핑한다.
2. BE 변경 범위를 서비스/API/DB/이벤트 단위로 구체화한다.
3. FE 변경 범위를 화면/컴포넌트/API 호출 단위로 구체화한다.
4. 인프라 변경이 필요한 항목(배포, Gateway, 환경변수 등)을 정리한다.
5. 디자인 선행 작업이 필요한 화면/컴포넌트를 식별한다.
6. 핵심 QA 시나리오를 Happy/Unhappy 기준으로 도출한다.
7. 파트별 변경 범위를 표로 정리하고, 타 팀 협업 필요 사항을 명시한다.

## 품질 기준

- 변경 범위가 파트별(FE/BE/인프라)로 구체적이어야 한다.
- "여러 곳" 같은 추상적 표현 대신 "3개 서비스, 5개 API" 식으로 정량적이어야 한다.
- QA 시나리오가 PRD 기능을 빠짐없이 커버해야 한다.
- 타 팀 협업 항목에 "누구에게 무엇을 요청하는지"가 명확해야 한다.
- 벤치마킹 후 소급 업데이트 시 공수가 재산정되어야 한다.
