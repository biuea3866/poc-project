## 분석 항목

1. **현재 데이터 구조 파악**: 이관 대상 데이터의 현재 스키마, 볼륨, 관계를 확인한다 (코드 탐색 필수).
2. **마이그레이션 계획**: 대량 데이터 이관 전략(배치 사이즈, 청크, 병렬 처리)을 수립한다.
3. **하위 호환성**: API 응답 구조, Kafka 이벤트 스키마, 배포 순서의 호환성을 분석한다.
4. **롤백 시나리오**: 마이그레이션 실패 시 롤백 절차와 데이터 정합성 복원 방안을 정의한다.
5. **Feature Flag 활용**: 단계별 전환을 위한 Feature Flag 전략을 설계한다.

### Strangler Fig 4단계

레거시 시스템을 점진적으로 교체하는 4단계 프로세스이다.

```
Phase 1: Identify (식별)
  → 마이그레이션 대상 기능/데이터 식별
  → 의존성 그래프 작성

Phase 2: Transform (변환)
  → 신규 서비스에 대상 기능 구현
  → 이중 실행(Shadow Mode)으로 결과 비교

Phase 3: Coexist (공존)
  → Intercepting Facade(API Gateway)로 트래픽 분배
  → Feature Flag로 신규/레거시 전환 제어
  → 동일 Kafka 토픽을 별도 Consumer Group으로 구독

Phase 4: Eliminate (제거)
  → 레거시 기능 비활성화
  → 모니터링 기간 후 레거시 코드/인프라 제거
```

**Greeting 플랫폼 적용 (FeatureFlag 기반):**
1. 신규 서비스: 새 Consumer Group으로 동일 Kafka 토픽을 구독한다
2. FeatureFlag ON → 신규 서비스 활성화, 레거시 비활성화
3. 문제 발생 시 → FeatureFlag OFF (배포 없이 롤백)
4. 안정화 후 → 레거시 코드를 제거한다

### Dual Write vs CDC 비교

데이터 동기화 전략을 선택할 때 아래 비교 기준을 활용한다. **일반적으로 CDC를 권장한다.** Dual-Write는 원자성을 보장할 수 없어 데이터 불일치 위험이 높다.

| 항목 | Dual-Write | CDC (Change Data Capture) |
|------|-----------|--------------------------|
| **원리** | 애플리케이션이 두 저장소에 동시 쓰기 | DB 트랜잭션 로그를 캡처하여 변경 전파 |
| **일관성** | 두 번째 쓰기 실패 시 불일치 | 트랜잭션 로그 기반 보장 |
| **성능 영향** | 쓰기 지연 증가 | 비동기, 원본 DB 영향 최소 |
| **코드 침투** | 비즈니스 코드 수정 필요 | 코드 수정 불필요 |
| **도구** | 직접 구현 | Debezium, AWS DMS |
| **권장 상황** | 간단한 1:1 동기화 | 대규모/복잡한 마이그레이션 |

### Transactional Outbox 패턴 (Debezium)

비즈니스 데이터 변경과 이벤트 발행의 원자성을 보장하는 패턴이다.

```
Service                              Kafka
  |                                    |
  +-- BEGIN TRANSACTION                |
  |  +-- 1. Business Table INSERT/UPDATE
  |  +-- 2. Outbox Table INSERT        |
  +-- COMMIT                           |
  |                                    |
  |  [Debezium CDC Connector]          |
  |  +-- 3. Outbox 변경 감지           |
  |  +-- 4. Kafka 메시지 발행 -------->|
```

**핵심 원리:**
- 비즈니스 데이터 변경과 이벤트 발행을 같은 DB 트랜잭션에서 처리한다
- Debezium이 Outbox 테이블의 변경을 감지하여 Kafka로 전달한다
- 분산 트랜잭션 문제를 로컬 트랜잭션으로 변환한다

**적용 시 고려사항:**
- Outbox 테이블에 데이터 이중 저장 (스토리지 비용)
- Outbox 테이블 정리(cleanup) 정책 필요
- PostgreSQL의 경우 Logical Decoding으로 Outbox 없이 직접 CDC 가능

### Expand-Contract 패턴 (무중단 스키마 변경)

DB 스키마를 무중단으로 변경하는 3단계 패턴이다.

```
Phase 1: Expand (확장)
  → 새 컬럼/테이블 추가 (nullable 또는 default 값)
  → 기존 코드는 변경 없이 동작

Phase 2: Migrate (마이그레이션)
  → 기존 데이터를 새 컬럼/테이블로 복사
  → 새 코드가 새 컬럼/테이블 사용 시작

Phase 3: Contract (축소)
  → 기존 컬럼/테이블 제거
  → 최소 1 릴리스 간격을 두고 제거
```

**마이그레이션 방식 비교:**

| 방식 | 다운타임 | 복잡도 | 데이터 규모 | 롤백 |
|------|---------|--------|-----------|------|
| **Big Bang** | 있음 | 낮음 | 소규모 | 백업 복원 |
| **Shadow Migration** | 없음 | 높음 | 대규모 | 신규 무시 |
| **CDC + Dual Read** | 없음 | 중간 | 대규모 | 읽기 전환 |
| **Expand-Contract** | 없음 | 중간 | 중규모 | 이전 릴리스 |

### 마이그레이션 리스크 평가 체크리스트

마이그레이션 계획 수립 시 아래 리스크를 사전 평가하고 대응 전략을 수립한다.

| 리스크 | 평가 항목 | 대응 전략 |
|--------|---------|----------|
| **데이터 손실** | 마이그레이션 중 데이터 누락 가능성 | CDC로 실시간 동기화, 건수 검증 스크립트 |
| **데이터 불일치** | 소스/타겟 간 데이터 차이 | 체크섬 비교, 샘플링 검증, Reconciliation Job |
| **성능 저하** | 마이그레이션 중 소스 DB 부하 | CDC는 비동기라 영향 최소, 배치 마이그레이션은 야간 실행 |
| **롤백 불가** | 마이그레이션 후 되돌리기 어려움 | 역방향 CDC 설정, 스냅샷 백업 |
| **긴 마이그레이션 시간** | 대량 데이터 이동 소요 시간 | 초기 벌크 복사 + CDC로 델타 동기화 |
| **스키마 호환성** | 소스/타겟 스키마 불일치 | 변환 매핑 문서, 데이터 타입 호환성 검증 |

**사전 준비 체크리스트:**
- [ ] 마이그레이션 대상 테이블/컬렉션 목록 확정
- [ ] 데이터 규모 파악 (행 수, 용량)
- [ ] 소스/타겟 스키마 매핑 문서 작성
- [ ] 롤백 계획 수립

**실행 체크리스트:**
- [ ] 초기 벌크 마이그레이션 실행
- [ ] CDC로 실시간 변경 동기화 시작
- [ ] 데이터 정합성 검증 (건수, 체크섬)
- [ ] 읽기 트래픽 전환 (Feature Flag)
- [ ] 쓰기 트래픽 전환 (Feature Flag)

**사후 검증 체크리스트:**
- [ ] 데이터 정합성 최종 검증
- [ ] 성능 모니터링 (Datadog)
- [ ] 레거시 동기화 중단
- [ ] 레거시 리소스 정리

## 작업 절차

1. 이관 대상 데이터의 현재 스키마를 greeting-db-schema에서 확인한다.
2. 데이터 볼륨(예상 건수, 크기)을 파악한다.
3. 이관 방식을 결정한다 (온라인/오프라인, Dual Write, Shadow Read).
4. 대량 데이터 처리 전략(배치 사이즈, 청크, 병렬 처리)을 수립한다.
5. API/Kafka/배포 순서의 하위 호환성을 분석한다.
6. 롤백 절차를 단계별로 정의한다.
7. Feature Flag를 활용한 단계별 전환 전략을 설계한다.
8. 마이그레이션 전/후 검증 방법(건수 비교, 샘플 검증)을 정의한다.

## 품질 기준

- 데이터 구조가 코드/스키마 기반으로 파악되어야 한다 (추측 금지).
- 데이터 볼륨 예측이 포함되어야 한다.
- 롤백 시나리오가 단계별로 구체적이어야 한다.
- Feature Flag 전략이 포함되어야 한다.
- 마이그레이션 전/후 검증 방법이 정의되어야 한다.
- Dual Write, Shadow Consumer 등 안전한 전환 패턴이 고려되어야 한다.
