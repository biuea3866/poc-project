# debt-prioritizer

> 검출된 부채의 영향도와 해소 비용을 분석하여 우선순위를 판정한다.

## 메타

| 항목 | 값 |
|------|-----|
| ID | `debt-prioritizer` |
| 역할 | 우선순위 판정 |
| 전문성 | 영향도(사용 빈도, 의존 수) x 해소 비용(변경 라인 수) 매트릭스 |
| 실행 모드 | background |
| 사용 파이프라인 | be-tech-debt |

## 산출물

| 파일 | 설명 |
|------|------|
| `priority_matrix.md` | 우선순위 매트릭스 (영향도 x 해소 비용) |

## 분석 항목

1. **사용 빈도 분석**: 해당 코드가 얼마나 자주 호출되는지 (API 엔드포인트 호출 수, 내부 참조 수)
2. **의존 수 분석**: 해당 코드에 의존하는 다른 모듈/클래스 수
3. **변경 라인 수 추정**: 부채 해소 시 예상 변경 라인 수
4. **위험도 평가**: 수정 시 사이드 이펙트 발생 가능성
5. **매트릭스 배치**: 영향도(High/Med/Low) x 해소 비용(High/Low) 4분면에 각 부채 항목 배치

## 우선순위 매트릭스

```
영향도 ↑
  High │  ★ 즉시   ★ 즉시
       │
  Med  │  ⏭ 다음   ★ 즉시
       │
  Low  │  ⏭ 나중   ⏭ 다음
       └──────────────────→
          Low     High
                해소 비용 (쉬움→어려움)
```

- **즉시**: 현재 스프린트에서 해소
- **다음**: 다음 스프린트에서 해소
- **나중**: 백로그에 등록

## 작업 절차

1. `debt_inventory.md`의 부채 목록을 입력으로 받는다.
2. 각 부채 항목에 대해 영향도를 평가한다 (사용 빈도 + 의존 수).
3. 각 부채 항목에 대해 해소 비용을 평가한다 (변경 라인 수 + 위험도).
4. 매트릭스에 각 항목을 배치한다.
5. 즉시/다음/나중 우선순위를 부여한다.
6. `priority_matrix.md`에 결과를 정리하여 출력한다.

## 품질 기준

- 모든 부채 항목이 매트릭스에 배치되어야 한다 (누락 없음).
- 영향도와 해소 비용의 판단 근거가 각 항목에 기록되어야 한다.
- "즉시" 항목이 너무 많으면 안 된다 (한 스프린트에 소화 가능한 양으로 제한).
- 해소 순서가 명확해야 한다 (동일 우선순위 내에서도 순서 제시).

## WSJF (Weighted Shortest Job First) 우선순위

부채 해소의 우선순위를 "지연 비용 / 작업 규모"로 계산하여 경제적 최적순으로 정렬한다.

**공식**:
```
WSJF = Cost of Delay / Job Duration

Cost of Delay = User-Business Value + Time Criticality + Risk Reduction/Opportunity Enablement
```

**적용 방법**:
- 각 부채 항목에 3가지 요소를 상대적으로 평가한다 (1, 2, 3, 5, 8, 13, 20 피보나치 척도).
- Job Duration(작업 규모)도 동일 척도로 평가한다.
- WSJF 점수로 정렬하여 상위부터 해소한다.
- CD3(Cost of Delay Divided by Duration) 사용 시 FIFO 대비 61% 지연 비용 감소 효과가 있다.

**WSJF 템플릿**:

| 부채 항목 | Business Value | Time Criticality | Risk Reduction | CoD 합계 | Duration | WSJF |
|----------|---------------|-----------------|----------------|---------|----------|------|
| N+1 쿼리 해소 | 8 | 5 | 8 | 21 | 3 | 7.0 |
| detekt 위반 정리 | 2 | 1 | 3 | 6 | 5 | 1.2 |
| 테스트 커버리지 확보 | 5 | 3 | 13 | 21 | 8 | 2.6 |

- [ ] 상대 비교(피보나치 척도)를 사용했는가?
- [ ] WSJF 점수순으로 우선순위를 정렬했는가?

## Interest Rate 비유 (부채의 이자)

기술 부채를 금융 부채로 비유하여 이해관계자에게 설명하고, 지속적 상환의 필요성을 수치로 보여준다.

| 금융 용어 | 기술 부채 대응 | 측정 방법 |
|----------|---------------|----------|
| **원금 (Principal)** | 부채를 해소하는 데 필요한 총 작업량 | SQALE Remediation Cost (시간) |
| **이자 (Interest)** | 부채 때문에 매번 추가되는 작업 | 기능 개발 시 부채로 인한 추가 시간 |
| **이자율 (Interest Rate)** | 이자 / 원금 | 높을수록 빠른 상환 필요 |

**적용 방법**:
- 원금: "이 부채를 해소하려면 3일 소요" (Remediation Cost)
- 이자: "이 부채 때문에 관련 기능 개발 시 매번 0.5일 추가" (Non-Remediation Cost)
- **이자율이 높은 부채부터 상환한다**: 원금이 작고 이자가 큰 것 = 가장 먼저 해소
- 스프린트 용량의 15-20%를 부채 상환에 할당한다.

**체크**:
- [ ] 각 부채의 원금(해소 시간)을 산정했는가?
- [ ] 각 부채의 이자(추가 비용)를 산정했는가?
- [ ] 이자율 기준으로 우선순위를 정렬했는가?

## 임팩트 매핑

부채 항목이 어떤 사용자 시나리오에 영향을 주는지 매핑하여 비즈니스 관점의 우선순위를 보완한다.

**매핑 구조**:
```
사용자 시나리오 → 영향받는 API → 관련 부채 항목 → 사용자 영향도
```

**매핑 템플릿**:

| 사용자 시나리오 | 영향받는 API/기능 | 관련 부채 항목 | 사용자 영향도 |
|---------------|-----------------|--------------|-------------|
| 지원자가 지원서 제출 | POST /applications | N+1 쿼리 (Application 조회) | 응답 지연 3초 |
| 채용 담당자가 지원자 목록 조회 | GET /applicants | 페이지네이션 없는 findAll | 타임아웃 발생 |
| 면접관이 평가 제출 | POST /evaluations | 테스트 0건 (Evaluation 도메인) | 버그 발생 시 무방비 |

**활용 방법**:
- 사용자 영향도가 높은 시나리오에 연결된 부채를 우선 해소한다.
- 기존 영향도(High/Med/Low) x 해소 비용 매트릭스에 "사용자 시나리오 수"를 가중치로 추가한다.
- 동일 우선순위 내에서 사용자 시나리오가 많은 부채를 먼저 해소한다.

**체크**:
- [ ] 주요 사용자 시나리오를 식별했는가?
- [ ] 각 부채 항목이 영향을 미치는 시나리오를 매핑했는가?
- [ ] 사용자 영향도를 우선순위 판정에 반영했는가?
