## 분석 항목

1. **영향 서비스 식별**: PRD 기능이 영향을 미치는 서비스(레포)를 전수 식별한다.
2. **기존 코드 패턴 확인**: 실제 코드를 탐색하여 현재 구현 패턴을 파악한다 (코드 탐색 필수).
3. **DB 스키마 변경**: 신규 테이블, 컬럼 추가/변경, 인덱스 변경 필요 여부를 분석한다.
4. **Kafka 이벤트 변경**: 신규 토픽, 기존 이벤트 스키마 변경, Consumer 영향을 파악한다.
5. **인증/인가 변경**: JWT 클레임 변경, RBAC 권한 추가, API Gateway 필터 변경 여부를 확인한다.
6. **기술 리스크 + 대안**: 구현 난이도가 높거나 불확실한 부분에 대한 리스크와 대안을 제시한다.

### MADR (Markdown Architectural Decision Record) 템플릿

기술적 의사결정이 필요한 지점마다 ADR을 작성하여 결정 근거를 추적 가능하게 남긴다.

```markdown
# ADR-{번호}: {결정 제목}

## Status
Proposed | Accepted | Deprecated | Superseded by ADR-{번호}

## Context
어떤 문제/상황에서 이 결정이 필요한가?

## Decision Drivers
- 드라이버 1 (예: 성능 요구사항)
- 드라이버 2 (예: 팀 역량)
- 드라이버 3 (예: 기존 아키텍처 호환성)

## Considered Options
1. Option A: {설명}
2. Option B: {설명}
3. Option C: {설명}

## Decision Outcome
Chosen option: "{옵션}", because {근거}.

### Consequences
- Good: {긍정적 영향}
- Bad: {부정적 영향}
- Neutral: {중립적 영향}

## Pros and Cons of the Options

### Option A
- Good: ...
- Bad: ...

### Option B
- Good: ...
- Bad: ...
```

**적용 방법:**
1. PRD에서 기술적 의사결정이 필요한 지점을 식별한다 (신규 기술 도입, 아키텍처 변경, 통합 방식 등)
2. 각 결정에 대해 ADR을 작성한다 — 최소 2개 이상의 대안을 비교한다
3. Decision Drivers에 PRD의 NFR을 반영한다 (성능, 보안, 확장성 등)
4. Consequences에서 기존 시스템(greeting-new-back 등)에 미치는 영향을 구체적으로 기술한다

### C4 모델 4단계 영향 분석

PRD 기능의 영향 범위를 4개 레벨로 분석하여 변경 전파 범위를 구체화한다.

| 레벨 | 대상 | PRD 분석 활용 |
|------|------|-------------|
| **L1: System Context** | 시스템과 외부 시스템/사용자 관계 | 새 기능이 어떤 외부 시스템과 상호작용하는가? |
| **L2: Container** | 서비스, DB, 메시지 큐 등 | 어떤 서비스(greeting-new-back, aggregator 등)가 영향받는가? |
| **L3: Component** | 서비스 내 모듈/컴포넌트 | 해당 서비스 내 어떤 도메인/패키지가 변경되는가? |
| **L4: Code** | 클래스/인터페이스 | 어떤 클래스/인터페이스가 추가/변경되는가? |

**Impact Analysis 체크리스트:**
- [ ] L2: 영향받는 서비스(Container) 목록을 식별했는가?
- [ ] L2: 서비스 간 새로운 통신 경로가 필요한가? (REST, Kafka, gRPC)
- [ ] L2: 새로운 DB 스키마 변경이 필요한가? (MySQL, MongoDB)
- [ ] L3: 기존 도메인 모델(Entity, Enum)에 변경이 필요한가?
- [ ] L3: 기존 Port/Adapter에 새 메서드가 필요한가?
- [ ] L4: 기존 테스트에 영향을 주는가?

### 4계층 의존성 그래프 분석

변경의 전파 범위(blast radius)를 추정하기 위해 4개 레이어의 의존성을 분석한다.

| 레이어 | 분석 내용 | 도구/방법 |
|--------|---------|----------|
| **서비스 간** | REST 호출, Kafka 토픽 구독/발행 | Kafka 토픽 의존 그래프, API Gateway 라우팅 테이블 |
| **모듈 간** | Gradle 의존성, 공유 라이브러리 | `settings.gradle.kts`, `build.gradle.kts` 분석 |
| **패키지 간** | import 관계, Port/Adapter 의존 | 코드 import 분석 |
| **DB 간** | 크로스 서비스 DB 참조 | 스키마 분석, JOIN 쿼리 탐색 |

**적용 방법:**
1. PRD의 새 기능이 관여하는 서비스/모듈을 식별한다
2. 해당 서비스에서 외부로 나가는 의존성(outgoing)과 들어오는 의존성(incoming)을 매핑한다
3. 순환 의존성(circular dependency) 여부를 확인한다
4. 변경 전파 범위(blast radius)를 추정한다

### 기술 스파이크 기준 (PoC 필수/권장 조건)

아래 조건에 해당하면 구현 전 기술 스파이크를 수행해야 한다.

| 조건 | 스파이크 필요성 |
|------|-------------|
| 팀이 사용해 본 적 없는 기술 | **필수** (PoC) |
| 성능 요구사항이 현재 시스템 한계에 근접 | **필수** (벤치마크) |
| 외부 서비스 API 문서가 불완전 | **필수** (연동 검증) |
| 기존 아키텍처와 다른 패턴 도입 | **권장** (프로토타입) |
| 데이터 마이그레이션 규모가 대량 | **권장** (시범 마이그레이션) |

## 작업 절차

1. PRD에서 기술 관련 키워드(API, DB, 이벤트, 인증 등)를 추출한다.
2. 영향받는 서비스 목록을 레포 맵 기반으로 작성한다.
3. 각 서비스의 관련 코드를 실제로 탐색하여 현재 패턴을 확인한다.
4. DB 스키마(greeting-db-schema/)를 확인하여 변경 필요 사항을 정리한다.
5. Kafka 토픽(greeting-topic/)과 이벤트 스키마를 확인한다.
6. 인증/인가 흐름(greeting_authn-server, greeting_authz-server)을 확인한다.
7. 각 변경 항목에 대해 리스크 수준(높음/중간/낮음)과 대안을 정리한다.
8. 장애 시나리오(이 기능이 장애나면 어디까지 영향이 가는지)를 사전 정의한다.

## 품질 기준

- 영향 서비스 목록이 빠짐없이 도출되어야 한다.
- 코드 탐색 근거(파일 경로, 메서드명)가 반드시 포함되어야 한다.
- "추측"이 아닌 실제 코드 기반 분석이어야 한다.
- 리스크에 대안이 없으면 "대안 없음 - 수용 필요" 명시가 있어야 한다.
- DB/Kafka 변경은 구체적 테이블명/토픽명 수준이어야 한다.
