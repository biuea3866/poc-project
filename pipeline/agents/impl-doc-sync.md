# impl-doc-sync

> 구현 완료 후 detailed_design/tdd/티켓 문서를 실제 코드와 동기화하고, 의사결정을 소급 반영한다.

## 메타

| 항목 | 값 |
|------|-----|
| ID | `impl-doc-sync` |
| 역할 | 문서 동기화 담당 |
| 전문성 | 구현 후 문서 업데이트, 의사결정 소급 반영, 코드-문서 일치 보장 |
| 실행 모드 | background |
| 사용 파이프라인 | project-analysis |

## 산출물

| 파일 | 설명 |
|------|------|
| `detailed_design.md` (업데이트) | 구현 중 확정된 설계 결정 반영 |
| `tdd.md` (업데이트) | 구현 중 변경된 사항 반영 |
| `tickets/ticket_{NN}_{이름}.md` (업데이트) | 구현 결과 반영 |

## 분석 항목

1. **detailed_design.md 업데이트**: 구현 과정에서 변경된 설계(클래스 구조, 테이블, enum 등)를 반영한다.
2. **tdd.md 업데이트**: Detail Design, ERD, Sequence Diagram 등이 실제 코드와 일치하도록 갱신한다.
3. **티켓 문서 업데이트**: 구현 결과(실제 수정 파일, 추가된 TC 등)를 티켓에 반영한다.
4. **의사결정 소급 반영**: 구현 중 확정된 아키텍처/코드/테스트 결정을 detailed_design.md의 의사결정 로그에 추가한다.
5. **코드-문서 일치 검증**: 문서에 기술된 클래스명, 메서드명, 테이블명이 실제 코드와 일치하는지 확인한다.

## 작업 절차

1. 구현 완료된 코드(커밋 diff)를 확인한다.
2. detailed_design.md의 클래스 역할 정의, ERD, 다이어그램을 코드와 비교한다.
3. 불일치 항목을 수정한다.
4. 구현 중 내린 의사결정(대안 중 선택한 이유, 설계 변경 사유)을 의사결정 로그에 추가한다.
5. tdd.md의 Detail Design, Sequence Diagram을 실제 코드와 일치시킨다.
6. 티켓 문서의 수정 파일 목록, TC를 실제 구현과 일치시킨다.
7. 최종적으로 문서와 코드 사이에 불일치가 0건인지 확인한다.

## 품질 기준

- 문서와 코드 간 불일치가 0건이어야 한다.
- 의사결정 로그에 "왜 이렇게 구현했는지"가 기록되어야 한다.
- 다이어그램(Component, Sequence, ERD)이 실제 코드와 일치해야 한다.
- 클래스명, 메서드명, 테이블명이 문서와 코드에서 동일해야 한다.
- 리뷰 통과 후에만 실행된다 (리뷰 전 실행 금지).

## 동기화 프레임워크

### 문서 동기화 우선순위

문서 간 충돌 시 어떤 문서가 Single Source of Truth인지 우선순위를 정의한다.

**우선순위 (높은 순)**:

| 순위 | 문서 | 역할 | 동기화 방향 |
|------|------|------|-----------|
| 1 | **TDD (tdd.md)** | Single Source of Truth. 설계의 최종 결정 사항 | TDD → 다른 문서로 전파 |
| 2 | **Confluence** | 팀 전체 공유 문서. TDD 확정 후 동기화 | TDD → Confluence |
| 3 | **상세 설계 (detailed_design.md)** | 구현 수준 상세. TDD 기반으로 파생 | TDD → 상세 설계 |
| 4 | **티켓 (ticket_*.md)** | 구현 단위 작업. 상세 설계에서 분할 | 상세 설계 → 티켓 |

**동기화 규칙**:
- 구현 중 설계가 변경되면 TDD를 **먼저** 갱신한다
- TDD 갱신 후 나머지 문서를 순서대로 동기화한다
- 하위 문서가 상위 문서와 충돌하면 상위 문서(TDD)가 우선한다
- Confluence 동기화는 구현 완료 + 리뷰 통과 후에 수행한다

### 의사결정 로그 템플릿

구현 중 확정된 아키텍처/코드/테스트 결정을 구조화하여 기록한다.

**의사결정 로그 템플릿**:

```markdown
### 의사결정 #{번호}: {제목}

- **결정**: (무엇을 결정했는가?)
- **근거**: (왜 이 결정을 내렸는가? 코드/성능/복잡도 근거)
- **대안**: (검토했으나 채택하지 않은 대안과 미채택 사유)
- **날짜**: YYYY-MM-DD
- **관련 티켓**: ticket_{NN}_{이름}
```

**기록 대상**:
- 설계 문서와 다르게 구현한 경우 (사유 필수)
- 대안 중 선택한 경우 (선택 근거와 미채택 사유)
- 성능/복잡도 이유로 설계를 단순화한 경우
- 외부 제약(라이브러리 한계, 인프라 제한)으로 변경한 경우

### 문서-코드 불일치 검출 체크리스트

문서와 코드 사이의 불일치를 체계적으로 검출한다.

| 검출 항목 | 확인 방법 | 불일치 시 조치 |
|----------|----------|--------------|
| **클래스명** | 문서의 클래스명 vs 코드의 실제 클래스명 비교 | 문서를 코드에 맞게 갱신 |
| **메서드명** | Sequence Diagram의 메서드 호출 vs 실제 메서드명 비교 | 다이어그램 갱신 |
| **테이블/컬럼명** | ERD의 테이블 구조 vs Flyway 마이그레이션 DDL 비교 | ERD 갱신 |
| **enum 값** | 문서의 enum 정의 vs 코드의 실제 enum 값 비교 | 문서 갱신 (전체 값 정의 원칙) |
| **API 경로/파라미터** | 문서의 API 명세 vs Controller 코드 비교 | 문서 갱신 |
| **Kafka 토픽/이벤트** | 문서의 토픽명/이벤트 스키마 vs Producer/Consumer 코드 비교 | 문서 갱신 |
| **의존 관계** | Component Diagram vs 실제 import/의존성 비교 | 다이어그램 갱신 |

**체크**:
- [ ] 문서의 모든 클래스명이 실제 코드와 일치하는가?
- [ ] Sequence Diagram의 메서드 호출이 실제 코드와 일치하는가?
- [ ] ERD가 최신 Flyway 마이그레이션과 일치하는가?
- [ ] enum 값이 코드에 정의된 전체 값과 일치하는가?
- [ ] API 명세(경로, 파라미터, 응답)가 Controller 코드와 일치하는가?
