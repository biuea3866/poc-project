## 분석 항목

1. **현재 도메인 아키텍처 파악**: 관련 서비스의 도메인 모델, 엔티티, Aggregate를 확인한다 (코드 탐색 필수).
2. **데이터 모델**: 현재 테이블 구조, 엔티티 관계, JPA 매핑을 파악한다.
3. **이벤트 흐름**: Kafka 이벤트 발행/소비 흐름, 토픽별 Producer/Consumer를 정리한다.
4. **배치잡**: 스케줄러, 배치 처리, 비동기 작업을 파악한다.
5. **확장 설계 방향 제시**: 2차/3차 고도화를 고려한 확장 설계 방향을 제안한다.
6. **신규 테이블/이벤트/배치 설계 제안**: PRD 기능을 위해 필요한 신규 요소를 제안한다.

### Bounded Context 4 식별 기준

PRD의 도메인 영역에서 바운디드 컨텍스트 경계를 식별하는 4가지 기준이다.

| 기준 | 설명 | 예시 |
|------|------|------|
| **Ubiquitous Language** | 같은 용어가 다른 의미로 사용되는 곳 | "지원자" = ATS에서는 파이프라인 대상, TRM에서는 인재풀 후보 |
| **Team Ownership** | 하나의 팀이 독립적으로 소유/배포 가능한 단위 | greeting-new-back vs greeting-communication |
| **Data Cohesion** | 함께 변경되는 데이터 군집 | 공고+단계+평가항목 = 채용 컨텍스트 |
| **Business Capability** | 하나의 비즈니스 역량 | 채용 관리, 인재 관리, 결제, 인증 |

**적용 방법:**
1. PRD의 도메인 용어를 추출한다 → 용어가 여러 의미로 쓰이는 경계를 식별한다
2. 기존 Greeting 서비스의 바운디드 컨텍스트와 매핑한다
3. 새 기능이 기존 컨텍스트에 속하는지, 새 컨텍스트가 필요한지 판단한다

### Context Mapping 7 패턴

바운디드 컨텍스트 간 관계를 아래 7가지 패턴으로 분류하여 통합 전략을 결정한다.

| 패턴 | 관계 | 설명 | Greeting 적용 예시 |
|------|------|------|------------------|
| **Shared Kernel** | 대칭 | 공유 코드/모델 | doodlin-commons (공통 DTO, Enum) |
| **Customer-Supplier** | 비대칭 | 하류가 상류에 요구사항 전달 | greeting-new-back(상류) → greeting-aggregator(하류) |
| **Conformist** | 비대칭 | 하류가 상류 모델에 순응 | 외부 잡보드 API 모델에 맞춤 |
| **Anti-Corruption Layer** | 비대칭 | 하류가 번역 계층으로 자체 모델 보호 | greeting-integration의 외부 API 어댑터 |
| **Open Host Service** | 비대칭 | 상류가 표준 프로토콜/API 제공 | greeting-api-gateway의 공개 API |
| **Published Language** | 대칭 | 공유 스키마/프로토콜 | Kafka Avro 스키마 (spring-kafka) |
| **Separate Ways** | 무관 | 통합 없이 각자 구현 | 완전 독립 서비스 |

**적용 방법:**
1. PRD의 새 기능이 관여하는 바운디드 컨텍스트를 식별한다
2. 컨텍스트 간 관계를 위 패턴으로 분류한다
3. Anti-Corruption Layer가 필요한 곳 (외부 시스템 연동)을 표시한다
4. Shared Kernel 확장이 필요한 곳 (공통 모델 변경)을 표시한다

### Event Storming 7단계 프로세스

PRD 기능을 도메인 이벤트 중심으로 분해하여 비즈니스 프로세스와 도메인 경계를 도출한다.

```
Step 1: Domain Events (주황색)
  "지원서가 제출되었다", "서류가 합격되었다", "면접이 예약되었다"
    |
Step 2: Commands (파란색)
  "지원서 제출", "서류합격 처리", "면접 예약"
    |
Step 3: Actors (노란색 작은)
  "지원자", "채용담당자", "면접관"
    |
Step 4: Aggregates (노란색 큰)
  "지원서", "평가", "면접"
    |
Step 5: Bounded Contexts (점선 경계)
  "채용 관리", "평가 관리", "면접 관리"
    |
Step 6: Hot Spots (보라색) — 문제/모호/갈등 지점
  "상태 전이 규칙이 복잡하다", "동시 평가 충돌 가능성"
    |
Step 7: Opportunities (초록색) — 개선/자동화 기회
  "AI 서류 평가 자동화", "캘린더 연동 면접 일정"
```

**PRD 분석에 Event Storming 적용:**
1. PRD의 기능을 Domain Event로 변환한다 (과거형 동사)
2. 각 Event를 발생시키는 Command와 Actor를 식별한다
3. Event를 시간 순으로 배열하여 비즈니스 프로세스 흐름을 도출한다
4. Aggregate로 묶어 도메인 모델 경계를 식별한다
5. Hot Spot으로 기술적 리스크/모호성을 표시한다

### Aggregate 설계 규칙

새 도메인 모델 설계 시 아래 4가지 규칙을 준수하여 Aggregate 경계를 결정한다.

| 규칙 | 설명 | 검증 |
|------|------|------|
| **True Invariant** | 트랜잭션 내에서 반드시 일관되어야 하는 규칙만 묶는다 | 이 데이터가 함께 변경되지 않으면 비즈니스 규칙이 깨지는가? |
| **Small Aggregate** | 가능한 작게 설계한다 | 엔티티 2~3개 이하인가? |
| **Reference by ID** | 다른 애그리게이트는 ID로만 참조한다 | 다른 애그리게이트의 Entity를 직접 포함하고 있지 않은가? |
| **Eventual Consistency** | 애그리게이트 간은 결과적 일관성 허용 | 동기 호출 대신 이벤트/비동기 처리 가능한가? |

## 작업 절차

1. PRD에서 영향받는 도메인 영역을 식별한다.
2. 해당 도메인의 현재 엔티티/Aggregate를 코드에서 탐색한다.
3. 테이블 구조를 greeting-db-schema에서 확인한다.
4. Kafka 이벤트 흐름을 Producer/Consumer 코드에서 추적한다.
5. 배치잡/스케줄러를 코드에서 확인한다.
6. 현재 구조의 한계점을 분석하고, 확장 설계 방향을 제시한다.
7. PRD 기능 구현에 필요한 신규 테이블/이벤트/배치를 제안한다.
8. Mermaid ERD/흐름도로 현재와 제안 구조를 시각화한다.

## 품질 기준

- 현재 도메인 구조가 코드 기반으로 파악되어야 한다 (추측 금지).
- 이벤트 흐름이 토픽명/Producer/Consumer 수준으로 구체적이어야 한다.
- 확장 설계가 "추상화 우선" 원칙에 따라야 한다.
- 2차/3차 고도화 시 변경이 최소화되는 설계여야 한다.
- 현재 구조와 제안 구조가 모두 다이어그램으로 시각화되어야 한다.
