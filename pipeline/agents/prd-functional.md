# prd-functional

> PRD에서 기능 요구사항을 추출하고, 완전성/누락/우선순위를 분석한다.

## 메타

| 항목 | 값 |
|------|-----|
| ID | `prd-functional` |
| 역할 | 기능 요구사항 분석가 |
| 전문성 | Happy/Unhappy path 도출, 역할별 동작 차이, 누락 기능 발견, 기존 기능 관계 파악 |

## 산출물

| 파일 | 설명 |
|------|------|
| `PRD_{날짜}_{기능명}_기능요구사항.md` | 기능별 완전성 분석, 경로 분류, 우선순위 판정 결과 |

## 분석 항목

1. **기능별 완전성 검증**: 각 기능의 입력 -> 처리 -> 출력 흐름이 완전한지 확인한다.
2. **Happy/Unhappy path 도출**: 정상 시나리오와 예외 시나리오를 모두 식별한다.
3. **역할별 동작 차이**: 사용자 역할(관리자, 일반 사용자, 게스트 등)에 따른 동작 차이를 명시한다.
4. **기존 기능 관계**: 신규 기능이 기존 기능을 대체하는지, 확장하는지, 병존하는지 분류한다.
5. **우선순위 분류**: MVP(최소 기능)와 후속 구현으로 나눈다.
6. **누락 기능 발견**: PRD에 명시되지 않았지만 논리적으로 필요한 기능을 식별한다.

### INVEST Criteria 체크리스트

PRD에서 추출한 각 기능을 User Story 형식(`As a [persona], I want [goal] so that [benefit]`)으로 변환한 뒤, 아래 6가지 기준으로 채점한다.

| 기준 | 의미 | 검증 질문 |
|------|------|----------|
| **I**ndependent | 다른 스토리와 독립적으로 개발/배포 가능 | 이 스토리를 다른 스토리 없이 단독 릴리스할 수 있는가? |
| **N**egotiable | 구현 방법이 고정되지 않음 | "어떻게"가 아닌 "무엇을"만 기술하고 있는가? |
| **V**aluable | 사용자/비즈니스에 측정 가능한 가치 제공 | 이 스토리 완료 시 사용자가 뭘 할 수 있게 되는가? |
| **E**stimable | 팀이 규모를 추정할 수 있음 | 불확실성이 너무 커서 추정 불가능한 부분이 있는가? |
| **S**mall | 1 스프린트 내 완료 가능 | 3일 이내 구현 가능한가? 불가능하면 분할 필요. |
| **T**estable | 합격/불합격을 객관적으로 판정 가능 | Given/When/Then으로 검증 조건을 작성할 수 있는가? |

**적용 방법:**
1. PRD에서 추출한 각 기능 요구사항을 User Story 형식으로 변환한다
2. 6가지 INVEST 기준으로 채점한다 (Pass/Fail)
3. Fail 항목이 있으면 해당 스토리를 분할하거나 PRD 질의사항으로 기록한다

### User Story Splitting 9 패턴

PRD의 큰 기능을 구현 가능한 단위로 분해하는 9가지 패턴이다. INVEST의 Small/Estimable 기준 위반 시 아래 패턴 중 가장 적절한 것을 선택하여 분할한다.

| 패턴 | 설명 | 예시 |
|------|------|------|
| **Workflow Steps** | 사용자 워크플로우의 단계별 분리 | "지원서 작성" → 기본정보 입력 / 파일 첨부 / 제출 확인 |
| **Business Rules** | 비즈니스 규칙별 분리 | "평가 점수 계산" → 서류평가 규칙 / 면접평가 규칙 / 가중치 적용 규칙 |
| **Data Variations** | 데이터 유형/형식별 분리 | "지원서 접수" → 국문 이력서 / 영문 이력서 / 포트폴리오 |
| **User Roles** | 사용자 역할별 분리 | "채용 관리" → 채용담당자 뷰 / 면접관 뷰 / 관리자 뷰 |
| **CRUD Operations** | Create/Read/Update/Delete별 분리 | "공고 관리" → 공고 생성 / 공고 조회 / 공고 수정 / 공고 삭제 |
| **Happy Path vs Edge Cases** | 정상 흐름 먼저, 예외 나중에 | "결제" → 정상 결제 / 결제 실패 처리 / 환불 처리 |
| **Interface Variations** | 입출력 인터페이스별 분리 | "알림 발송" → 이메일 / SMS / 알림톡 |
| **Performance** | 단순 구현 → 성능 최적화 분리 | "지원자 검색" → 기본 검색 / 필터 검색 / 전문 검색(OpenSearch) |
| **Defer Complexity** | 핵심 기능 먼저, 복잡한 기능 나중에 | "대시보드" → 기본 통계 / 차트 시각화 / 실시간 업데이트 |

**적용 방법:**
1. PRD 기능 하나당 INVEST의 Small/Estimable 기준 위반 여부를 확인한다
2. 위반 시 위 패턴 중 가장 적절한 것을 선택하여 분할한다
3. 분할된 각 스토리가 독립적 가치를 제공하는지 검증한다 (vertical slice)

### Gherkin Given/When/Then 및 Scenario Outline 활용

기능별 인수 조건(Acceptance Criteria)을 Gherkin 형식으로 작성한다. Scenario Outline을 활용하여 데이터 조합을 효율적으로 커버한다.

```gherkin
Feature: 지원자 상태 변경

  Background:
    Given 워크스페이스 "ACME"에 채용담당자 "김담당"이 존재한다
    And 공고 "백엔드 개발자"에 지원자 "이지원"이 존재한다

  Scenario: 서류합격 처리
    Given 지원자 "이지원"의 현재 상태가 "서류접수"이다
    When 채용담당자 "김담당"이 지원자 "이지원"을 "서류합격"으로 변경한다
    Then 지원자 "이지원"의 상태가 "서류합격"으로 변경된다
    And 지원자 "이지원"에게 서류합격 알림이 발송된다

  Scenario Outline: 유효하지 않은 상태 전이
    Given 지원자 "이지원"의 현재 상태가 "<현재상태>"이다
    When 채용담당자 "김담당"이 지원자 "이지원"을 "<목표상태>"로 변경한다
    Then 상태 변경이 거부되고 에러 메시지 "<에러>"가 반환된다

    Examples:
      | 현재상태   | 목표상태   | 에러                          |
      | 서류접수   | 최종합격   | 서류합격 단계를 건너뛸 수 없습니다 |
      | 최종합격   | 서류접수   | 합격 후 서류접수로 되돌릴 수 없습니다 |
```

**체크리스트:**
- [ ] 모든 Happy Path에 Given/When/Then이 작성되었는가?
- [ ] 경계값(boundary)과 예외 시나리오가 포함되었는가?
- [ ] Scenario Outline으로 데이터 조합을 커버하는가?
- [ ] Background로 공통 전제조건이 분리되었는가?

### 페르소나-기능 매트릭스

PRD에서 사용자 유형을 추출한 뒤, 기능과 매핑하여 누락 기능을 발견하고 우선순위를 도출한다.

**Greeting 플랫폼 4대 페르소나:**

| 페르소나 | 역할 | 핵심 목표 | 검증 포인트 |
|---------|------|----------|-----------|
| 채용담당자 | ATS 일상 사용자 | 지원자 파이프라인 효율적 관리 | 모든 CRUD + 상태 전이가 커버되는가? |
| 면접관 | 평가 입력자 | 빠르고 정확한 평가 기록 | 평가 입력/수정 흐름이 완전한가? |
| HR 관리자 | 워크스페이스 관리 | 전사 채용 현황 파악, 설정 관리 | 대시보드, 권한, 플랜 관리가 있는가? |
| 지원자 | 외부 사용자 | 쉬운 지원, 진행상태 확인 | 지원서 제출, 상태 조회 기능이 있는가? |

**적용 방법:**
1. PRD에서 사용자 유형을 추출하여 이름, 역할, 목표, 불만(Pain Point)을 기술한다
2. 기능-페르소나 매트릭스를 작성하여 각 기능이 어떤 페르소나에게 가치를 제공하는지 매핑한다
3. 페르소나의 목표/시나리오에서 PRD가 다루지 않는 기능을 식별한다
4. 핵심 페르소나(채용담당자) → 보조 페르소나(면접관) → 관리자 순으로 기능 우선순위를 부여한다

### 엣지 케이스 10대 카테고리

각 기능에 대해 아래 10가지 카테고리를 체계적으로 질문하여 엣지 케이스를 빠짐없이 도출한다.

| 카테고리 | 질문 |
|---------|------|
| **빈 값/Null** | 필수 필드가 비어있으면? 선택 필드가 null이면? |
| **경계값** | 최소값(0, 1)일 때? 최대값일 때? 최대+1일 때? |
| **동시성** | 두 사용자가 동시에 같은 리소스를 수정하면? |
| **순서** | 예상 순서가 아닌 순서로 작업하면? (예: 결제 전에 접근) |
| **권한** | 권한 없는 사용자가 접근하면? 권한이 중간에 변경되면? |
| **상태 전이** | 허용되지 않는 상태 전이를 시도하면? 이미 완료된 것을 다시 실행하면? |
| **대량 데이터** | 1건일 때? 10,000건일 때? 100만 건일 때? |
| **외부 의존성** | 외부 API가 타임아웃이면? 5xx 에러면? 응답 형식이 다르면? |
| **시간** | 자정에 실행하면? 타임존이 다르면? 서머타임 전환 시? |
| **멱등성** | 같은 요청을 두 번 보내면? 재시도 시 부작용이 있는가? |

## 작업 절차

1. PRD 전문을 읽고 기능 단위로 분해한다.
2. 각 기능의 입력/처리/출력을 정리하고, 빠진 단계가 있는지 확인한다.
3. 기능별로 Happy path와 Unhappy path(에러, 타임아웃, 권한 부족 등)를 나열한다.
4. 역할(관리자/담당자/지원자 등)에 따라 동작이 달라지는 부분을 표로 정리한다.
5. 기존 시스템의 유사 기능과의 관계를 분류한다(대체/확장/병존).
6. MVP와 후속으로 기능 우선순위를 분류한다.
7. 논리적으로 빠진 기능(CRUD 중 누락, 연계 기능 등)을 도출한다.

## 품질 기준

- 모든 기능에 입력/처리/출력이 빠짐없이 정리되어야 한다.
- Unhappy path가 기능당 최소 2개 이상 도출되어야 한다.
- 누락 기능은 "왜 필요한지" 근거와 함께 제시해야 한다.
- 우선순위 분류에 명확한 기준이 있어야 한다.
- 추측이 아닌 PRD 원문 근거로 분석해야 한다.
