## 분석 항목

1. **MySQL 스키마**: greeting-db-schema/ 레포에서 관련 테이블의 DDL, 컬럼, 인덱스를 확인한다.
2. **Flyway 히스토리**: 마이그레이션 파일 히스토리를 확인하여 스키마 변경 이력을 파악한다.
3. **MongoDB**: MongoDB를 사용하는 컬렉션, 문서 구조, 인덱스를 확인한다.
4. **Redis 캐시키**: Redis에 저장하는 키 패턴, TTL, 사용 용도를 파악한다.
5. **OpenSearch**: 인덱스 매핑, 검색 쿼리 패턴을 확인한다.
6. **Read/Write Split**: 읽기/쓰기 분리 전략이 적용되어 있는지 확인한다.
7. **인덱스/쿼리 패턴**: QueryDSL 쿼리 패턴, 인덱스 활용, N+1 가능성을 분석한다.

## 작업 절차

1. gap_analysis.md와 architecture.md를 입력으로 받아 분석 대상 테이블을 확인한다.
2. greeting-db-schema/ 레포에서 관련 DDL을 탐색한다.
3. Flyway 마이그레이션 파일을 시간순으로 확인하여 변경 이력을 파악한다.
4. MongoDB 사용 여부를 코드에서 확인하고, 컬렉션/문서 구조를 파악한다.
5. Redis 사용 패턴을 코드에서 확인한다 (키 패턴, TTL, 캐시 전략).
6. OpenSearch 사용 여부를 확인하고, 인덱스 매핑과 쿼리 패턴을 파악한다.
7. QueryDSL 쿼리를 분석하여 인덱스 활용 여부와 N+1 가능성을 확인한다.
8. ERD로 테이블 관계를 시각화한다.

## 품질 기준

- MySQL 스키마가 DDL 수준으로 구체적이어야 한다 (컬럼명, 타입, 인덱스).
- Flyway 히스토리에서 최근 변경 이력이 포함되어야 한다.
- Redis 캐시키에 키 패턴과 TTL이 명시되어야 한다.
- 쿼리 패턴에서 N+1 가능성이 분석되어야 한다.
- ERD가 Mermaid로 시각화되어야 한다.
- 모든 분석이 코드/스키마 기반이어야 한다 (추측 금지).

## 분석 프레임워크

### 인덱스 분석 전략

인덱스의 효과성을 사용 빈도 기반으로 분석하고, EXPLAIN을 활용하여 실제 쿼리 계획을 검증한다.

**분석 단계**:

1. **인덱스 현황 수집**: DDL에서 정의된 인덱스 목록을 추출한다
2. **쿼리 패턴 매핑**: QueryDSL 코드에서 WHERE/JOIN/ORDER BY 조건을 수집한다
3. **EXPLAIN 활용**: 주요 쿼리의 실행 계획을 분석한다

**EXPLAIN 결과 판독 기준**:

| 항목 | 좋은 상태 | 경고 상태 | 조치 |
|------|----------|----------|------|
| type | ref, eq_ref, const | ALL, index (풀 스캔) | 인덱스 추가 검토 |
| rows | 예상 대비 적절 | 테이블 전체 행수와 유사 | WHERE 조건 최적화 |
| Extra | Using index | Using filesort, Using temporary | 인덱스 커버링 또는 ORDER BY 최적화 |

**체크**:
- [ ] 모든 QueryDSL 쿼리의 WHERE 조건에 대응하는 인덱스가 있는가?
- [ ] 복합 인덱스의 컬럼 순서가 쿼리 패턴과 일치하는가? (선두 컬럼 활용)
- [ ] 사용되지 않는 인덱스(Dead Index)가 있는가?
- [ ] 인덱스가 과도하여 쓰기 성능에 영향을 주는가?

### 읽기/쓰기 분리 패턴 (CQRS 적용 범위)

데이터 접근 패턴을 읽기/쓰기로 분리하여 최적화 전략을 수립한다.

**CQRS 적용 수준**:

| 수준 | 설명 | 적용 조건 |
|------|------|----------|
| **Level 0** | 단일 모델 | 읽기/쓰기 비율이 유사, 단순 CRUD |
| **Level 1** | 읽기 전용 DTO 분리 | 읽기 쿼리가 복잡하거나, 다수 테이블 JOIN 필요 |
| **Level 2** | Read Replica 분리 | 읽기 트래픽이 쓰기의 10배 이상 |
| **Level 3** | 이벤트 소싱 + 읽기 저장소 분리 | 감사 추적 필수, 상태 재구성 필요 |

**Greeting 적용 판단 기준**:
- 읽기/쓰기 비율 분석: 해당 도메인의 API를 GET(읽기)과 POST/PUT/DELETE(쓰기)로 분류한다
- 읽기 쿼리 복잡도: JOIN 3개 이상이면 읽기 전용 DTO 분리를 검토한다
- 실시간 요구: 읽기 지연 허용 범위에 따라 Read Replica 적용 여부를 판단한다

**체크**:
- [ ] 해당 도메인의 읽기/쓰기 비율을 분석했는가?
- [ ] 읽기 전용 DTO(Projection)가 필요한 쿼리를 식별했는가?
- [ ] Read Replica 적용이 필요한 트래픽 수준인가?

### 데이터 성장 예측 방법

현재 데이터 볼륨에서 미래 성장을 예측하여 스키마/인덱스 전략을 수립한다.

**예측 공식**:
```
예상 데이터량 = 현재 볼륨 × 성장률 × 보존 기간
```

**성장률 추정 기준**:

| 데이터 유형 | 성장 패턴 | 추정 방법 |
|-----------|----------|----------|
| 지원자(Applicant) | 워크스페이스 수 × 공고당 지원자 수 | 월간 신규 지원자 수 트렌드 |
| 이벤트/로그 | 사용자 활동 × 이벤트 종류 수 | 일간 이벤트 수 × 보존 기간 |
| 설정(Config) | 워크스페이스 수 × 설정 항목 수 | 워크스페이스 증가율 기반 |

**임계값 경고**:

| 테이블 행 수 | 상태 | 대응 |
|-------------|------|------|
| < 100만 | 안전 | 현행 유지 |
| 100만 ~ 1000만 | 주의 | 인덱스 최적화, 쿼리 튜닝 |
| 1000만 ~ 1억 | 경고 | 파티셔닝, 아카이빙 검토 |
| 1억+ | 위험 | 샤딩 또는 데이터 저장소 분리 |

**체크**:
- [ ] 현재 데이터 볼륨을 테이블별로 파악했는가?
- [ ] 월간/연간 성장률을 추정했는가?
- [ ] 1년/3년 후 예상 데이터량을 산정했는가?
- [ ] 임계값 초과가 예상되는 테이블에 대응 전략이 있는가?
