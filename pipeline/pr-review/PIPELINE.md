# 코드 리뷰 (PR Review) 파이프라인

> PR이 올라왔을 때 변경 영향 범위, 사이드 이펙트, 누락 검토

---

## 파이프라인 흐름

```
[INPUT] PR URL 또는 변경 파일 목록
    │
    ▼
━━ Phase 1: 변경 범위 파악 ━━━━━━━━━━━━━━━━━━━━━━━━
    │  팀장이 PR diff 확인
    │  - 변경 파일 목록 정리
    │  - 변경 유형 분류 (기능/버그픽스/리팩토링/설정)
    │  - 영향 받는 도메인 식별
    │
    ▼
━━ Phase 2: 심층 분석 (병렬) ━━━━━━━━━━━━━━━━━━━━━━
    │  ┌─ pr-impact-analyst: 영향 범위 분석
    │  └─ review-code: 코드 품질 분석
    │
    ▼
━━ Phase 3: 리뷰 종합 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    │  팀장이 결과 종합
    │  - Must Fix (반드시 수정)
    │  - Should Fix (수정 권장)
    │  - Nice to Have (선택)
    │  - 누락된 변경 사항
    │
    ▼
[OUTPUT] 코드 리뷰 보고서
```

---

## 에이전트 역할 정의

### 팀장 (메인 Claude)
- PR diff 파싱 및 변경 파일 분류
- 에이전트 결과 종합
- 최종 리뷰 코멘트 작성 (심각도별 분류)

### 분석 에이전트

| 에이전트 | 역할 | 목표 |
|---------|------|------|
| [`pr-impact-analyst`](../agents/pr-impact-analyst.md) | 영향 범위 분석가 | 변경된 코드가 다른 곳에 미치는 영향 추적 |
| [`review-code`](../agents/review-code.md) | 코드 품질 분석가 | 코드 품질, 패턴 준수, 잠재적 버그 검출 |

### 페르소나 에이전트 (선택적 병행 스폰)

| 에이전트 | 관점 | 스폰 조건 |
|---------|------|----------|
| [`be-senior`](../agents/be-senior.md) | 프로덕션 안전성, 엣지 케이스, 성능 | 기능 추가, 결제/인증 등 고위험 PR |
| [`be-tech-lead`](../agents/be-tech-lead.md) | 아키텍처 일관성, 서비스 간 영향 | 크로스 레포 변경, 공유 라이브러리 변경 |

---

## 에이전트 할당 규칙

| PR 유형 | 할당 에이전트 |
|---------|-------------|
| 단일 파일 수정 (간단) | 에이전트 없이 팀장만 |
| 기능 추가 (단일 레포) | `pr-impact-analyst` + `review-code` |
| 기능 추가 (크로스 레포) | `pr-impact-analyst` + `review-code` + `be-tech-lead` |
| 리팩토링 | `pr-impact-analyst` + `be-senior` |
| 버그 픽스 | `review-code`만 (수정 적절성 + 사이드 이펙트) |
| 공유 라이브러리 변경 | `pr-impact-analyst` + `review-code` + `be-tech-lead` |
| 설정/인프라 변경 | `pr-impact-analyst`만 |
| 결제/인증 등 고위험 | `pr-impact-analyst` + `review-code` + `be-senior` |

## 리뷰 심각도 분류

```
🔴 Must Fix (Blocker)
   - 런타임 에러 발생 가능
   - 데이터 손상 가능
   - 보안 취약점
   - 다른 서비스 장애 유발 가능

🟡 Should Fix (Warning)
   - 성능 저하 가능
   - 패턴 위반
   - 에러 핸들링 부족
   - 테스트 누락

🟢 Nice to Have (Suggestion)
   - 코드 가독성 개선
   - 네이밍 개선
   - 중복 코드 제거
   - 주석 추가
```

## 출력 어조

산출물은 **팀원과 공유하는 문서**입니다. 읽는 사람이 지치지 않도록 아래 원칙을 따릅니다.

- **핵심부터** — 결론·액션 아이템을 앞에 씁니다. 이유와 배경은 뒤에 써도 됩니다.
- **짧게** — 한 문장으로 쓸 수 있으면 세 문장으로 쓰지 않습니다.
- **구체적으로** — "여러 곳" 대신 "3곳", "느릴 수 있음" 대신 "users 테이블 full scan" 처럼 씁니다.
- **표·불릿 우선** — 비교·목록은 문장보다 표나 불릿으로 씁니다.
- **중립적으로** — 문제를 발견해도 단정짓지 않고 확인 사항으로 전달합니다.

## verification 파이프라인과의 연계

PR 리뷰 시 verification 결과가 있으면 함께 참조한다.

```
verification 통과 여부 확인:
  .analysis/verification/results/{날짜}_{티켓번호}/verification_report.md
  → "위반 0건" 이면 설계 원칙 검증은 스킵 (이미 통과)
  → "위반 N건" 이면 해당 위반이 PR에서 수정되었는지 확인
```

| 시점 | 파이프라인 | 역할 |
|------|----------|------|
| PR 생성 전 | **verification** | 설계 원칙 + AC 검증 (자체 검증) |
| PR 생성 후 | **pr-review** (이 파이프라인) | 코드 품질 + 영향 범위 + 사이드 이펙트 (팀 리뷰) |

## 기술 성장 포인트

> 코드 리뷰는 다른 사람의 코드에서 배우는 가장 효과적인 방법.

| 시도 | 설명 |
|------|------|
| **"왜"를 물어보기** | "이건 왜 이렇게 했나요?"가 가장 좋은 리뷰 코멘트. 정답을 제시하는 것보다 질문이 성장에 도움 |
| **대안 제시** | 지적만 하지 않고 "이렇게 하면 어떨까요?" 대안을 함께 제시. 대안을 떠올리는 것 자체가 훈련 |
| **패턴 인식** | 반복되는 문제(nullable, God class, 순환 의존)를 패턴으로 인식. 팀 컨벤션 개선 제안으로 연결 |
| **좋은 코드 칭찬** | 잘 작성된 코드를 명시적으로 칭찬. 어떤 점이 좋은지 설명하면 팀 전체 수준이 올라감 |

## 출력 형식

분석 결과는 `.analysis/pr-review/results/` 디렉토리에 저장
