# 티켓 작성 가이드

## 티켓 분해 원칙 — 의존성·병목 최소화

티켓은 **서로의 의존성과 병목을 최소화**하도록 작성합니다. 병렬 작업 가능성이 클수록 좋은 분해입니다.

### 좋은 분해
- 한 티켓 = 한 책임. 다른 티켓의 클래스/스키마를 기다리지 않아도 시작 가능
- 공통 인터페이스·DTO·상수는 **선행 티켓**으로 별도 분리
- 도메인이 다르면 티켓도 분리 (Applicant 작업과 Opening 작업을 한 티켓에 묶지 않음)
- 레이어가 다르면 가능한 분리 (Controller와 Repository를 다른 티켓으로)

### 안 좋은 분해 (수정 필요)
- 한 티켓에서 3개 이상 도메인을 동시에 건드림 → **도메인별 분리**
- 후행 티켓 5개 이상이 선행 의존하는 거대 티켓 → **공통 부분 추출 + 도메인별 후속 티켓**
- "전체 마이그레이션" 같은 단일 티켓 → **단계별 분리**

### 불가피한 병목
후행 티켓이 import/참조해야 하는 공통 산출물은 분리할 수 없는 병목입니다. 다음을 포함하되 이에 한정되지 않습니다:
- 공통 코드 (Entity, DomainService, 추상 인터페이스, 공통 DTO·상수·enum, 유틸리티)
- 공통 스키마 (DB 마이그레이션, Kafka 토픽·Avro, OpenAPI)
- 공통 설정 (환경 변수, 빌드 의존성, 모듈 구조)
- 공통 인프라 (외부 API client, 인증/인가 미들웨어)

이런 산출물은 **단일 선행 티켓**으로 분리하고, 후행 티켓이 모두 이를 기다리도록 의존 관계를 명시합니다.

> TPM이 티켓을 분해할 때 이 원칙을 적용합니다. 분해 결과에서 후행 의존이 3개 이상인 티켓이 보이면 분해 자체를 다시 검토합니다.

### Fan-out 너비 목표

티켓 분해의 목표는 **각 wave의 ready 셋 크기를 최대화**하는 것입니다.
좋은 분해는 DAG를 위상정렬했을 때 평탄한 횡(layer)이 많이 나옵니다.

- **후행 의존 카운트 ≥ 3 티켓**: 분해 재검토 트리거. 공통 산출물 인터페이스를 더 잘게 쪼개
  fan-out 면적을 늘립니다. 예) "공통 API 정의" 1개 → "auth API", "rate-limit API", "DTO 정의" 3개로 분해.
- **목표 평균 wave 너비**: 팀 동시 작업 인원의 70% 이상.
  예) 팀 4명 → 평균 wave 너비 ≥ 3.
- **체크 방법**: TPM이 산출한 DAG로 위상정렬 시뮬레이션을 돌려 wave 너비 분포를 확인합니다.
  모든 wave 너비가 1~2라면 분해 실패. 거대 병목을 더 쪼개거나 도메인 경계를 다시 그립니다.
- **반(反)패턴**: wave 1=1개, wave 2=1개, wave 3=1개… 형태의 직선형 DAG는 직렬화로 간주.
  팀이 1명이 아니라면 사실상 분해 안 한 것과 같습니다.

### 패키지 이전 티켓 작성 시 (금지 패턴)

패키지 리네임/이전이 작업 범위에 들어가면 다음을 **티켓 작업 범위에 명시**한다.

| 항목 | 명시 필수 |
|---|---|
| Port 인터페이스 처리 | "발견 시 제거, Repository 직접 주입 전환" |
| Adapter 정리 | "단순 위임 Adapter 제거" |
| Anemic Entity | "Rich Domain Model로 재배치" |
| 호출부 갱신 | "구 패키지 import 100% 갱신, 구 디렉토리 0 파일" |
| typealias | "금지" 명시 |

**자동 reject 패턴**: 티켓 작업 범위에 "디렉토리 이동 + package 선언 변경"만 적혀 있으면 TPM이 분해 단계에서 반려.

### Single Writer per File (한 파일 = 한 티켓)

같은 wave 안의 두 티켓이 동일 파일을 수정하면 wave 종료 시점의 dev 머지에서 충돌이 발생합니다.
TPM은 분해 직후 다음을 확인합니다:

1. **수정 예상 파일 집합 추정** — 각 티켓이 만지는 파일을 도메인 패키지·컨트롤러·설정·빌드 스크립트 단위로 적습니다.
2. **같은 wave 내 교집합 ∅ 검증** — 위상정렬한 wave 안에서 두 티켓이 같은 파일을 수정하면 분해 실패.
3. **공통 파일 수정 분리** — 교집합이 있으면 공통 파일 수정을 별도 **통합 티켓**으로 추출해
   후행 wave에 단독 배치합니다.

#### 권장 패턴

| 작업 유형 | 배치 |
|---|---|
| 도메인 서비스 신설 (새 패키지 추가) | 같은 wave OK — 충돌 없음 |
| 공통 컨트롤러 와이어업 (예: ApiController) | 마지막 wave에 단독 통합 티켓 |
| 공통 설정 (application.yml, build.gradle.kts) | 선행 wave 단독 또는 통합 티켓 |
| SSE Hub·Notification Publisher 같은 공유 컴포넌트 | 정의는 선행 단독, 사용은 후행 도메인별 |

#### 검증 예시 (<COMPANY>-dashboard 회고)

Wave 3에 BE-03~BE-10이 들어가는데 모두 새 패키지(`my/dashboard/github/`, `my/dashboard/cs/`, ...)만
만들므로 파일 교집합 ∅, 같은 wave 배치 안전. ApiController 와이어업은 별도 BE-17으로 빠져
마지막 wave 단독 배치. NotificationSseHub 같은 공유 컴포넌트는 BE-14에서 정의를 마치고
BE-15b에서 사용 — 별도 wave로 분리되어 충돌 없음.

## md 파일 구조

티켓 md 파일은 다음 순서로 작성합니다.

```markdown
# [접두사-NN] 티켓 제목

## 작업 내용 (설계 의도)
### 변경 사항
{왜 이 변경이 필요한지, 설계 의도 수준으로 기술}

## 다이어그램
### 처리 흐름
{sequenceDiagram}
### 클래스 의존
{flowchart LR}

## 테스트 케이스

### 단위 테스트 (Unit)
| ID | 대상 | 케이스 (한 문장) |

### 레포지토리 테스트 (Repository / Persistence)
| ID | 대상 | 케이스 (한 문장) |

### 시나리오 테스트 (Scenario / Integration)
| ID | 시나리오 | 케이스 (한 문장) |
```

## 포함하지 않는 것

- **AC(인수 조건)**: md에도 Jira에도 넣지 않습니다.
- **생성/수정 파일 목록**: 코드베이스가 기준입니다. 문서에 파일 목록을 관리하지 않습니다.

## 코드 / SQL

설계 의도는 다이어그램과 텍스트로 표현합니다. 코드와 SQL은 **정말 필요한 경우에만** 포함합니다.

필요한 경우: 복잡한 쿼리 조건, Avro 스키마 초안, 마이그레이션 DDL 등 텍스트로 표현이 불가한 경우.

포함 시 반드시 `<details>` 숨기기 섹션에 넣습니다.

```markdown
<details>
<summary>DDL 참고</summary>

```sql
ALTER TABLE applicants ADD COLUMN rejected_at DATETIME(6) NULL;
```

</details>
```

## Jira 등록

티켓 md 작성 후 Jira 등록은 `/jira-ticket` 스킬을 사용합니다.
ADF 변환, 렌더링 검토, 의존 링크 연결까지 스킬에서 처리합니다.

## 테스트 케이스

### 작성 원칙

티켓마다 **단위 / 레포지토리 / 시나리오** 세 계층의 테스트 케이스를 모두 작성합니다.
한 계층만 채우는 티켓은 분해 또는 설계가 부족하다는 신호입니다.

| 계층 | 검증 대상 | 구현 도구 | 외부 의존 |
|---|---|---|---|
| **단위 (Unit)** | 도메인 모델·서비스의 순수 로직, 상태 전이, 정책, 검증 규칙 | Kotest BehaviorSpec + MockK / 순수 함수 | 없음 (mock도 최소) |
| **레포지토리 (Repository)** | JPA 매핑·QueryDSL 쿼리·인덱스·트랜잭션·낙관락·고유제약·페이징 | `@DataJpaTest` + **Testcontainers**(실DB) + QueryDSL | 실 DB 컨테이너 |
| **시나리오 (Scenario)** | UseCase·Facade 한 흐름 End-to-End — 외부 이벤트 진입 → 도메인 → DB·발행 → 상태 변화 | `@SpringBootTest` + Testcontainers(DB·Redis·Kafka) | 실 DB + 메시지 브로커 + 외부 stub |

세 계층 모두 한 문장으로 적고, ID는 `<계층약자>-<번호>` 형식을 권장합니다 (`U-01`, `R-01`, `S-01`).

### 계층별 케이스 도출 체크리스트

**단위 (Unit)** — 적어도 다음을 커버:
- 정상 경로 1건 (Happy path)
- 도메인 불변식(invariant) 위반 시 예외 발생
- 상태 전이 규칙: 허용되는 전이 ✅ / 금지된 전이 ❌
- 경계값(`0`, `MAX`, `null`, 빈 문자열, 미래/과거 일자)
- 정책 분기(if/when) — 각 분기마다 1건

**레포지토리 (Repository)** — 적어도 다음을 커버:
- save → findById 라운드트립으로 필드/타입 정확히 매핑되는지 (특히 `ZonedDateTime`·enum·JSON 컬럼)
- 핵심 조회 쿼리 결과 개수·정렬·페이징
- 고유 제약 / 외래키 정책 위반 시 동작
- 낙관락 충돌 시 `OptimisticLockException`
- 인덱스 의도대로 사용되는지 (필요 시 `EXPLAIN` 또는 쿼리 카운트)

**시나리오 (Scenario)** — 적어도 다음을 커버:
- 메인 시나리오 (정상 입력 → 정상 결과) — 외부 이벤트 / API / Kafka 메시지 진입부터 시작
- 멱등성 (같은 메시지 N회 처리 시 상태 1회만 변경)
- 보상/롤백 (중간 단계 실패 시 트랜잭션·도메인 이벤트 일관성)
- 외부 시스템 장애 stub (5xx, 타임아웃) 시의 재시도·서킷 동작
- 발행되는 도메인 이벤트 페이로드와 토픽 검증

### 작성 예시

```markdown
### 단위 테스트 (Unit)
| ID | 대상 | 케이스 |
|---|---|---|
| U-01 | `<엔티티A>` | 만료 시각이 현재 시각 이전이면 만료 상태로 전이된다 |
| U-02 | `<엔티티A>` | 만료 시각이 미래면 만료 전이는 IllegalStateException을 던진다 |
| U-03 | `<엔티티A>RejectPolicy` | 이미 합격된 건은 불합격 처리 시 도메인 예외가 발생한다 |
| U-04 | `<엔티티A>DomainService` | 중복 키 입력 시 첫 번째 호출만 성공한다 |

### 레포지토리 테스트 (Repository / Persistence)
| ID | 대상 | 케이스 |
|---|---|---|
| R-01 | `<엔티티A>Repository` | save 후 ZonedDateTime이 UTC로 저장되고 조회 시 원본 zone과 같은 instant로 복원된다 |
| R-02 | `<엔티티A>QueryDslRepository` | 상태=PENDING + 만료시각<NOW 조건의 페이징 조회가 createdAt asc 정렬로 반환된다 |
| R-03 | `<엔티티A>Repository` | 동일 (workspaceId, externalId)로 insert 시 unique 제약 위반이 발생한다 |
| R-04 | `<엔티티A>Repository` | 두 트랜잭션이 같은 row를 수정하면 후행 커밋이 OptimisticLockException을 던진다 |

### 시나리오 테스트 (Scenario / Integration)
| ID | 시나리오 | 케이스 |
|---|---|---|
| S-01 | 만료 처리 메인 플로우 | expired-applicant kafka 메시지 수신 시 DB 상태가 EXPIRED로 바뀌고 `<엔티티A>.expired` 이벤트가 발행된다 |
| S-02 | 멱등성 | 같은 messageId의 메시지를 3회 수신해도 상태 변경과 이벤트 발행은 1회만 일어난다 |
| S-03 | 외부 시스템 장애 | 통지 서비스가 5xx를 반환하면 재시도 후 보상 트랜잭션으로 도메인 상태가 롤백된다 |
| S-04 | 동시성 | 같은 엔티티에 대한 만료/취소 요청이 동시에 도착해도 종단 상태가 한 가지로 수렴한다 |
```

### 형식 규칙

- 케이스는 **한 문장**으로 작성합니다. Given/When/Then을 풀어서 쓰지 않습니다.
- 주어·동사·관찰 가능한 결과를 한 문장에 담습니다.
  - ✅ "만료된 <엔티티A>는 상태가 EXPIRED로 변경된다"
  - ✅ "중복 메시지 수신 시 멱등하게 처리된다"
  - ❌ "EXPIRED 처리 동작 검증" (관찰 결과 없음)
  - ❌ "엣지 케이스 처리" (모호함)
- ID는 계층 약자 + 일련번호. 의존 케이스가 있으면 비고에 `depends: U-02` 같이 표기합니다.
- 외부 시스템 의존이 1개라도 들어가면 **시나리오** 계층입니다. 단위에 두지 않습니다.

### 매핑: 테스트 케이스 → 테스트 파일

티켓 구현 시 위 ID는 그대로 테스트 메서드/Describe 이름의 접두사로 사용합니다.

```kotlin
class ApplicantExpirePolicyTest : BehaviorSpec({
    Given("만료 시각이 현재 시각 이전인 <엔티티A>") {
        When("만료 처리를 호출하면") {
            Then("[U-01] 상태가 EXPIRED로 전이된다") { ... }
        }
    }
})
```

이렇게 하면 PR 리뷰어가 티켓의 케이스 ID로 grep만 해도 누락 여부를 즉시 확인할 수 있습니다.
