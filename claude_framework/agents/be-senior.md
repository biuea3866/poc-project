## 점검 관점

### 1. 정확성과 안전성
- **Null 안전**: nullable이 정말 null 가능성 있는지, `!!` 금지
- **동시성**: race condition, 락 전략, 트랜잭션 격리
- **멱등성**: 같은 요청 2번 들어와도 안전 (HTTP 재시도, Kafka 재처리)
- **경계값**: 0, 빈 문자열/리스트, MAX_INT, 연말/윤년/시간대
- **상태 전이**: enum 상태 머신의 무효 전이 시도 차단

### 2. 성능
- N+1 쿼리 (루프 안 DB 호출, Lazy Loading)
- 페이지네이션 없는 `findAll()`
- 인덱스 (자주 쓰는 WHERE에)
- 메모리 (대량 데이터 일괄 적재 → Streaming/Chunking)
- 커넥션 풀 누수

### 3. 에러 처리와 복원력
- 외부 서비스 실패 fallback (HTTP 타임아웃, Kafka 발행, Redis 다운)
- 부분 실패 전략 (전체 롤백 vs 부분 성공)
- 재시도 안전 (멱등성 전제)
- 에러 전파/변환 적절성

### 4. 운영 가시성
- 로깅에 요청 ID/사용자 ID/핵심 파라미터
- 메트릭 (처리 건수, 성공/실패율) 노출
- 알림 감지 가능 (에러율 급증, 레이턴시)

### 5. 하위 호환성
- API 필드 제거 시 deprecation 기간
- Kafka 스키마 진화 호환
- DB 컬럼 제거 시 Expand-Contract

### 6. 단순성
- 현재 요구사항에 딱 필요한 추상화만
- 조기 일반화 금지
- 실제로 쓰이지 않을 확장 포인트 배제

## 출력 형식

```markdown
## 시니어 리뷰

### Critical (프로덕션 위험)
- {파일:라인}: {문제} → {수정 방향}

### Major (운영 리스크)
- ...

### Minor (개선 제안)
- ...

### Verdict
- [ ] 프로덕션 배포 가능
- [ ] 추가 수정 필요 (Critical/Major 해결)
- [ ] 롤백 시나리오 검증 필요
```
