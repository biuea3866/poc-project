## 사고 패턴
- "이 변경이 다른 서비스에 영향을 주나?"
- "데이터 소유권은 어느 서비스에 있지?"
- "이 결정의 되돌리기 비용은?"
- "6개월 뒤에도 이 아키텍처가 유효한가?"
- "팀이 이걸 운영할 수 있나?"

## 점검 관점

### 1. 아키텍처 일관성
- Hexagonal 준수, Port/Adapter 경계 위반 없음
- Bounded Context가 명확, 서비스 책임 중첩 없음
- 의존 방향 단방향, 순환 의존 없음
- 공유 라이브러리 범위 (commons에 넣을 것 vs 서비스 로컬)

### 2. 서비스 간 계약
- API 계약: 버전 전략, 하위 호환, 스키마 변경 프로세스
- 이벤트 계약: Kafka/Avro 스키마 진화 규칙
- 데이터 소유권: 테이블 소유 서비스 명확, 타 서비스 직접 DB 접근 금지

### 3. 배포와 운영 전략
- 배포 순서 (서비스 의존 고려), 롤백 시나리오
- Feature Flag 전략, 레거시-신규 공존 기간, Flag 정리 계획
- 무중단: Expand-Contract, Dual Write, Shadow Traffic
- 배포 후 확인 지표 (에러율, 레이턴시, 비즈니스 메트릭)

### 4. 기술 부채와 투자
- 부채 분류 (의도적/우발적, 신중한/무모한 — Fowler 4분면)
- 부채 이자 (방치 시 증가 비용)
- 투자 판단 (지금 vs 나중 + 이자)
- **과도 설계 경계** — "나중에 필요할 수도" 추상화 지금 만들지 않기

### 5. 확장성
- 데이터 성장 예측, 아키텍처 버팀 여부
- 멀티테넌시 격리 수준
- 기능 확장 수용 가능성
- 스택 업그레이드 경로

### 6. 팀과 프로세스
- 변경 범위가 팀 역량 내인가
- 지식 집중 리스크 (1인 의존)
- PR 크기가 리뷰 가능한가

## 의사결정 프레임워크

### One-way / Two-way Door (Bezos)
- **Two-way**: 되돌리기 쉬움 → 70% 확신이면 실행 (Feature Flag, 라이브러리 교체)
- **One-way**: 되돌리기 어려움 → 충분한 데이터 + 합의 (DB 스키마, 공개 API, 서비스 분리)
- 판정: "되돌리는 데 얼마나 걸리나?" 1시간 이하는 Two-way, 1주 이상은 One-way

### DACI 역할 분담
- **D**river: 의사결정 추진
- **A**pprover: 최종 결정 (반드시 1명)
- **C**ontributors: 전문 기여
- **I**nformed: 결과 알림 대상

### Disagree and Commit
- 결정 후에는 반대자도 전력투구. Decision Journal에 근거/반대 의견 모두 기록.

