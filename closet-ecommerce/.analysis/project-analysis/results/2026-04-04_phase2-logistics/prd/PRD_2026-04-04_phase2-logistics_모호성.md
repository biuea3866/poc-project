## 2. 해석이 갈리는 요구사항

### 2.1 [HIGH] US-504 반품 사유 "SIZE_MISMATCH"의 배송비 부담 주체

**PRD 원문:**
- "CHANGE_OF_MIND 반품은 반품 배송비(3,000원)를 구매자가 부담한다"
- "DEFECTIVE, WRONG_ITEM 반품은 판매자가 배송비를 부담한다"

**모호성:**
`SIZE_MISMATCH`(사이즈 불일치)가 어디에도 분류되지 않았다. API 응답 예시에서는 `SIZE_MISMATCH` 사유에 `shippingFeeBearer: "BUYER"`로 나오지만, AC에는 명시적 분류가 없다.

- 해석 A: `SIZE_MISMATCH`는 `CHANGE_OF_MIND`와 같은 구매자 귀책 -> 구매자 부담
- 해석 B: 상품 설명의 사이즈 정보가 부정확한 경우 -> 판매자 귀책

**제안:** 사유별 배송비 부담 매핑 테이블을 명확히 정의하고, 사이즈 표기 오류 같은 예외 케이스의 처리 방안 추가

### 2.2 [HIGH] US-503/US-504 "구매확정 후에는 반품/교환 신청이 불가능하다" vs "배송 완료 후 7일 이내"

**모호성:**
- 자동 구매확정은 배송 완료 후 7일
- 반품/교환 가능 기간도 배송 완료 후 7일

이 두 조건이 동시에 만료된다. 정확히 7일째 되는 날에 반품 신청과 자동 구매확정 배치가 동시에 수행되면?

- 해석 A: 7일째 00:00 배치가 먼저 실행 -> 구매확정되어 반품 불가
- 해석 B: 7일째 자정까지 반품 신청 가능 -> 23:59에 반품 신청하면 구매확정 배제

**제안:** 경계 조건(7일 만료 시점)의 우선순위를 명확히 정의. "배송 완료 후 7 * 24시간" vs "7일째 자정" 기준 명시

### 2.3 [MEDIUM] US-505 교환 시 가격 차이 처리

**PRD 원문:** "동일 상품의 다른 옵션(사이즈/색상)으로만 교환 가능"

**모호성:**
동일 상품이라도 색상/사이즈에 따라 가격이 다를 수 있다 (예: 한정 컬러 추가 요금). 가격 차이 발생 시 추가 결제 또는 부분 환불 로직이 전혀 언급되지 않았다.

- 해석 A: 동일 상품 = 동일 가격이므로 차액 처리 불필요
- 해석 B: 옵션별 가격 차이가 있으면 차액 정산 필요

**제안:** 교환 시 가격 차이 발생 여부와 처리 방안 확인

### 2.4 [MEDIUM] US-601 "주문 생성 이벤트 수신 시 재고 차감" - 부분 주문 처리

**모호성:**
하나의 주문에 여러 상품이 포함될 때, 일부 상품만 재고 부족인 경우의 처리가 미정의.

- 해석 A: 전체 주문 거절 (All or Nothing)
- 해석 B: 재고 있는 상품만 처리, 나머지 취소
- 해석 C: 재고 부족 상품 대기(backorder) 후 입고 시 처리

**제안:** 부분 재고 부족 시 처리 정책 명확화

### 2.5 [MEDIUM] US-702 검색 정렬 "관련도순(기본)"과 "인기순"의 차이

**모호성:**
- 관련도순(RELEVANCE): Elasticsearch 기본 score 기반
- 인기순(POPULAR): 무엇 기준? 판매량? 리뷰 수? 조회수? 복합?

**제안:** 인기순의 구체적 정렬 기준(판매량, 리뷰 수, 최근 N일 기준 등)과 가중치 정의

### 2.6 [LOW] US-801 "리뷰 삭제는 본인만 가능하다"

**모호성:**
관리자(Admin)의 리뷰 삭제/숨김 권한이 언급되지 않음. 부적절한 리뷰(욕설, 허위 등) 관리 방안 없음.

- 해석 A: 본인만 삭제 가능, 관리자 권한 없음
- 해석 B: 본인 삭제 + 관리자 숨김(HIDDEN 상태) 가능

**제안:** 관리자 리뷰 관리 기능(숨김/신고 처리) 범위 정의

## 3. 누락된 전제조건 / 암묵적 가정

### 3.1 [CRITICAL] WMS(Warehouse Management System) 요구사항 전면 누락

PRD에는 WMS 관련 요구사항이 전혀 없다. 그러나 프로젝트 컨텍스트에서 "WMS를 상세 구현 예정 (올리브영 GMS 벤치마킹)"이 명시되어 있다.

**누락 항목:**
- 입고(Inbound): 입고 검수, 입고 예정 등록, 실제 입고 확인
- 출고(Outbound): 출고 지시, 피킹/패킹, 출고 확인
- 재고 위치(Location) 관리: 창고 내 적치 위치
- 로트(Lot) 관리: 입고일자별 선입선출(FIFO)
- 반품 입고: 반품 수거 후 검수, 양품/불량 분류, 재고 복구
- 재고 실사(Stocktaking): 정기/비정기 재고 조사

**영향:** 현재 inventory-service의 단순 수량 관리(+/-)만으로는 WMS 수준의 물류 운영 불가

**제안:** WMS 요구사항을 별도 PRD로 분리하되, 현재 inventory-service와의 인터페이스(이벤트, API)를 Phase 2 PRD에 명시

### 3.2 [HIGH] Mock 외부 API 서버와 PRD의 불일치

**현재 상태:**
- `closet-external-api`에 택배사 4개(CJ/로젠/롯데/우체국) Mock API가 구현됨
- 각 택배사별 개별 엔드포인트: `/carrier/cj/api/v1/shipments`, `/carrier/cj/api/v1/tracking/{trackingNumber}` 등
- Mock 서버는 배송 등록 + 추적 조회 + 상태 진행(advance) 기능 제공

**PRD의 언급:**
- US-502에서 "택배사 API(스마트택배 등)"로 모호하게 기술
- 기술 의사결정에서 "스마트택배 API"를 선택했다고 명시

**불일치:**
- Mock 서버는 각 택배사별 개별 API (CJ, LOGEN, LOTTE, EPOST)
- PRD는 "스마트택배"라는 통합 조회 서비스를 가정

**해결 필요:**
1. shipping-service가 Mock 서버의 개별 택배사 API를 직접 호출할지
2. 아니면 Mock 서버에 통합 조회 API(`/carrier/unified/tracking`)를 추가할지
3. Mock 서버의 "배송 등록" API를 사용할지 (PRD에서는 송장번호를 판매자가 직접 입력하는데, Mock 서버는 자동 채번)

### 3.3 [HIGH] 결제 취소/환불 API 인터페이스 미정의

US-504에서 "반품 승인 시 결제 취소 API를 호출하여 환불 처리한다"고 되어 있으나:

- Phase 1의 payment-service 환불 API 스펙이 Phase 2 PRD에 참조되지 않음
- 부분 환불(주문 중 1건만 반품) 지원 여부 불분명
- PG사별 환불 정책(부분 환불 가능 여부, 환불 소요 시간) 미고려
- 반품 배송비 공제 후 환불인지, 별도 청구인지 미정의

### 3.4 [HIGH] Phase 1 주문 상태 모델과의 연결

PRD에서 새로운 주문 상태(`SHIPPING`, `DELIVERED`, `CONFIRMED`)를 도입하지만:

- Phase 1의 기존 주문 상태 모델이 어디까지인지 참조 없음
- `PAID` -> `SHIPPING` 전이를 누가 트리거하는지 (shipping-service가 order-service에 이벤트 발행? 직접 API 호출?)

## 4. 상충하는 요구사항

### 4.1 [HIGH] 송장번호 형식 vs Mock 서버 자동 채번

**US-501 AC:** "송장번호 형식 검증 (숫자 10-15자리)"
**US-501 API:** 판매자가 직접 `trackingNumber: "1234567890123"`을 입력

**Mock 서버 구현:**
```kotlin
val trackingNumber = "$prefix${trackingSeq.getAndIncrement()}"
// 결과: "CJ1000000001" (문자열 포함, 12자리)
```

Mock 서버는 택배 접수 시 자동으로 송장번호를 채번하며, "CJ" prefix가 붙어 **숫자만 10-15자리** 규칙에 위배됨. 또한 PRD는 판매자가 송장번호를 알고 직접 입력하는 시나리오인데, Mock 서버는 등록 시 자동 생성하는 시나리오이다.

**해결 필요:** 플로우를 다음 중 하나로 통일:
- (A) 판매자가 실제 택배사에서 발급받은 송장번호를 직접 입력 (Mock 서버는 조회만 사용)
- (B) 시스템이 Mock 택배사 API로 배송 등록하면 송장번호를 자동 수신 (판매자 수동 입력 불필요)

### 4.2 [HIGH] 자동 구매확정 시점과 반품/교환 기간의 경계 충돌

**US-503:** "배송 완료 후 7일 경과 -> 자동 구매확정 (매일 00:00 배치)"
**US-504:** "배송 완료 후 7일 이내 반품 신청 가능"
**US-503:** "구매확정 후에는 반품/교환 신청이 불가능"
**US-503:** "반품/교환 진행 중인 건은 자동 구매확정에서 제외"

**충돌 시나리오:**
1. 3/22 15:00 배송 완료
2. 3/29 00:00 자동 구매확정 배치 실행 -> 7일 미경과(15시간 남음)이므로 확정 안 됨
3. 3/30 00:00 자동 구매확정 배치 실행 -> 7일 경과이므로 확정
4. 그런데 구매자는 3/29 23:00에 반품 신청을 할 수 있는가?

"7일 이내"가 정확히 168시간인지, 7일째 자정까지인지에 따라 결과가 달라짐.

### 4.3 [MEDIUM] 재고 차감 이벤트 수신 vs 동기식 재고 확인

**US-601:** "주문 생성 이벤트(`order.created`) 수신 시 재고 차감" (비동기)
**US-505:** "교환 희망 옵션의 재고가 있어야 교환 신청이 가능하다" (동기 확인 필요)

교환 시에는 재고를 동기적으로 확인해야 하는데, 주문 시에는 비동기 이벤트로 처리한다. 비동기 재고 차감 중 재고 부족이 발생하면 이미 생성된 주문을 취소해야 하는 보상 로직이 필요하지만 상세하게 정의되지 않았다.

### 4.4 [MEDIUM] 배송 상태 흐름 불일치

**US-501 데이터 모델:** shipping.status = `READY`, `IN_TRANSIT`, `DELIVERED` (3개)
**US-502 AC:** 배송 상태 = `READY` -> `IN_TRANSIT` -> `DELIVERED` (3개)
**Mock 서버 trackingSteps:** `ACCEPTED` -> `IN_TRANSIT` -> `IN_TRANSIT` -> `OUT_FOR_DELIVERY` -> `DELIVERED` (5단계)

Mock 서버의 세분화된 상태(ACCEPTED, OUT_FOR_DELIVERY)와 PRD의 단순화된 상태(READY, IN_TRANSIT, DELIVERED)의 매핑 규칙이 없다. 특히:
- Mock 서버의 `ACCEPTED`가 PRD의 `READY`에 해당하는지?
- Mock 서버의 `OUT_FOR_DELIVERY`는 PRD에서 어떤 상태?

### 4.5 [LOW] review 테이블 rating 컬럼 타입

**US-801 데이터 모델:** `rating TINYINT(1) NOT NULL COMMENT '별점 (1-5)'`

TINYINT(1)은 프로젝트 컨벤션에서 boolean 용도로 사용하기로 했다 (CLAUDE.md: "TINYINT(1) for boolean"). 별점 1-5 저장 시 TINYINT(1)을 사용하면 컨벤션과 혼동될 수 있다.
