## 1. 기능별 완전성 검증

### 1.1 배송 도메인 (shipping-service)

#### US-501: 송장 등록 + 주문 상태 변경

| 항목 | 분석 |
|------|------|
| **입력** | orderId, carrier, trackingNumber |
| **처리** | 송장 등록 -> 주문 상태 PAID->SHIPPING 변경 -> Kafka 이벤트 발행 |
| **출력** | Shipping 엔티티 (id, orderId, carrier, trackingNumber, status=READY) |

**완전성 이슈:**
1. **주문 상태 불일치**: PRD는 `PAID -> SHIPPING` 전이를 명시하나, 기존 OrderStatus enum에 `SHIPPING` 상태가 없다. 기존 상태는 `PAID -> PREPARING -> SHIPPED -> DELIVERED -> CONFIRMED`. PRD의 `SHIPPING`은 기존의 `SHIPPED`에 해당하는지, 아니면 `PREPARING`과 `SHIPPED` 사이의 새로운 상태인지 모호하다.
2. **셀러 ID 부재**: PRD의 shipping 테이블에 seller_id가 없으나, 기존 Flyway 스키마(V1__init_shipping.sql)에는 seller_id가 존재한다. 판매자별 배송 관리에 seller_id 필수.
3. **수령인 정보 부재**: PRD의 shipping 테이블에 receiver_name, receiver_phone, address가 없으나, 기존 Flyway 스키마에는 존재. 배송 라벨 출력 등에 필수.
4. **권한 검증 누락**: 판매자가 자신의 주문에 대해서만 송장 등록 가능한지에 대한 AC 누락.
5. **송장번호 형식**: "숫자 10-15자리"로 명시하나, 실제 택배사별 송장번호 형식이 다름 (CJ: 숫자 12자리, 롯데: 숫자 12자리, 로젠: 숫자 11자리, 우체국: 숫자 13자리). 택배사별 검증 규칙 필요.

#### US-502: 택배사 API 연동 배송 추적

| 항목 | 분석 |
|------|------|
| **입력** | shippingId (path parameter) |
| **처리** | 택배사 API 호출 -> 추적 이력 조회 -> Redis 캐싱 -> 배송 상태 업데이트 |
| **출력** | currentStatus, estimatedDelivery, trackingLogs[] |

**완전성 이슈:**
1. **Mock 택배사 연동 활용 미명시**: closet-external-api에 CJ/로젠/롯데/우체국 Mock 택배사가 이미 구현되어 있으나, PRD에서 이 Mock API 활용 방안을 명시하지 않음. "스마트택배 API"를 언급하나 기존 인프라를 먼저 활용해야 한다.
2. **배송 상태 세분화 부족**: PRD는 `READY -> IN_TRANSIT -> DELIVERED` 3단계만 정의하나, 기존 CarrierService의 trackingSteps는 `ACCEPTED -> IN_TRANSIT -> IN_TRANSIT -> OUT_FOR_DELIVERY -> DELIVERED` 5단계를 지원. PRD 상태와 택배사 상태 간 매핑 규칙 필요.
3. **estimatedDelivery 산출 로직 부재**: 예상 배송일 계산 방법 미정의. 택배사 API에서 제공하는지, 자체 계산인지 불명확.
4. **배송 상태 변경 -> 주문 상태 동기화**: 배송 완료(DELIVERED) 시 주문의 SHIPPED -> DELIVERED 전이를 누가 트리거하는지 명확하지 않음.
5. **조회 키**: shippingId로 조회하나, 구매자 입장에서는 orderId로 조회하는 것이 자연스러움. orderId 기반 조회 API 누락.

#### US-503: 자동 구매확정 (7일)

| 항목 | 분석 |
|------|------|
| **입력** | 배치: DELIVERED 상태 + 7일 경과 주문 / API: orderId |
| **처리** | 배치 or 수동 확정 -> DELIVERED -> CONFIRMED 상태 변경 -> Kafka 이벤트 |
| **출력** | 확정된 주문 정보 |

**완전성 이슈:**
1. **반품/교환 진행 중 판별**: "반품/교환 진행 중인 건은 제외"라고 하나, 어떤 상태가 "진행 중"인지 정의 필요. return_request, exchange_request 테이블의 어떤 status 값이 "진행 중"에 해당하는지.
2. **배치 실패 처리**: 배치 실행 중 특정 건 실패 시 나머지 건 처리 전략 (skip & continue? rollback all?).
3. **자동 확정 알림**: 구매자에게 "자동 구매확정 예정" 사전 알림 (예: D-1) 부재. 사용자 경험상 필요.
4. **정산 이벤트**: `order.confirmed` 이벤트 발행 후 정산 서비스(settlement-service) 수신 측 PRD 부재.

#### US-504: 반품 신청 + 수거 + 검수 + 환불

| 항목 | 분석 |
|------|------|
| **입력** | orderId, orderItemId, reason, reasonDetail |
| **처리** | 반품 신청 -> 수거 스케줄링 -> 수거 완료 -> 검수 -> 승인/거절 -> 환불 or 알림 |
| **출력** | ReturnRequest 엔티티 + 환불 결과 |

**완전성 이슈:**
1. **수거 스케줄링 상세 부재**: `REQUESTED -> PICKUP_SCHEDULED` 전이의 트리거가 누구인지 불명확. 판매자가 수동으로 수거 스케줄을 등록하는지, 시스템이 자동으로 택배사에 수거 요청하는지.
2. **부분 반품**: 하나의 주문에 여러 상품이 있을 때 일부만 반품하는 시나리오의 금액 계산 규칙 부재. 배송비 분담, 할인 재계산 등.
3. **반품 배송비 결제 방법**: 구매자 부담 시 3,000원을 어떻게 결제하는지 (원결제수단 차감? 별도 결제? 환불금 차감?).
4. **검수 기준/체크리스트**: 반품 검수(INSPECTING) 시 판매자가 확인해야 할 항목 정의 부재.
5. **반품 기간 기산점**: "배송 완료 후 7일"에서 "배송 완료"는 shipping 테이블의 delivered_at 기준인지, 주문의 DELIVERED 상태 전이 시점인지.
6. **교환 후 반품**: 교환으로 받은 상품의 반품 가능 여부 미정의.
7. **환불 금액 계산**: 부분 반품 시 프로모션/쿠폰 적용 상품의 환불 금액 계산 규칙 부재.

#### US-505: 교환 신청 (사이즈 교환)

| 항목 | 분석 |
|------|------|
| **입력** | orderId, orderItemId, reason, reasonDetail, exchangeOptionId |
| **처리** | 교환 신청 -> 재고 선점 -> 수거 -> 재발송 -> 완료 |
| **출력** | ExchangeRequest 엔티티 |

**완전성 이슈:**
1. **가격 차이 처리**: 교환 옵션(사이즈/색상)에 추가 가격(additionalPrice)이 다를 수 있으나, 가격 차이 처리 규칙 부재. 기존 ProductOption에 additionalPrice 필드가 존재.
2. **재고 선점 타임아웃**: 교환 요청 시 새 옵션 재고를 선점하지만, 수거 완료까지 기간이 길어질 수 있음. 선점 유지 기간 및 타임아웃 처리 미정의.
3. **교환 배송비 왕복 6,000원**: 단순변심 기준이지만, 무료배송 상품과 유료배송 상품의 차이 미고려.
4. **교환 취소**: 교환 신청 후 취소 가능 시점 및 절차 미정의. REQUESTED 상태에서만 가능한지, PICKUP_SCHEDULED에서도 가능한지.
5. **교환 재발송 송장**: 재발송 시 새로운 shipping 레코드 생성 및 추적 방법 미정의.
6. **동일 상품 재고 소진**: 교환 희망 옵션 재고가 교환 처리 중 소진될 경우의 폴백 처리 부재.

### 1.2 재고 도메인 (inventory-service)

#### US-601: SKU별 재고 관리 + 주문 시 차감

| 항목 | 분석 |
|------|------|
| **입력** | Kafka 이벤트(order.created, order.cancelled) / API(inventoryId, quantity) |
| **처리** | 재고 차감/복구 + 이력 기록 |
| **출력** | 변경된 재고 수량 + inventory_history 레코드 |

**완전성 이슈:**
1. **기존 스키마와 PRD 불일치**: 기존 Flyway 스키마는 `inventory_item` (total_quantity, available_quantity, reserved_quantity) 3단 구조이나, PRD는 `inventory` (quantity) 단일 수량. 기존 스키마가 더 정교함. **기존 스키마를 기준으로 구현해야 한다.**
2. **SKU 매핑**: PRD는 별도 sku 컬럼을 두나, 기존 스키마는 product_option_id를 UK로 사용. ProductOption.skuCode와의 관계 정리 필요.
3. **초기 재고 등록 API 부재**: 상품 등록 시 재고를 어떻게 초기 설정하는지 (product.created 이벤트 연동? 수동 입고?). 입고(INBOUND) API 스펙 부재.
4. **재고 예약(Reserve) vs 차감(Deduct)**: 기존 스키마는 RESERVE/RELEASE 개념이 있으나 PRD는 직접 차감만 명시. 주문 생성 시 예약 -> 결제 완료 시 확정 -> 결제 실패 시 해제 패턴이 기존 구조에 더 적합.
5. **배치 재고 차감**: 주문에 여러 OrderItem이 있을 때 여러 SKU를 한 번에 차감하는 방법 (all-or-nothing? partial?).

#### US-602: Redis 분산 락 기반 동시성 제어

| 항목 | 분석 |
|------|------|
| **입력** | 재고 차감/복구 요청 |
| **처리** | Redisson 분산 락 획득 -> 재고 변경 -> JPA 낙관적 락 2차 방어 |
| **출력** | 정합성 보장된 재고 수량 |

**완전성 이슈:**
1. **기존 version 컬럼**: 이미 Flyway 스키마에 `version BIGINT NOT NULL DEFAULT 0` 존재. DDL 추가 불필요.
2. **락 범위**: SKU 단위 락이므로 여러 SKU를 동시 차감할 때 데드락 가능성. 락 획득 순서 규칙(SKU 코드 오름차순 등) 필요.
3. **Redisson 설정**: Redis Sentinel/Cluster 환경에서의 Redisson 락 동작 명시 필요.
4. **성능 지표**: 100스레드 동시성 테스트 기준만 있고, 목표 TPS/응답시간 미정의.

#### US-603: 안전재고 알림

| 항목 | 분석 |
|------|------|
| **입력** | 재고 차감 이벤트 (내부) / 안전재고 설정 API |
| **처리** | 잔여 수량 <= safety_stock -> 알림 이벤트 -> 중복 방지 (Redis 24h) |
| **출력** | Kafka 이벤트 (inventory.low_stock / inventory.out_of_stock) |

**완전성 이슈:**
1. **알림 수신 채널**: Kafka 이벤트 발행까지만 정의. 판매자에게 실제 알림 전달(이메일/SMS/앱푸시) 방법은 notification-service에 위임하는 것으로 보이나, 해당 서비스 PRD 부재.
2. **안전재고 설정 권한**: 판매자만 설정 가능한지, 관리자도 가능한지.
3. **재고 0 시 상품 상태 변경**: out_of_stock 이벤트 수신 시 상품 노출 중단(품절 표시) 처리가 product-service 측에 필요하나 미정의.

#### US-604: 재입고 알림

| 항목 | 분석 |
|------|------|
| **입력** | 알림 신청: productId, optionId / 재입고 이벤트 수신 |
| **처리** | 알림 신청 저장 -> 재고 입고 시 WAITING 대상 조회 -> 알림 발행 -> NOTIFIED 전이 |
| **출력** | Kafka 이벤트 (inventory.restock_notification) |

**완전성 이슈:**
1. **입고 이벤트 트리거**: 재고 입고(INBOUND) 이벤트를 누가 발행하는지 불명확. 입고 API가 US-601에 명시되어 있지 않음.
2. **대량 알림 처리**: 인기 상품 재입고 시 수만 건의 알림이 발생할 수 있음. 배치 발행 전략 필요.
3. **알림 신청 조회 API 부재**: 본인의 재입고 알림 신청 목록 조회 API 누락.
4. **품절이 아닌 상품에 대한 신청 방지**: 재고가 있는 상품에 재입고 알림을 신청하는 것을 방지하는 검증 로직 필요.

### 1.3 검색 도메인 (search-service)

#### US-701: Elasticsearch 상품 인덱싱 (Kafka CDC)

| 항목 | 분석 |
|------|------|
| **입력** | Kafka 이벤트 (product.created/updated/deleted) |
| **처리** | Elasticsearch 인덱싱 (nori 분석기) |
| **출력** | closet-products 인덱스 문서 |

**완전성 이슈:**
1. **product 이벤트 발행 측**: closet-product 서비스에서 Kafka 이벤트를 발행하는 코드가 기존에 있는지 확인 필요. 없다면 product-service 수정도 Phase 2 스코프.
2. **인덱싱 필드 중 salesCount/reviewCount/avgRating**: 이 필드들은 product-service가 아닌 order-service, review-service에서 관리. 초기 인덱싱 시 이 데이터를 어디서 가져오는지, 업데이트 주기는 어떻게 되는지.
3. **DLQ 재처리 전략**: DLQ에 저장된 실패 메시지의 수동/자동 재처리 방법 미정의.
4. **인덱스 버저닝/무중단 재인덱싱**: 스키마 변경 시 Blue-Green 인덱싱 전략 부재.

#### US-702: 한글 형태소 분석 (nori) + 키워드 검색

**완전성 이슈:**
1. **유의어 사전 관리**: "바지=팬츠" 등 유의어를 어디서 관리하는지. Elasticsearch synonym filter 설정? DB? 파일?
2. **오타 교정 로직**: "suggestion" 필드가 응답에 있으나, 구현 방법(fuzzy query? did-you-mean?) 미정의.
3. **검색 결과 캐싱**: 동일 검색어에 대한 캐싱 전략 부재.
4. **검색 로그**: 검색어 로깅 (US-705 인기 검색어용) 위치 미정의.

#### US-703: 필터 (카테고리/브랜드/가격/색상/사이즈)

**완전성 이슈:**
1. **facet count 성능**: 대용량 데이터에서 모든 필터의 facet count를 매번 계산하면 성능 이슈. 캐싱 또는 approximate count 전략 필요.
2. **카테고리 계층 구조**: 대/중/소 카테고리 코드 체계가 기존 closet-product의 Category 엔티티와 일치하는지 확인 필요.
3. **재고 연동 필터**: 품절 상품 제외 필터 미포함. 구매자 경험상 중요.

#### US-704: 자동완성

**완전성 이슈:**
1. **50ms 응답 목표**: edge_ngram + nori 조합의 인덱스 설계가 이 목표를 달성할 수 있는지 성능 검증 필요.
2. **최근 검색어**: 사용자 개인의 최근 검색어 기능 미포함. 무신사 벤치마킹 시 필수.
3. **검색어 자동완성 vs 상품 자동완성**: 검색어 텍스트만 반환하는지, 상품 미리보기까지 제공하는지.

#### US-705: 인기 검색어

**완전성 이슈:**
1. **금칙어 목록 관리**: 금칙어 CRUD API 부재. 관리자가 어디서 관리하는지.
2. **어뷰징 방지**: 동일 사용자의 반복 검색으로 인기 검색어 조작 방지 로직 미정의.
3. **sliding window 구현**: 1시간 단위 Redis Sorted Set 관리의 구체적 키 설계 미정의.

### 1.4 리뷰 도메인 (review-service)

#### US-801: 텍스트 + 포토 리뷰

**완전성 이슈:**
1. **이미지 업로드 선행**: multipart/form-data로 리뷰+이미지를 동시 전송하나, 대용량 이미지 처리 시 타임아웃 우려. presigned URL 방식 검토 필요.
2. **리뷰 수정 시 이미지 처리**: 기존 이미지 삭제/추가/교체 시나리오 미정의.
3. **리뷰 신고/블라인드**: 부적절한 리뷰 신고 및 관리자 블라인드 처리 기능 부재.
4. **주문 확인 검증**: 구매확정 상태 확인을 위해 order-service 동기 호출 필요. 서비스 간 의존성 관리 방안 부재.

#### US-802: 사이즈 후기 (키/몸무게/핏 평가)

**완전성 이슈:**
1. **개인정보 동의**: 키/몸무게는 민감 정보. 개인정보 수집 동의 절차 미정의.
2. **"나와 비슷한 체형" 필터 정확도**: +-5 범위가 키(cm)와 몸무게(kg) 모두에 적용되는지 불명확.
3. **사이즈 추천 알고리즘**: "정사이즈 추천" 판단 기준이 PERFECT 비율 기반인지, 다른 알고리즘인지.

#### US-803: 리뷰 포인트 적립 (100P/300P)

**완전성 이슈:**
1. **포인트 회수 시점**: 리뷰 삭제 시 즉시 회수인지, 배치 회수인지.
2. **일일 한도 5,000P**: 자정 기준 리셋인지, 24시간 rolling window인지.
3. **member-service 포인트 기능**: Phase 1 member-service에 포인트 적립/차감 API가 있는지 확인 필요. 없다면 Phase 2 스코프에 포함해야 함.

#### US-804: 리뷰 집계 (평균 별점, 사이즈 분포)

**완전성 이슈:**
1. **집계 데이터 정합성**: 실시간 갱신 vs 배치 보정 간 정합성 보장 방법 미정의.
2. **ES 동기화 주기**: review.summary.updated 이벤트로 ES를 업데이트하나, 리뷰가 빈번하면 ES 업데이트가 과도. 디바운싱/배치 전략 필요.

### 3.4 역할별 누락 기능

- **판매자**: 상품별 재고 입고(INBOUND) 직접 등록 기능 부재. WMS 연동 전까지 수동 입고 필수.

- **관리자**: 재고 수동 조정(ADJUST) 기능이 PRD에 없음. 재고 실사 후 보정 필수.
- **구매자**: 배송 알림(카카오 알림톡 등) 수신 설정 부재.

## 6. 누락 기능 발견

### 6.1 WMS (창고관리시스템) -- 상세 구현 필요

PRD에 WMS 관련 기능이 전혀 포함되어 있지 않다. 올리브영 GMS 벤치마킹 기반으로 다음 기능이 추가 필요하다.

#### WMS-001: 입고 관리 (Inbound/Receiving)

- **입고 예정(ASN - Advanced Shipping Notice)**: 셀러/브랜드가 물류센터로 발송 예정 정보 등록
- **입고 검수**: 수량 확인, 불량 검수, 바코드 스캔
- **입고 확정**: 검수 통과 상품 재고 반영 (inventory_item.total_quantity += 입고수량)
- **입고 반려**: 불량/오배송 상품 셀러 반송
- **입고 이력**: 입고번호, 입고일, 상품/SKU별 수량, 검수 결과 기록

```
상태 흐름: EXPECTED -> ARRIVED -> INSPECTING -> CONFIRMED / REJECTED
```

필요 테이블:
- `inbound_order` (입고 주문)
- `inbound_order_item` (입고 상품 항목)
- `inbound_inspection` (입고 검수 결과)

#### WMS-002: 보관/로케이션 관리 (Storage/Location)

- **로케이션 체계**: 창고(Warehouse) > 구역(Zone) > 통로(Aisle) > 선반(Rack) > 셀(Cell)
- **로케이션 코드**: `W01-Z01-A01-R01-C01` 형식
- **상품-로케이션 매핑**: SKU별 보관 위치 관리
- **로케이션 유형**: 일반, 냉장, 대형, 귀중품
- **재고 로케이션**: 동일 SKU가 여러 로케이션에 분산 보관 가능

필요 테이블:
- `warehouse` (창고)
- `location` (로케이션)
- `location_inventory` (로케이션별 재고)

#### WMS-003: 피킹 (Picking)

- **피킹 지시서 생성**: 주문 확정 시 피킹 지시서 자동 생성
- **피킹 유형**: 단건 피킹, 배치 피킹(여러 주문 묶어서), 웨이브 피킹(시간대별)
- **피킹 경로 최적화**: 로케이션 순서 최적화
- **피킹 확인**: 바코드 스캔으로 피킹 확인
- **부분 피킹**: 재고 부족 시 가용 수량만 피킹

```
상태 흐름: CREATED -> ASSIGNED -> IN_PROGRESS -> COMPLETED / PARTIAL / CANCELLED
```

필요 테이블:
- `pick_order` (피킹 지시서)
- `pick_order_item` (피킹 항목)
- `pick_assignment` (피킹 작업자 배정)

#### WMS-004: 출고 (Outbound/Packing)

- **패킹 (포장)**: 피킹 완료 상품 검품 + 포장
- **포장재 선택**: 상품 크기/수량에 따른 포장재 자동 추천
- **출고 검수**: 상품/수량 재확인, 바코드 검증
- **송장 출력**: 택배사 API 연동 송장 자동 생성
- **출고 확정**: 택배사 인수 확인

```
상태 흐름: PICKED -> PACKING -> INSPECTED -> SHIPPED
```

필요 테이블:
- `outbound_order` (출고 주문)
- `outbound_order_item` (출고 항목)
- `packing_info` (포장 정보)

#### WMS-005: 재고 실사 (Stock-taking/Cycle Count)

- **정기 실사**: 월별/분기별 전체 재고 실사
- **순환 실사(Cycle Count)**: ABC 분석 기반 주기적 부분 실사
- **실사 절차**: 실사 계획 -> 카운트 -> 차이 분석 -> 조정 승인 -> 재고 보정
- **불일치 보고**: 장부 수량 vs 실물 수량 차이 분석
- **재고 조정(Adjustment)**: 실사 결과에 따른 재고 수량 보정

```
상태 흐름: PLANNED -> IN_PROGRESS -> COUNTED -> REVIEWING -> APPROVED / REJECTED
```

필요 테이블:
- `stock_take` (재고 실사)
- `stock_take_item` (실사 항목)
- `stock_adjustment` (재고 조정)

#### WMS-006: 반품 입고 (Return Receiving)

- **반품 수거 상품 입고 검수**: 반품 상품 상태 확인 (정상/파손/오염)
- **재판매 판정**: 재판매 가능 여부 판단
- **재입고**: 재판매 가능 시 가용 재고 복구
- **불량 처리**: 재판매 불가 시 폐기/B급 상품 전환
- **교환 출고**: 교환 건의 신규 상품 출고 연결

필요 테이블:
- `return_receiving` (반품 입고)
- `return_inspection_result` (반품 검수 결과)

## 7. 분석 요약

### 7.1 PRD 품질 평가

| 항목 | 평가 | 상세 |
|------|------|------|
| User Story 형식 | 양호 | As-a / I-want-to / So-that 구조 일관 |
| AC 구체성 | 보통 | Happy path는 명확하나 Unhappy path 부족 |
| 데이터 모델 | 미흡 | 기존 Flyway 스키마와 불일치. 기존 스키마가 더 정교함 |
| API 스펙 | 양호 | RESTful, 요청/응답 명세 구체적 |
| 이벤트 흐름 | 양호 | 서비스 간 Kafka 이벤트 흐름도 제공 |
| 기존 코드 연계 | 미흡 | Phase 1 코드/스키마 반영 부족 |
| WMS | 부재 | 창고관리시스템 전혀 미포함 |

### 7.2 핵심 리스크

1. **데이터 모델 이중화**: PRD 스키마와 기존 Flyway 스키마가 다름. 기존 스키마 기준으로 구현하되 PRD의 비즈니스 요구사항은 반영해야 함.
2. **주문 상태 머신 확장**: Phase 1의 `OrderStatus`에 배송/반품/교환 상태를 어떻게 통합할지 설계 필요. 주문 레벨 상태 vs 아이템 레벨 상태 분리 전략.
3. **WMS 스코프 확대**: 입고/보관/피킹/출고/재고실사 전체 구현은 상당한 공수. 별도 Sprint 또는 Phase 2.5로 분리 검토 필요.
4. **서비스 간 의존성**: 배송<->주문<->재고<->결제 간 이벤트 기반 통합의 보상 트랜잭션(SAGA) 설계 필요.
5. **Phase 1 서비스 수정**: product-service(이벤트 발행), payment-service(환불), member-service(포인트), order-service(상태 확장) 수정이 선행되어야 함.

### 7.3 다음 단계 제안

1. WMS 상세 PRD 별도 작성 (입고/보관/피킹/출고/재고실사)
2. 기존 Flyway 스키마와 PRD 데이터 모델 통합 (기존 스키마 기준)
3. OrderStatus 확장 설계 (주문 레벨 vs 아이템 레벨 상태 분리)
4. 서비스 간 이벤트 흐름 SAGA 패턴 상세 설계
