## 최우선 결정 (Q-01 ~ Q-06)

### Q-01: Mock 서버 연동 방식
**결정: (B) 각 택배사 개별 API 직접 호출**
- shipping-service에 CarrierClient 인터페이스 + 택배사별 구현체(Strategy 패턴)
- "스마트택배" 통합 API는 사용하지 않음 — Mock 서버가 이미 개별 API로 구현됨
- 택배사 추가 시 구현체만 추가하면 되므로 확장성 확보

### Q-02: 송장 등록 플로우
**결정: (B) 시스템 자동 채번**
- 판매자가 "배송 시작" 버튼 → shipping-service가 Mock 택배사 API에 배송 등록 → 송장번호 자동 수신
- 판매자 수동 입력 옵션도 병행 지원 (기존 송장이 있는 경우)
- 송장 형식 검증: `[A-Z]{2}[0-9]{10,15}` (prefix 허용으로 변경)

### Q-03: 반품 사유별 배송비 부담
**결정: 아래 매핑 테이블 확정**

| 사유 | 배송비 부담 | 금액 |
|------|-----------|------|
| DEFECTIVE (불량) | SELLER | 0원 (판매자 부담) |
| WRONG_ITEM (오배송) | SELLER | 0원 (판매자 부담) |
| SIZE_MISMATCH (사이즈 불일치) | BUYER | 3,000원 |
| CHANGE_OF_MIND (단순변심) | BUYER | 3,000원 |

- 교환 배송비: 반품 3,000원 + 재배송 3,000원 = 6,000원 (왕복)
- 배송비는 DB `shipping_fee_policy` 테이블로 관리 (하드코딩 X)

### Q-04: 구매확정/반품 기한 "7일" 기준
**결정: 168시간(시각 기준)**
- 배송 완료 시각(`delivered_at`) 기준 정확히 168시간
- 자동 구매확정 배치: 매일 00:00 + 12:00 (2회)
- **반품/교환이 구매확정보다 우선**: 168시간 만료 시점에 반품 신청이 접수 중이면 구매확정 배제
- 배치 쿼리: `WHERE delivered_at <= NOW() - INTERVAL 168 HOUR AND status NOT IN ('RETURN_REQUESTED', 'EXCHANGE_REQUESTED')`

### Q-05: 재고 차감 시점
**결정: 주문 생성 시점 (order.created) 차감, 결제 실패 시 자동 복구**
- 주문 생성 → Kafka `order.created` → inventory-service 재고 차감 (reserved++)
- 결제 완료 → `order.paid` → 재고 확정 (reserved--, available 유지)
- 결제 실패/주문 취소 → `order.cancelled` → 재고 복구 (reserved--, available++)
- **15분 내 결제 미완료 시 자동 취소** (주문 TTL)
- 부분 재고 부족: **All or Nothing** — 전체 주문 거절

### Q-06: WMS 구현 범위
**결정: Phase 2에 포함. inventory-service와 통합 모듈로 구현**
- closet-wms를 별도 모듈로 분리하지 않고, closet-inventory 내에 wms 패키지로 구현
- Phase 2 Sprint 5-6: inventory 기본 + WMS 입고/출고
- Phase 2 Sprint 7-8: WMS 피킹/재고실사 + 검색/리뷰
- 올리브영 GMS 벤치마킹 기반

## 높은 우선순위 결정 (Q-07 ~ Q-14)

### Q-07: 반품 배송비 공제 방식
**결정: 환불 시 배송비 공제 후 환불**
- 구매자 부담인 경우: `환불금액 = 결제금액 - 반품배송비(3,000원)`
- PG사에 부분 취소(cancelAmount) 요청

### Q-08: 교환 시 가격 차이
**결정: 동일 상품 = 동일 가격. 차액 없음**
- 옵션(사이즈/색상)별 가격 차이가 있는 경우 교환 불가 → 반품 후 재주문 안내
- MVP 단계에서는 동일 가격 교환만 지원

### Q-09: 반품 검수 기준
**결정: 판매자 검수 (물류센터 대행)**
- 반품 수거 완료 → INSPECTING 상태 → 판매자 3영업일 내 승인/거절
- 3영업일 초과 시 자동 승인 (Auto-approve)
- 검수 기준: 태그 부착 여부, 사용 흔적, 포장 상태

### Q-10: 부분 재고 부족 시 처리
**결정: All or Nothing**
- 5개 상품 중 1개라도 재고 부족이면 전체 주문 거절
- `inventory.insufficient` 이벤트 발행 → order-service에서 주문 실패 처리
- 사용자에게 "재고 부족 상품: {상품명}" 안내

### Q-11: 인기순(POPULAR) 정렬 기준
**결정: 복합 점수 (최근 30일 기준)**
- `score = 판매량 * 0.4 + 리뷰 수 * 0.3 + 평균 별점 * 0.2 + 조회수 * 0.1`
- 매일 새벽 배치로 ES 인덱스의 `popularity_score` 필드 갱신
- Sprint 5에서는 판매량 기준 단순 정렬, Sprint 7에서 복합 점수로 고도화

### Q-12: 유의어 사전 관리
**결정: DB 테이블 + ES synonym filter**
- `search_synonym` 테이블: (word, synonyms, created_at)
- 초기 데이터: 패션 도메인 유의어 100개 seed (바지=팬츠, 상의=탑, 아우터=자켓 등)
- 관리자 API로 CRUD, 변경 시 ES 인덱스 리로드

### Q-13: 리뷰 수정 범위
**결정: 텍스트 + 이미지 수정 가능, 별점 수정 불가**
- 별점 변경 시 집계 재계산 복잡도가 높으므로 Phase 2에서는 별점 고정
- 수정 이력: `review_edit_history` 테이블로 보존
- 수정 횟수 제한: 최대 3회

### Q-14: 이미지 리사이즈 방식
**결정: 서비스 내부 동기 처리**
- AWS Lambda 비용 대비 학습 프로젝트에서는 과도
- `Thumbnailator` 라이브러리로 서비스 내 동기 리사이즈 (400x400 썸네일)
- 원본 + 썸네일 모두 로컬 스토리지 저장 (S3는 Phase 3)

- Consumer: SELECT → INSERT (UNIQUE KEY) → 비즈니스 로직 → COMMIT
