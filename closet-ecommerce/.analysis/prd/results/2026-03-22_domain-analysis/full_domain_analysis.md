## 1. 비즈니스 도메인 전체 맵

### 1.1 무신사 비즈니스 모델 분석 (B2C 마켓플레이스)

무신사는 **패션 특화 온라인 마켓플레이스**로, 브랜드(셀러)와 소비자를 연결하는 플랫폼이다.
핵심 비즈니스 모델은 다음과 같다:

| 구분 | 설명 |
|------|------|
| 플랫폼 유형 | B2B2C 마켓플레이스 (셀러가 입점하여 소비자에게 판매) |
| 핵심 가치 | 패션 큐레이션 + 가격 경쟁력 + 커뮤니티 |
| 차별점 | 스트릿/캐주얼 패션 특화, 코디 콘텐츠, 리뷰 문화 |
| 거래 방식 | 위탁판매(셀러 직배송) + 직매입(무신사 스탠다드 등 PB) |
| 타겟 | 10~30대 남녀 (초기 남성 → 현재 유니섹스 확장) |

**플랫폼 참여자 구조:**

```
┌─────────────────────────────────────────────────┐
│                  Closet Platform                │
│                                                 │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐  │
│  │  브랜드   │    │ 플랫폼   │    │  구매자   │  │
│  │ (셀러)   │────│ (Closet) │────│ (회원)   │  │
│  │          │    │          │    │          │  │
│  │ 상품등록  │    │ 큐레이션  │    │ 탐색/구매 │  │
│  │ 재고관리  │    │ 정산     │    │ 리뷰작성  │  │
│  │ CS응대   │    │ CS중재   │    │ 코디공유  │  │
│  └──────────┘    └──────────┘    └──────────┘  │
└─────────────────────────────────────────────────┘
```

### 1.3 사용자 유형별 Journey Map

#### 구매자 Journey

```
[인지] → [탐색] → [비교] → [의사결정] → [구매] → [사용] → [피드백]

1. 인지
   - SNS 광고 (인스타그램, 유튜브)
   - 인플루언서 코디 포스팅
   - 지인 추천

2. 탐색
   - 앱/웹 접속
   - 매거진/OOTD 스냅 탐색
   - 카테고리 브라우징
   - 키워드 검색 ("남자 봄 자켓")
   - 랭킹 확인 (실시간/주간/월간)

3. 비교
   - 상품 상세 페이지 (사이즈표, 핏 정보)
   - 리뷰 확인 (사이즈 후기 - 키/몸무게/평소사이즈/선택사이즈)
   - 코디 제안 (이 상품과 함께 구매한 상품)
   - 가격 비교 (할인율, 쿠폰 적용가)

4. 의사결정
   - 위시리스트 등록 (가격 변동 알림)
   - 쿠폰 확인/다운로드
   - 사이즈 선택 고민 (AI 사이즈 추천)

5. 구매
   - 장바구니 담기 (브랜드별 묶음)
   - 배송비 정책 확인 (브랜드별 무료배송 기준)
   - 결제 수단 선택 (카드/간편결제/가상계좌)
   - 쿠폰/적립금 적용
   - 주문 완료

6. 사용
   - 배송 추적
   - 상품 수령 및 착용
   - 사이즈 교환/반품 (단순변심 7일, 불량 14일)

7. 피드백
   - 리뷰 작성 (텍스트/포토/영상 + 사이즈 정보)
   - 적립금 획득
   - 코디 스냅 업로드
   - 재구매 / 브랜드 팔로우
```

#### 셀러 Journey

```
[입점] → [상품관리] → [판매] → [CS] → [정산] → [성장]

1. 입점
   - 입점 신청서 제출 (브랜드 정보, 사업자 등록증)
   - 심사 (상품 퀄리티, 브랜드 적합성)
   - 계약 체결 (수수료율, 정산 주기)
   - 파트너센터 계정 발급

2. 상품관리
   - 상품 등록 (카테고리, 옵션, 이미지, 상세 설명)
   - 사이즈 가이드 입력
   - 재고 관리 (SKU별)
   - 가격 정책 설정 (정가, 할인가, 시즌오프)

3. 판매
   - 주문 접수 확인
   - 출고 처리 (송장 입력)
   - 기획전/프로모션 참여

4. CS
   - 고객 문의 답변 (48시간 이내 의무)
   - 반품/교환 승인 및 처리
   - 불량/오배송 클레임 대응

5. 정산
   - 구매확정 건 자동 정산 (D+7 기준)
   - 수수료 차감 후 정산금 입금
   - 정산 내역 확인 (파트너센터)

6. 성장
   - 판매 통계 분석 (매출, 전환율, 반품률)
   - 브랜드관 운영
   - 광고 집행 (검색 광고, 배너)
   - 리뷰 관리
```

### 1.4 전체 비즈니스 플로우

```
[셀러: 상품 등록]
       │
       ▼
[플랫폼: 상품 검수/승인]
       │
       ▼
[전시: 카테고리 배치, 검색 인덱싱, 랭킹 반영]
       │
       ▼
[구매자: 탐색 & 검색]
       │
       ▼
[구매자: 장바구니 담기]
       │
       ▼
[주문 생성] ──── 재고 예약 (Inventory)
       │
       ▼
[결제 요청] ──── PG사 승인 (Payment)
       │
       ├── 결제 실패 → 재고 복원, 주문 취소
       │
       ▼
[결제 완료] ──── 셀러 알림 (Notification)
       │              쿠폰 사용 처리 (Promotion)
       │              적립금 차감 (Member)
       ▼
[셀러: 출고 처리] ──── 송장 입력 (Shipping)
       │
       ▼
[배송 중] ──── 배송 추적 (택배사 API)
       │
       ▼
[배송 완료] ──── 자동 구매확정 대기 (D+7)
       │
       ├── 반품/교환 접수 → 수거 → 검수 → 환불/재발송
       │
       ▼
[구매확정] ──── 정산 대상 등록 (Settlement)
       │          리뷰 작성 가능 (Review)
       │          포인트 적립 (Member)
       ▼
[정산] ──── 수수료 차감 → 셀러 정산금 지급
```

## 2. 도메인별 상세 분석 (15개 도메인)

### 2.1 Product (상품) 도메인

#### 도메인 책임과 범위

상품 도메인은 **플랫폼에서 판매되는 모든 상품의 카탈로그 정보를 관리**한다.
상품의 등록, 수정, 승인, 전시 상태 관리, 옵션/SKU 구조, 카테고리 분류, 브랜드 정보를 포함한다.

**범위 포함:**
- 상품 기본 정보 (이름, 설명, 이미지, 가격)
- 카테고리 트리 (대/중/소분류)
- 상품 옵션 (사이즈 × 색상 조합)
- SKU (Stock Keeping Unit) 관리
- 브랜드 정보
- 사이즈 가이드
- 상품 상태 관리 (임시저장/심사중/판매중/품절/판매중지)

**범위 제외:**
- 재고 수량 관리 (→ Inventory)
- 가격 할인/쿠폰 (→ Promotion)
- 검색 인덱싱 (→ Search)
- 전시 순서/배치 (→ Display)

#### 핵심 비즈니스 규칙

1. **상품 등록 규칙**
   - 상품명은 2~100자, 브랜드명을 포함할 수 없다 (별도 필드)
   - 대표 이미지 최소 1장, 최대 10장 필수
   - 상세 설명 이미지 최소 1장 필수
   - 최소 1개 이상의 옵션 조합(SKU)이 존재해야 판매 가능
   - 정가(listPrice) >= 판매가(sellingPrice) 불변
   - 판매가는 0보다 커야 한다

2. **카테고리 규칙**
   - 최대 3depth (대분류 → 중분류 → 소분류)
   - 상품은 반드시 소분류(leaf) 카테고리에만 등록 가능
   - 카테고리 변경 시 해당 카테고리의 수수료율이 적용

3. **옵션/SKU 규칙**
   - 옵션은 최대 2축 (사이즈 × 색상)
   - 각 옵션 조합마다 고유 SKU 코드 부여
   - SKU 코드 형식: `{브랜드코드}-{카테고리코드}-{시퀀스}-{옵션코드}`
   - 동일 옵션 조합의 중복 SKU 불가

4. **상품 상태 전이 규칙**
   ```
   DRAFT → PENDING_REVIEW → APPROVED → ON_SALE
                  │                       │
                  ▼                       ▼
              REJECTED              SOLD_OUT (자동: 전 SKU 재고 0)
                  │                       │
                  ▼                       ▼
               DRAFT              ON_SALE (재입고 시)
                                        │
                                        ▼
                                  DISCONTINUED (판매중지)
   ```

5. **브랜드 규칙**
   - 승인된 셀러만 브랜드 등록 가능
   - 브랜드명 중복 불가 (대소문자 무시)
   - PB 브랜드는 플랫폼만 등록 가능

6. **시즌 규칙**
   - 시즌 형식: `{연도}{시즌코드}` (예: 2026SS, 2026FW)
   - 시즌코드: SS(봄여름), FW(가을겨울), PRE_SS, PRE_FW

#### 불변 조건 (Invariants)

- `Product.listPrice >= Product.sellingPrice > 0`
- `Product.skuList.size() >= 1` (판매 상태일 때)
- `Product.categoryId`는 반드시 leaf 카테고리를 가리켜야 한다
- `Product.brandId`는 반드시 승인된(APPROVED) 브랜드를 참조해야 한다
- `SKU.optionCombination`은 동일 Product 내에서 유일해야 한다
- 상품 상태가 ON_SALE이면 대표 이미지가 1장 이상이어야 한다

#### 도메인 용어 사전 (Ubiquitous Language)

| 용어 | 영문 | 정의 |
|------|------|------|
| 상품 | Product | 판매를 위해 등록된 하나의 아이템. 여러 SKU를 가질 수 있다 |
| SKU | Stock Keeping Unit | 옵션 조합별 최소 관리 단위. 색상×사이즈 = 1 SKU |
| 카테고리 트리 | Category Tree | 대/중/소로 계층화된 상품 분류 체계 |
| 리프 카테고리 | Leaf Category | 자식이 없는 최하위 카테고리. 상품은 여기에만 등록 |
| 브랜드 | Brand | 상품을 만드는 패션 브랜드. 셀러가 등록, 플랫폼이 승인 |
| 옵션 | Option | 사이즈, 색상 등 상품의 선택 가능한 변형 축 |
| 옵션값 | Option Value | 옵션 축의 구체적 값 (예: S, M, L / 블랙, 화이트) |
| 정가 | List Price | 원래 판매 가격 (할인 전) |
| 판매가 | Selling Price | 실제 소비자 구매 가격 (셀러 설정 할인 후) |
| 사이즈 가이드 | Size Guide | 실측 사이즈 정보 (총장, 어깨너비, 가슴단면 등) |
| 핏 | Fit | 의류의 실루엣 유형 (오버핏, 레귤러핏, 슬림핏) |
| 시즌 | Season | 상품이 속한 판매 시즌 (2026SS, 2026FW 등) |
| 상품 상태 | Product Status | 상품의 라이프사이클 단계 (임시저장/심사중/판매중 등) |
| 대표 이미지 | Main Image | 목록/검색에 노출되는 상품 대표 사진 |
| 상세 이미지 | Detail Image | 상품 상세 페이지에 보여지는 설명 이미지 |

#### 다른 도메인과의 관계

| 대상 도메인 | 관계 유형 | 방향 | 통신 방식 | 설명 |
|------------|----------|------|----------|------|
| Inventory | 데이터 참조 | Product → Inventory | Kafka Event | 상품/SKU 생성 시 재고 초기화 이벤트 발행 |
| Search | 데이터 동기화 | Product → Search | Kafka Event | 상품 변경 시 검색 인덱스 업데이트 이벤트 |
| Display | 데이터 참조 | Display → Product | Sync API | 전시에서 상품 정보 조회 |
| Order | 데이터 참조 | Order → Product | Sync API | 주문 시 상품 정보 스냅샷 |
| Review | 데이터 참조 | Review → Product | Sync API | 리뷰에서 상품 정보 조회 |
| Seller | 소유 관계 | Seller → Product | Sync API | 셀러가 상품을 등록/관리 |
| Promotion | 데이터 참조 | Promotion → Product | Sync API | 프로모션 대상 상품 지정 |

#### 핵심 유스케이스

1. **상품 등록 (셀러)**
   - 셀러가 파트너센터에서 상품 정보 입력
   - 카테고리 선택 → 옵션 설정 → SKU 생성 → 이미지 업로드 → 임시저장/심사요청
   - 심사 승인 시 ProductApprovedEvent 발행 → Inventory 초기화 + Search 인덱싱

2. **상품 옵션/SKU 변경**
   - 판매중 상품의 옵션 추가는 가능하나 삭제는 재고 0일 때만 가능
   - SKU 추가 시 SkuCreatedEvent 발행

3. **상품 가격 변경**
   - 판매가 변경 시 PriceChangedEvent 발행
   - 위시리스트 등록자에게 가격 변동 알림 트리거

4. **상품 판매 중지**
   - 셀러 요청 또는 플랫폼 관리자가 판매 중지 처리
   - 미출고 주문이 있으면 판매 중지 불가 (확인 필요)
   - ProductDiscontinuedEvent 발행 → Search에서 제거

5. **카테고리 관리 (관리자)**
   - 카테고리 추가/수정/비활성화
   - 비활성화 시 해당 카테고리 상품 자동 판매중지

#### 엣지 케이스 / 예외 상황

- **이미지 업로드 실패**: 비동기 처리로 S3 업로드 실패 시 재시도 3회 후 알림
- **카테고리 변경 시 수수료 충돌**: 기존 주문은 기존 수수료율 유지, 신규 주문부터 변경
- **동시 수정 충돌**: 옵티미스틱 락(version 필드)으로 동시 수정 방지
- **대량 상품 일괄 등록**: 엑셀 업로드 → 비동기 처리 → 결과 알림

### 2.2 Order (주문) 도메인

#### 도메인 책임과 범위

주문 도메인은 **구매자의 구매 의사 표현부터 주문 완료까지의 전체 라이프사이클을 관리**한다.
장바구니, 주문 생성, 주문 상태 관리, 부분 취소/교환 처리를 포함한다.

**범위 포함:**
- 장바구니 관리 (담기, 수량 변경, 삭제)
- 주문서 생성 (주문 번호, 주문 항목, 배송지, 가격 계산)
- 주문 상태 머신 (주문완료 → 결제완료 → 배송중 → 배송완료 → 구매확정)
- 부분 취소 / 전체 취소
- 교환 요청
- 주문 이력 관리

**범위 제외:**
- 결제 처리 (→ Payment)
- 재고 차감/복원 (→ Inventory)
- 배송 추적 (→ Shipping)
- 배송비 계산은 포함하되, 배송비 정책 자체는 Seller 도메인과 협력

#### 핵심 비즈니스 규칙

1. **장바구니 규칙**
   - 동일 SKU는 수량 합산 (최대 10개)
   - 장바구니 최대 100개 상품
   - 판매중지/품절 상품은 장바구니에서 비활성 표시
   - 비회원 장바구니는 로컬스토리지, 로그인 시 병합

2. **주문 생성 규칙**
   - 주문번호 형식: `ORD-{yyyyMMdd}-{랜덤8자리}` (예: ORD-20260322-A1B2C3D4)
   - 한 주문에 여러 셀러 상품 포함 가능 → 셀러별 배송비 개별 계산
   - 주문 시점에 상품 정보 스냅샷 저장 (이후 상품 변경에 영향받지 않도록)
   - 주문 최소 금액: 없음 (배송비는 셀러 정책에 따름)
   - 주문 총액 = Σ(상품금액) + Σ(배송비) - 쿠폰할인 - 적립금사용

3. **주문 상태 전이 규칙**
   ```
   ORDER_CREATED → PAYMENT_COMPLETED → PREPARING → SHIPPED → DELIVERED → PURCHASE_CONFIRMED
        │                │                │            │           │
        ▼                ▼                ▼            ▼           ▼
   ORDER_CANCELLED  CANCEL_REQUESTED  CANCEL_REQUESTED  RETURN_REQUESTED  (정산 진행)
                     (전액 환불)      (전액 환불)     EXCHANGE_REQUESTED
                                                           │
                                                           ▼
                                                    RETURN_COMPLETED
                                                    EXCHANGE_COMPLETED
   ```

4. **부분 취소 규칙**
   - 주문 항목(OrderItem) 단위로 부분 취소 가능
   - 이미 배송중인 항목은 취소 불가 (반품으로 전환)
   - 부분 취소 시 쿠폰/적립금 재계산 (비례 배분)
   - 최소 1개 항목은 남아있어야 부분 취소 가능 (전부 취소는 전체 취소)

5. **배송비 정책**
   - 셀러별 무료배송 기준 금액 설정 (예: 5만원 이상 무료)
   - 부분 취소 후 무료배송 기준 미달 시 배송비 추가 청구하지 않음 (셀러 부담)
   - 제주/도서산간 추가 배송비: 3,000원

6. **자동 구매확정**
   - 배송완료 후 7일 경과 시 자동 구매확정
   - 반품/교환 접수 시 자동 구매확정 타이머 중지

#### 불변 조건 (Invariants)

- `Order.totalAmount = Σ(OrderItem.subtotal) + Σ(deliveryFee) - couponDiscount - pointsUsed`
- `Order.status` 전이는 반드시 상태 머신 규칙을 따라야 한다
- `OrderItem.quantity > 0`
- `OrderItem.unitPrice > 0`
- 주문 생성 시 모든 OrderItem의 SKU가 판매중(ON_SALE) 상태여야 한다
- 쿠폰 할인 + 적립금은 상품 총액을 초과할 수 없다 (배송비는 할인 대상이 아님)

#### 도메인 용어 사전

| 용어 | 영문 | 정의 |
|------|------|------|
| 장바구니 | Cart | 구매 전 상품을 담아두는 임시 공간 |
| 주문 | Order | 구매 의사가 확정된 거래 단위 |
| 주문 항목 | Order Item | 주문 내 개별 상품(SKU) + 수량 |
| 주문 번호 | Order Number | 주문을 식별하는 고유 코드 |
| 주문 상태 | Order Status | 주문의 현재 처리 단계 |
| 부분 취소 | Partial Cancellation | 주문 내 일부 항목만 취소 |
| 교환 | Exchange | 상품을 동일 상품의 다른 옵션으로 교체 |
| 배송비 | Delivery Fee | 배송에 부과되는 비용 |
| 구매확정 | Purchase Confirmation | 구매자가 상품 수령을 확인하여 거래 완료 |
| 주문 스냅샷 | Order Snapshot | 주문 시점의 상품 정보 복사본 |
| 무료배송 기준 | Free Shipping Threshold | 배송비 면제 최소 주문 금액 |

#### 다른 도메인과의 관계

| 대상 도메인 | 관계 유형 | 방향 | 통신 방식 | 설명 |
|------------|----------|------|----------|------|
| Product | 정보 참조 | Order → Product | Sync API | 주문 시 상품 정보 스냅샷 |
| Payment | 트랜잭션 | Order ↔ Payment | Kafka Event | 결제 요청/결과 이벤트 교환 |
| Inventory | 트랜잭션 | Order → Inventory | Kafka Event | 재고 예약/차감/복원 |
| Shipping | 라이프사이클 | Order ↔ Shipping | Kafka Event | 배송 상태 동기화 |
| Promotion | 계산 | Order → Promotion | Sync API | 쿠폰/적립금 적용/반환 |
| Member | 정보 참조 | Order → Member | Sync API | 배송지, 등급할인 조회 |
| Settlement | 트리거 | Order → Settlement | Kafka Event | 구매확정 시 정산 등록 |

#### 핵심 유스케이스

1. **장바구니 관리**: 상품 담기/수량변경/삭제, 브랜드별 묶음 표시
2. **주문 생성**: 장바구니 → 주문서 → 배송지/결제수단 선택 → 주문 확정
3. **주문 취소**: 결제완료~출고 전 단계에서 취소 요청 → 결제 취소 + 재고 복원
4. **부분 취소**: 특정 항목만 취소 → 금액 재계산 → 부분 환불
5. **자동 구매확정 처리**: 배치 스케줄러로 D+7 자동 구매확정

#### 엣지 케이스 / 예외 상황

- **동시 주문 시 재고 부족**: 재고 예약 실패 시 해당 항목 제거 + 사용자 알림
- **결제 중 가격 변동**: 주문서 생성 시점 가격 고정, 결제 금액 불일치 시 주문 실패
- **장바구니 상품 판매중지**: 장바구니 조회 시 실시간 상태 체크, 비활성 표시
- **부분 취소 시 쿠폰 최소 사용 금액 미달**: 쿠폰 전액 반환, 할인 전 금액으로 재계산

### 2.3 Payment (결제) 도메인

#### 도메인 책임과 범위

결제 도메인은 **주문에 대한 결제 처리 및 환불을 관리**한다.
PG사(Toss Payments) 연동, 결제 승인/취소/환불, 다양한 결제 수단 지원을 포함한다.

**범위 포함:**
- PG사 결제 승인 요청 및 결과 처리
- 결제 취소 (전액/부분)
- 환불 처리 (카드 환불, 계좌 환불)
- 가상계좌 입금 대기/확인
- 결제 이력 관리
- 결제 수단 관리

**범위 제외:**
- 주문 금액 계산 (→ Order)
- 쿠폰/적립금 할인 (→ Promotion, Member)
- 정산 (→ Settlement)

#### 핵심 비즈니스 규칙

1. **결제 수단 규칙**
   - 지원 수단: 신용/체크카드, 간편결제(카카오페이, 네이버페이, 토스페이), 가상계좌, 무통장입금
   - 가상계좌 입금 기한: 24시간 (초과 시 자동 주문 취소)
   - 최소 결제 금액: 100원

2. **결제 승인 규칙**
   - PG사 결제 키(paymentKey) 발급 → 서버 승인 API 호출 (2단계 인증)
   - 승인 요청 시 주문 금액과 PG 전달 금액 일치 검증 필수
   - 결제 승인 타임아웃: 30초
   - 타임아웃 시 PG 상태 조회로 실제 승인 여부 확인 후 처리

3. **결제 취소/환불 규칙**
   - 카드 결제: 부분 취소 가능 (원거래 기준)
   - 가상계좌: 부분 취소 불가 → 전액 환불 후 재결제
   - 환불 계좌 입력 필수 (가상계좌, 무통장 결제 시)
   - 부분 환불 총액은 원결제 금액을 초과할 수 없다

4. **멱등성 규칙**
   - 결제 요청은 idempotencyKey로 중복 방지
   - 동일 idempotencyKey로 재요청 시 기존 결과 반환

5. **결제 상태 전이**
   ```
   READY → IN_PROGRESS → DONE → PARTIAL_CANCELLED → CANCELLED
               │                                         ▲
               ▼                                         │
           FAILED                              DONE ─────┘
           EXPIRED (가상계좌)
   ```

#### 불변 조건

- `Payment.amount == Order.paymentAmount` (주문 결제 금액과 일치)
- `Payment.refundedAmount <= Payment.amount` (환불 총액 <= 원결제 금액)
- `Payment.status` 전이는 상태 머신 규칙 준수
- 결제 완료(DONE) 상태에서만 취소/환불 가능

#### 도메인 용어 사전

| 용어 | 영문 | 정의 |
|------|------|------|
| 결제 | Payment | 주문에 대한 금전 거래 |
| 결제 키 | Payment Key | PG사가 발급한 결제 식별자 |
| 결제 승인 | Payment Approval | PG사에 결제 확정 요청 |
| 결제 취소 | Payment Cancellation | 승인된 결제를 취소 |
| 부분 환불 | Partial Refund | 결제 금액 중 일부만 환불 |
| 가상계좌 | Virtual Account | 1회용 입금 전용 계좌 |
| PG사 | Payment Gateway | 결제 대행사 (Toss Payments) |
| 멱등성 키 | Idempotency Key | 중복 결제 방지를 위한 고유 키 |
| 정산 대상 금액 | Settleable Amount | 수수료 차감 전 셀러에게 지급할 금액 |
| 결제 수단 | Payment Method | 결제에 사용된 수단 (카드/간편결제 등) |

#### 다른 도메인과의 관계

| 대상 도메인 | 관계 유형 | 방향 | 통신 방식 | 설명 |
|------------|----------|------|----------|------|
| Order | 트랜잭션 | Payment ↔ Order | Kafka Event | 결제 결과 → 주문 상태 변경 |
| Inventory | 보상 트랜잭션 | Payment → Inventory | Kafka Event | 결제 실패 시 재고 복원 이벤트 |
| Settlement | 데이터 제공 | Settlement → Payment | Sync API | 정산 시 결제 정보 조회 |
| Promotion | 보상 트랜잭션 | Payment → Promotion | Kafka Event | 결제 취소 시 쿠폰 반환 |
| Member | 보상 트랜잭션 | Payment → Member | Kafka Event | 결제 취소 시 적립금 반환 |

#### 핵심 유스케이스

1. **결제 승인**: 주문확정 → PG 승인 요청 → 결과 처리 → PaymentCompletedEvent 발행
2. **결제 취소(전액)**: 주문취소 → PG 취소 요청 → PaymentCancelledEvent 발행
3. **부분 환불**: 부분취소 → 환불 금액 계산 → PG 부분취소 → PartialRefundedEvent 발행
4. **가상계좌 입금 확인**: PG 웹훅 수신 → 입금 확인 → 주문 상태 변경
5. **결제 실패 복구**: 결제 실패/타임아웃 → PG 상태 조회 → 보상 트랜잭션 실행

#### 엣지 케이스 / 예외 상황

- **결제 승인 후 주문 처리 실패**: 결제 취소 보상 트랜잭션 실행
- **PG사 장애**: 서킷 브레이커 + 결제 불가 안내 + 대체 PG 전환
- **이중 결제**: 멱등성 키로 방지, 발생 시 자동 감지 + 환불
- **가상계좌 부분 입금**: 입금 금액 불일치 시 입금 대기 유지 + 고객 안내

### 2.4 Inventory (재고) 도메인

#### 도메인 책임과 범위

재고 도메인은 **SKU별 재고 수량의 정확한 관리와 동시성 제어**를 담당한다.

**범위 포함:**
- SKU별 가용 재고, 예약 재고, 실물 재고 관리
- 재고 예약 (주문 시) / 차감 (결제 완료 시) / 복원 (취소 시)
- 안전재고 임계값 알림
- 재고 입고/출고 이력
- 동시성 제어 (비관적 락)

**범위 제외:**
- 상품/SKU 정보 (→ Product)
- 출고/배송 처리 (→ Shipping)

#### 핵심 비즈니스 규칙

1. **재고 수량 규칙**
   - `가용재고 = 실물재고 - 예약재고`
   - 가용재고가 0 이하이면 주문 불가
   - 실물재고는 음수가 될 수 없다
   - 예약재고는 실물재고를 초과할 수 없다

2. **재고 예약/차감/복원 플로우**
   ```
   주문 생성 시: 가용재고 차감, 예약재고 증가 (RESERVE)
   결제 완료 시: 예약재고 차감, 실물재고 차감 (DEDUCT)
   주문 취소 시: 예약재고 차감, 가용재고 복원 (RESTORE)
   반품 완료 시: 실물재고 증가 (RESTOCK)
   ```

3. **동시성 제어**
   - SKU별 비관적 락 (SELECT FOR UPDATE)
   - 락 타임아웃: 3초
   - 데드락 방지: SKU ID 오름차순으로 락 획득

4. **안전재고 규칙**
   - 셀러별 안전재고 임계값 설정 가능 (기본값: 5)
   - 가용재고 <= 안전재고 시 셀러에게 알림
   - 가용재고 0 시 상품 품절 처리 이벤트 발행

5. **재입고 규칙**
   - 품절 상품 재입고 시 RestockedEvent 발행
   - 재입고 알림 구독자에게 알림 트리거

#### 불변 조건

- `Inventory.totalStock >= 0`
- `Inventory.reservedStock >= 0`
- `Inventory.reservedStock <= Inventory.totalStock`
- `Inventory.availableStock = Inventory.totalStock - Inventory.reservedStock >= 0`
- 재고 변경은 반드시 이력(StockHistory)에 기록되어야 한다

#### 도메인 용어 사전

| 용어 | 영문 | 정의 |
|------|------|------|
| 실물재고 | Total Stock | 물리적으로 보유한 총 재고 수량 |
| 예약재고 | Reserved Stock | 주문으로 예약되었으나 아직 출고되지 않은 수량 |
| 가용재고 | Available Stock | 실제 주문 가능한 재고 (실물 - 예약) |
| 재고 예약 | Stock Reservation | 주문 시 가용재고를 줄이고 예약재고를 늘리는 행위 |
| 재고 차감 | Stock Deduction | 결제 완료 시 예약재고와 실물재고를 줄이는 행위 |
| 재고 복원 | Stock Restoration | 취소 시 예약재고를 줄이고 가용재고를 늘리는 행위 |
| 안전재고 | Safety Stock | 품절 방지를 위한 최소 유지 수량 |
| 재입고 | Restock | 품절 상품에 재고가 추가되는 것 |
| 재고 이력 | Stock History | 재고 변동 기록 (유형, 수량, 시점, 원인) |
| 비관적 락 | Pessimistic Lock | 동시 접근 방지를 위한 DB 레벨 잠금 |

#### 다른 도메인과의 관계

| 대상 도메인 | 관계 유형 | 방향 | 통신 방식 | 설명 |
|------------|----------|------|----------|------|
| Product | 초기화 | Product → Inventory | Kafka Event | SKU 생성 시 재고 레코드 초기화 |
| Order | 예약/복원 | Order → Inventory | Kafka Event | 주문 생성/취소 시 재고 조작 |
| Payment | 차감 | Payment → Inventory | Kafka Event | 결제 완료 시 재고 확정 차감 |
| Shipping | 복원 | Shipping → Inventory | Kafka Event | 반품 완료 시 재고 복원 |
| Notification | 알림 트리거 | Inventory → Notification | Kafka Event | 품절/재입고 시 알림 |
| Display | 데이터 참조 | Display → Inventory | Sync API | 품절 여부 표시 |

#### 핵심 유스케이스

1. **재고 예약**: 주문 생성 이벤트 수신 → SKU별 비관적 락 → 가용재고 확인 → 예약
2. **재고 차감**: 결제 완료 이벤트 수신 → 예약재고 → 실물재고 차감
3. **재고 복원**: 주문 취소 이벤트 수신 → 예약재고 해제 → 가용재고 복원
4. **재고 입고**: 셀러가 파트너센터에서 입고 처리 → 실물재고 증가
5. **품절/재입고 처리**: 가용재고 0 → 품절 이벤트 / 입고 → 재입고 이벤트

#### 엣지 케이스 / 예외 상황

- **동시 주문 재고 경합**: 비관적 락으로 순차 처리, 재고 부족 시 후순위 주문 실패
- **예약 후 결제 미완료**: 예약 타임아웃 30분 → 미결제 시 자동 예약 해제
- **재고 음수 방지**: DB 레벨 CHECK 제약조건 + 애플리케이션 레벨 이중 검증
- **대량 주문(100개+) 재고 처리**: 배치 단위 락으로 데드락 방지

### 2.5 Shipping (배송) 도메인

#### 도메인 책임과 범위

배송 도메인은 **상품의 출고부터 배송 완료, 반품 수거까지의 물류 프로세스를 관리**한다.

**범위 포함:**
- 택배사 연동 (CJ대한통운, 한진, 롯데택배 등)
- 송장 번호 관리
- 배송 상태 추적 (택배사 API 폴링)
- 자동 구매확정 타이머
- 반품 수거 요청
- 반품 검수 처리

**범위 제외:**
- 주문 상태 관리 (→ Order)
- 환불 처리 (→ Payment)
- 배송비 금액 계산 (→ Order)

#### 핵심 비즈니스 규칙

1. **출고 규칙**
   - 결제 완료 후 셀러는 영업일 기준 3일 이내 출고 의무
   - 출고 미처리 3일 초과 시 자동 알림 → 5일 초과 시 자동 취소
   - 송장 번호 형식: 택배사별 규격 검증

2. **배송 추적 규칙**
   - 택배사 API 폴링 주기: 2시간
   - 배송 상태: 집하 → 간선이동 → 배달출발 → 배송완료
   - 배송완료 후 자동 구매확정 타이머 시작 (7일)

3. **반품 수거 규칙**
   - 반품 접수 시 자동 수거 요청 (택배사 API)
   - 수거 완료 후 셀러에게 검수 요청
   - 검수 결과: 승인(환불 진행) / 거부(반송)
   - 반품 사유: 단순변심(구매자 배송비 부담) / 상품불량(셀러 배송비 부담)

4. **배송 상태 전이**
   ```
   READY_TO_SHIP → SHIPPED → IN_TRANSIT → OUT_FOR_DELIVERY → DELIVERED
                                                                 │
                                                                 ▼
                                                         AUTO_CONFIRMED (D+7)

   [반품 플로우]
   RETURN_REQUESTED → PICKUP_SCHEDULED → PICKUP_COMPLETED → INSPECTING → RETURN_APPROVED / RETURN_REJECTED
   ```

#### 불변 조건

- 송장번호는 출고 시 반드시 존재해야 한다
- 배송 상태 전이는 순방향만 가능 (역전이 불가)
- 반품 수거는 배송완료 후 7일 이내에만 요청 가능

#### 도메인 용어 사전

| 용어 | 영문 | 정의 |
|------|------|------|
| 출고 | Shipment | 셀러가 상품을 발송하는 행위 |
| 송장번호 | Tracking Number | 택배 추적을 위한 고유 번호 |
| 배송추적 | Delivery Tracking | 택배사 API로 배송 상태를 실시간 조회 |
| 구매확정 | Purchase Confirmation | 배송 완료 후 거래가 최종 확정되는 것 |
| 반품수거 | Return Pickup | 반품 상품을 수거하는 택배 서비스 |
| 검수 | Inspection | 반품 상품의 상태를 확인하는 과정 |
| 택배사 | Courier | 배송을 수행하는 물류 업체 |
| 집하 | Collection | 택배사가 상품을 수거하는 단계 |
| 간선이동 | Line Haul | 허브 터미널 간 상품 이동 |
| 배달출발 | Out for Delivery | 최종 배송 담당자가 배달을 시작한 단계 |

#### 다른 도메인과의 관계

| 대상 도메인 | 관계 유형 | 방향 | 통신 방식 | 설명 |
|------------|----------|------|----------|------|
| Order | 상태 동기화 | Shipping → Order | Kafka Event | 배송 상태 변경 → 주문 상태 동기화 |
| Payment | 환불 트리거 | Shipping → Payment | Kafka Event | 반품 승인 → 환불 진행 |
| Inventory | 재고 복원 | Shipping → Inventory | Kafka Event | 반품 검수 완료 → 재고 복원 |
| Settlement | 정산 트리거 | Shipping → Settlement | Kafka Event | 구매확정 → 정산 등록 |
| Notification | 알림 | Shipping → Notification | Kafka Event | 배송 상태 변경 → 구매자 알림 |
| Seller | 출고 요청 | Shipping → Seller | Kafka Event | 결제 완료 → 셀러 출고 알림 |

#### 핵심 유스케이스

1. **출고 처리**: 셀러가 송장 입력 → 택배사 API로 송장 등록 → ShippedEvent 발행
2. **배송 추적**: 스케줄러로 택배사 API 폴링 → 상태 변경 감지 → 이벤트 발행
3. **자동 구매확정**: 배송완료 D+7 → 자동 구매확정 → PurchaseConfirmedEvent 발행
4. **반품 수거**: 반품 접수 → 택배사 수거 요청 → 수거 완료 → 셀러 검수
5. **반품 검수**: 셀러 검수 → 승인/거부 → 환불 또는 반송

#### 엣지 케이스

- **택배사 API 장애**: 폴링 실패 시 재시도 + 수동 상태 업데이트 지원
- **오배송**: 셀러에게 재발송 요청 + 기존 상품 수거
- **배송 분실**: 택배사 클레임 → 플랫폼 중재 → 환불 처리
- **출고 기한 초과**: 자동 알림 → 자동 취소 → 페널티 기록

### 2.6 Member (회원) 도메인

#### 도메인 책임과 범위

회원 도메인은 **사용자 인증/인가, 회원 정보 관리, 등급 시스템, 포인트, 배송지를 관리**한다.

**범위 포함:**
- 회원가입/로그인 (이메일, 소셜)
- JWT 기반 인증
- 회원 등급 관리 (일반/실버/골드/플래티넘)
- 포인트(적립금) 관리
- 배송지 관리
- 위시리스트

**범위 제외:**
- API Gateway 라우팅 (→ 인프라)
- 셀러 계정 (→ Seller)

#### 핵심 비즈니스 규칙

1. **회원 등급 규칙**

   | 등급 | 조건 (최근 6개월) | 혜택 |
   |------|-----------------|------|
   | 일반 (BASIC) | 가입 즉시 | 기본 적립 1% |
   | 실버 (SILVER) | 누적 구매 30만원 이상 | 적립 2% + 실버 전용 쿠폰 |
   | 골드 (GOLD) | 누적 구매 100만원 이상 | 적립 3% + 골드 전용 쿠폰 + 무료배송 쿠폰 |
   | 플래티넘 (PLATINUM) | 누적 구매 300만원 이상 | 적립 5% + 전용 쿠폰 + 무료배송 + 우선 CS |

   - 등급 산정: 매월 1일 자동 재산정 (최근 6개월 구매확정 기준)
   - 등급 하락: 조건 미달 시 한 단계씩 하락 (한 번에 두 단계 이상 하락 없음)

2. **포인트 규칙**
   - 구매확정 시 등급별 비율로 적립
   - 리뷰 작성 시 추가 적립 (텍스트 200P, 포토 500P, 영상 1000P)
   - 유효기간: 적립일로부터 1년
   - 최소 사용 단위: 1,000P
   - 포인트로 결제 가능 금액: 주문 총액의 최대 30%

3. **배송지 규칙**
   - 최대 10개 등록 가능
   - 기본 배송지 1개 필수
   - 삭제 시 기본 배송지가 삭제되면 다음 최근 배송지가 기본으로 설정

4. **소셜 로그인**
   - 지원: 카카오, 네이버, 구글, 애플
   - 최초 소셜 로그인 시 자동 회원가입
   - 동일 이메일 기존 계정 존재 시 연동 제안

5. **위시리스트**
   - 최대 200개
   - 가격 변동 알림 (판매가 하락 시)
   - 재입고 알림 구독 연계

#### 불변 조건

- `Member.email`은 시스템 내 유일해야 한다
- `Member.point >= 0` (포인트 잔액은 음수 불가)
- 기본 배송지는 항상 1개 존재해야 한다 (배송지가 있는 경우)
- 등급 전이는 한 번에 한 단계씩만 가능

#### 도메인 용어 사전

| 용어 | 영문 | 정의 |
|------|------|------|
| 회원 | Member | 플랫폼에 가입한 사용자 |
| 등급 | Grade | 구매 금액 기반 회원 등급 (BASIC/SILVER/GOLD/PLATINUM) |
| 포인트 | Point | 구매/리뷰 시 적립되는 사이트 내 화폐 |
| 적립금 | Mileage | 포인트와 동의어, 주문 시 할인 수단으로 사용 |
| 배송지 | Shipping Address | 회원이 등록한 배송 주소 |
| 기본 배송지 | Default Address | 주문 시 자동 선택되는 배송 주소 |
| 위시리스트 | Wishlist | 관심 상품 저장 목록 |
| 소셜 계정 | Social Account | 외부 OAuth 연동 계정 (카카오, 네이버 등) |
| 회원 탈퇴 | Withdrawal | 회원이 서비스 이용을 중지하는 것 (soft delete) |
| 휴면 계정 | Dormant Account | 1년 이상 미접속 시 자동 전환되는 상태 |

#### 다른 도메인과의 관계

| 대상 도메인 | 관계 유형 | 방향 | 통신 방식 | 설명 |
|------------|----------|------|----------|------|
| Order | 정보 제공 | Order → Member | Sync API | 배송지, 등급정보 조회 |
| Payment | 포인트 | Payment → Member | Kafka Event | 결제/취소 시 포인트 차감/복원 |
| Review | 포인트 적립 | Review → Member | Kafka Event | 리뷰 작성 시 포인트 적립 |
| Promotion | 등급 혜택 | Promotion → Member | Sync API | 등급별 쿠폰 자동 발급 |
| Notification | 알림 수신 | Notification → Member | Sync API | 알림 대상 회원 정보 |
| Product | 위시리스트 | Member → Product | Sync API | 상품 정보 조회 |

#### 핵심 유스케이스

1. **회원가입**: 이메일/비밀번호 또는 소셜 로그인으로 가입
2. **로그인/토큰 발급**: 인증 → Access Token + Refresh Token 발급
3. **등급 재산정**: 월간 배치 → 최근 6개월 구매확정 금액 → 등급 변경
4. **포인트 관리**: 적립/사용/만료 처리, 잔액 조회
5. **배송지 관리**: CRUD + 기본 배송지 설정

#### 엣지 케이스

- **동일 이메일 소셜 계정 충돌**: 연동 제안 화면 → 비밀번호 인증 후 연동
- **포인트 만료 시 사용중**: 이미 사용된 포인트는 만료 대상에서 제외 (FIFO 방식)
- **등급 재산정 중 주문 발생**: 등급 산정은 스냅샷 기준, 실시간 변동 무관
- **탈퇴 후 재가입**: 동일 이메일 재가입 가능 (개인정보 삭제 완료 후)

### 2.7 Search (검색) 도메인

#### 도메인 책임과 범위

검색 도메인은 **상품 검색, 필터링, 정렬, 자동완성, 인기 검색어를 제공**한다.

**범위 포함:**
- Elasticsearch 기반 상품 검색
- nori 한글 형태소 분석
- 카테고리/브랜드/가격/사이즈/색상 필터
- 정렬 (인기순, 최신순, 가격순, 할인율순, 리뷰순)
- 자동완성 (Search-as-you-type)
- 인기검색어 (실시간/일간/주간)
- 연관검색어

**범위 제외:**
- 상품 원본 데이터 (→ Product)
- 랭킹/전시 (→ Display)

#### 핵심 비즈니스 규칙

1. **검색 인덱싱 규칙**
   - 상품 CUD 이벤트 수신 → ES 인덱스 업데이트 (비동기)
   - 인덱싱 지연: 최대 5초 이내 (Near Real-Time)
   - 인덱스 필드: 상품명, 브랜드명, 카테고리, 설명, 태그, 시즌, 핏, 색상

2. **한글 분석 규칙**
   - nori 형태소 분석기 + 동의어 사전 (예: 바지=팬츠, 티셔츠=티=반팔)
   - 자모 분리 검색 지원 (ㄴㅇㅋ → 나이키)
   - 오타 교정 (fuzzy matching, edit distance 2)
   - 영한 변환 (dkssud → 안녕)

3. **필터 규칙**
   - 카테고리: 다중 선택 가능 (OR 조건)
   - 브랜드: 다중 선택 가능
   - 가격: 범위 필터 (min-max)
   - 사이즈: 다중 선택 (OR)
   - 색상: 다중 선택 (OR)
   - 할인율: 범위 필터
   - 필터 간: AND 조건

4. **자동완성 규칙**
   - 최소 2자 입력 시 발동
   - 최대 10개 추천
   - 검색 빈도 + 판매량 가중치로 정렬

5. **인기검색어 규칙**
   - 실시간: 10분 단위 갱신, 상위 20개
   - 일간/주간: 검색 빈도 + 클릭 수 가중 합산
   - 금지어 필터링 (부적절한 키워드 제외)

#### 불변 조건

- 검색 결과에 판매중지/삭제 상품이 포함되면 안 된다
- 검색 인덱스와 상품 DB의 최종 일관성 보장 (최대 5초 지연)
- 자동완성은 100ms 이내 응답

#### 도메인 용어 사전

| 용어 | 영문 | 정의 |
|------|------|------|
| 검색어 | Search Query | 사용자가 입력한 검색 키워드 |
| 형태소 분석 | Morphological Analysis | 한글을 의미 단위로 분리하는 처리 (nori) |
| 자동완성 | Autocomplete | 입력 중 추천 검색어를 보여주는 기능 |
| 인기검색어 | Popular Keywords | 많이 검색된 키워드 순위 |
| 연관검색어 | Related Keywords | 현재 검색어와 관련된 다른 검색어 |
| 필터 | Filter | 검색 결과를 좁히는 조건 |
| 패싯 | Facet | 필터별 결과 건수 집계 |
| 동의어 사전 | Synonym Dictionary | 같은 의미의 단어 매핑 |
| 금지어 | Banned Word | 검색/자동완성에서 제외되는 부적절한 단어 |
| 검색 점수 | Relevance Score | 검색 결과의 관련도 점수 |

#### 다른 도메인과의 관계

| 대상 도메인 | 관계 유형 | 방향 | 통신 방식 | 설명 |
|------------|----------|------|----------|------|
| Product | 데이터 동기화 | Product → Search | Kafka Event | 상품 변경 → 인덱스 업데이트 |
| Display | 데이터 제공 | Search → Display | Sync API | 전시에서 검색 API 호출 |
| Inventory | 데이터 참조 | Inventory → Search | Kafka Event | 품절 시 검색 결과에서 표시 변경 |
| Review | 데이터 참조 | Review → Search | Kafka Event | 리뷰 수/평점 인덱스 업데이트 |

#### 핵심 유스케이스

1. **상품 검색**: 키워드 입력 → 형태소 분석 → ES 쿼리 → 결과 반환 (필터/정렬/페이징)
2. **자동완성**: 2자 이상 입력 → ES completion suggester → 추천어 반환
3. **인덱스 업데이트**: 상품 이벤트 수신 → ES 도큐먼트 UPSERT/DELETE
4. **인기검색어 갱신**: 검색 로그 집계 → Redis ZSET 업데이트 → 10분 단위 갱신
5. **필터 패싯 조회**: 검색 결과에 대한 카테고리별/브랜드별/가격대별 건수 집계

#### 엣지 케이스

- **ES 장애**: 서킷 브레이커 → DB 기반 폴백 검색 (성능 저하 허용)
- **인덱스 불일치**: 주기적 전체 재인덱싱 배치 (주 1회)
- **검색 폭주 (타임세일)**: ES 클러스터 오토스케일링 + 결과 캐싱(Redis, 1분 TTL)
- **악의적 검색 (DoS)**: Rate Limiting (IP당 분당 60회)

### 2.8 Review (리뷰) 도메인

#### 도메인 책임과 범위

리뷰 도메인은 **구매자의 상품 후기 작성/관리와 사이즈 피드백 시스템을 담당**한다.

**범위 포함:**
- 별점 리뷰 (1~5점)
- 텍스트/포토/영상 리뷰
- 사이즈 후기 (키, 몸무게, 평소 사이즈, 선택 사이즈, 핏감)
- 리뷰 보상 (포인트)
- "도움돼요" 투표
- 리뷰 신고/관리

**범위 제외:**
- 포인트 적립 처리 (→ Member)
- 상품 평점 집계 (→ Product)

#### 핵심 비즈니스 규칙

1. **리뷰 작성 규칙**
   - 구매확정된 주문 항목에 대해서만 작성 가능
   - 동일 주문 항목당 리뷰 1회만 가능 (수정은 가능)
   - 리뷰 수정: 작성 후 30일 이내
   - 리뷰 삭제: 작성자만 가능 (soft delete)
   - 최소 텍스트 20자 이상

2. **사이즈 후기 규칙**
   - 선택 필드: 키(cm), 몸무게(kg), 평소 사이즈, 선택 사이즈
   - 핏감 투표: 작아요 / 정사이즈 / 커요
   - 사이즈 후기 집계 → 상품 상세의 "사이즈 리뷰 요약" 제공

3. **리뷰 보상**
   - 텍스트 리뷰: 200P
   - 포토 리뷰 (1장 이상): 500P
   - 영상 리뷰 (10초 이상): 1,000P
   - 사이즈 후기 포함 시 추가 100P
   - 보상은 리뷰 등록 즉시 지급 (삭제 시 회수)

4. **"도움돼요" 규칙**
   - 회원만 투표 가능
   - 1회만 가능 (취소 가능)
   - 도움돼요 수 기준 리뷰 정렬 지원

5. **리뷰 관리 규칙**
   - 비속어/광고 자동 필터링
   - 신고 3회 이상 시 자동 숨김 → 관리자 검토
   - 가품 의심 리뷰: 관리자 비공개 처리 가능

#### 불변 조건

- `Review.rating` 범위: 1~5 정수
- `Review.content.length >= 20`
- 구매확정된 OrderItem에 대해서만 Review 생성 가능
- 동일 OrderItem에 대한 Review는 1개만 존재

#### 도메인 용어 사전

| 용어 | 영문 | 정의 |
|------|------|------|
| 리뷰 | Review | 구매 후 작성하는 상품 후기 |
| 별점 | Rating | 1~5점으로 표현하는 만족도 |
| 포토 리뷰 | Photo Review | 사진이 포함된 리뷰 |
| 영상 리뷰 | Video Review | 영상이 포함된 리뷰 |
| 사이즈 후기 | Size Feedback | 키/몸무게/사이즈 정보가 포함된 리뷰 |
| 핏감 | Fit Feedback | 사이즈가 작은지/정사이즈인지/큰지 평가 |
| 도움돼요 | Helpful Vote | 리뷰가 도움되었음을 표시하는 투표 |
| 리뷰 보상 | Review Reward | 리뷰 작성 시 지급되는 포인트 |
| 베스트 리뷰 | Best Review | 도움돼요 수가 많은 상위 리뷰 |
| 리뷰 신고 | Review Report | 부적절한 리뷰를 신고하는 행위 |

#### 다른 도메인과의 관계

| 대상 도메인 | 관계 유형 | 방향 | 통신 방식 | 설명 |
|------------|----------|------|----------|------|
| Order | 권한 확인 | Review → Order | Sync API | 구매확정 여부 확인 |
| Product | 평점 집계 | Review → Product | Kafka Event | 리뷰 등록/수정/삭제 → 평점 재계산 |
| Member | 포인트 적립 | Review → Member | Kafka Event | 리뷰 보상 포인트 적립 이벤트 |
| Search | 데이터 동기화 | Review → Search | Kafka Event | 리뷰 수/평점 변경 → 검색 인덱스 업데이트 |

#### 핵심 유스케이스

1. **리뷰 작성**: 구매확정 확인 → 리뷰 저장 → 포인트 적립 이벤트 → 평점 재계산 이벤트
2. **리뷰 조회**: 상품별 리뷰 목록 (최신순/도움돼요순/별점순) + 사이즈 요약
3. **도움돼요 투표**: 투표 토글 → 카운트 업데이트
4. **리뷰 신고**: 신고 접수 → 자동 숨김 기준 체크 → 관리자 알림
5. **사이즈 리뷰 요약**: 핏감 투표 집계 → 크기 그래프 표시

#### 엣지 케이스

- **리뷰 삭제 후 포인트 회수**: 리뷰 삭제 → 포인트 차감 (잔액 부족 시 마이너스 방지)
- **구매 없이 리뷰 시도**: OrderItem 검증 → 거부
- **대량 리뷰 어뷰징**: 동일 IP/계정 리뷰 속도 제한 (일 10건)

### 2.9 Promotion (프로모션) 도메인

#### 도메인 책임과 범위

프로모션 도메인은 **쿠폰, 타임세일, 적립금 이벤트 등 각종 할인/혜택 정책을 관리**한다.

**범위 포함:**
- 쿠폰 생성/발급/사용/반환
- 타임세일 (시간 한정 할인)
- 적립금 이벤트 (가입 축하금, 생일 적립금 등)
- 중복 사용 규칙
- 선착순 발급 동시성 제어

**범위 제외:**
- 회원 포인트 잔액 관리 (→ Member)
- 결제 금액 계산 (→ Order)

#### 핵심 비즈니스 규칙

1. **쿠폰 유형**
   | 유형 | 설명 | 예시 |
   |------|------|------|
   | 정액 할인 | 고정 금액 할인 | 5,000원 할인 |
   | 정률 할인 | 비율 할인 (최대 할인 한도) | 10% 할인 (최대 5,000원) |
   | 무료배송 | 배송비 면제 | |
   | 카테고리 | 특정 카테고리 전용 | 아우터 20% 할인 |
   | 브랜드 | 특정 브랜드 전용 | 나이키 15% 할인 |

2. **쿠폰 발급 규칙**
   - 회원당 동일 쿠폰 1회만 발급
   - 선착순 쿠폰: Redis DECR 기반 동시성 제어
   - 유효기간: 발급일 기준 / 고정 종료일 두 가지 방식
   - 최소 주문 금액 조건 설정 가능

3. **쿠폰 사용 규칙**
   - 주문당 쿠폰 최대 2개 적용 가능 (상품쿠폰 1 + 장바구니쿠폰 1)
   - 쿠폰 할인 + 적립금은 상품 총액의 50%까지만 적용 가능
   - 특정 상품 제외 설정 가능 (PB 상품, 이미 할인중인 상품)
   - 사용 후 취소 시 자동 반환 (유효기간 내인 경우만)

4. **타임세일 규칙**
   - 시작/종료 시각 설정
   - 수량 한정 가능
   - 중복 참여 불가 (동일 상품 재구매 방지)
   - 타임세일 가격 < 일반 판매가

5. **선착순 동시성 규칙**
   - Redis DECR로 원자적 수량 차감
   - 수량 0 이하 시 즉시 발급 종료
   - 발급 후 DB 비동기 기록

#### 불변 조건

- `Coupon.discountAmount > 0` (정액) 또는 `Coupon.discountRate > 0 && discountRate <= 100` (정률)
- 정률 쿠폰의 `maxDiscountAmount > 0`
- `Coupon.usedAt`이 null이 아니면 재사용 불가
- 쿠폰 유효기간 내에서만 사용 가능
- 선착순 쿠폰의 `remainingQuantity >= 0`

#### 도메인 용어 사전

| 용어 | 영문 | 정의 |
|------|------|------|
| 쿠폰 | Coupon | 할인 혜택을 제공하는 디지털 증서 |
| 쿠폰 정책 | Coupon Policy | 쿠폰의 종류, 할인 규칙, 사용 조건 정의 |
| 발급 | Issuance | 쿠폰을 회원에게 배포하는 행위 |
| 선착순 | First-Come-First-Served | 제한 수량을 선착순으로 발급 |
| 타임세일 | Time Sale | 시간 한정 할인 행사 |
| 정액 할인 | Fixed Discount | 고정 금액 할인 |
| 정률 할인 | Percentage Discount | 비율 기반 할인 |
| 최대 할인 한도 | Max Discount Cap | 정률 할인의 최대 할인 금액 |
| 최소 주문 금액 | Min Order Amount | 쿠폰 사용 가능한 최소 주문 금액 |
| 쿠폰 반환 | Coupon Restoration | 주문 취소 시 쿠폰을 회원에게 재부여 |

#### 다른 도메인과의 관계

| 대상 도메인 | 관계 유형 | 방향 | 통신 방식 | 설명 |
|------------|----------|------|----------|------|
| Order | 할인 적용 | Order → Promotion | Sync API | 주문 시 쿠폰/적립금 적용 가능 여부 확인 |
| Payment | 반환 | Payment → Promotion | Kafka Event | 결제 취소 → 쿠폰 반환 |
| Member | 등급 연계 | Promotion → Member | Sync API | 등급별 자동 쿠폰 발급 |
| Product | 대상 지정 | Promotion → Product | Sync API | 카테고리/브랜드 기반 쿠폰 대상 조회 |
| Notification | 알림 | Promotion → Notification | Kafka Event | 쿠폰 발급/만료 임박 알림 |

#### 핵심 유스케이스

1. **쿠폰 발급**: 다운로드 요청 → 중복 체크 → 수량 확인 → 발급
2. **선착순 쿠폰 발급**: Redis DECR → 성공 시 DB 비동기 기록
3. **쿠폰 적용**: 주문서에서 쿠폰 선택 → 조건 검증 → 할인 금액 계산
4. **쿠폰 반환**: 주문 취소 이벤트 → 유효기간 체크 → 반환
5. **타임세일 관리**: 시작/종료 예약 → 가격 자동 변경

#### 엣지 케이스

- **선착순 100장 동시 요청 1,000건**: Redis DECR 원자 연산으로 정확히 100장만 발급
- **쿠폰 적용 후 가격 변경**: 주문서 확정 시점 가격 기준, 이후 변경 무관
- **만료 임박 쿠폰 사용**: 쿠폰 사용 시점에 유효기간 재검증
- **부분 취소 시 쿠폰 최소 금액 미달**: 쿠폰 전액 반환 + 할인 제거 후 재계산

### 2.10 Display (전시) 도메인

#### 도메인 책임과 범위

전시 도메인은 **사용자에게 보여지는 화면 구성과 콘텐츠 배치를 관리**한다.

**범위 포함:**
- 메인 페이지 구성 (배너, 추천 상품, 신상품)
- 상품 랭킹 (실시간/일간/주간/월간)
- 기획전 관리
- 배너 관리 (메인, 카테고리, 브랜드관)
- 브랜드관 페이지
- 카테고리별 신상품/인기 상품

**범위 제외:**
- 상품 원본 데이터 (→ Product)
- 검색 기능 (→ Search)
- 콘텐츠 (매거진, OOTD → Content)

#### 핵심 비즈니스 규칙

1. **랭킹 규칙**
   - 실시간 랭킹: 최근 1시간 판매량 기준 (Redis ZSET)
   - 일간 랭킹: 전일 판매량 + 리뷰수 + 위시리스트 수 가중합
   - 주간/월간: 해당 기간 집계
   - 카테고리별, 성별, 연령대별 랭킹 제공
   - 갱신 주기: 실시간(10분), 일간(매일 00시), 주간(매주 월요일)

2. **기획전 규칙**
   - 시작/종료일 설정
   - 수동 상품 큐레이션 + 자동 추천 혼합
   - 최대 200개 상품 포함 가능
   - 전시 순서 수동 설정 가능

3. **배너 규칙**
   - 위치별 배너 슬롯 (메인 상단, 카테고리 상단, 중간 등)
   - 노출 기간 설정 (시작일~종료일)
   - 클릭 로그 수집 → CTR 분석
   - A/B 테스트 지원

4. **브랜드관 규칙**
   - 입점 브랜드별 전용 페이지
   - 브랜드 대표 이미지, 소개, 전체 상품, 인기 상품
   - 브랜드 팔로우 기능

#### 불변 조건

- 전시에 포함된 상품은 반드시 ON_SALE 상태여야 한다
- 배너 노출 기간이 겹치는 동일 위치 배너의 총 수는 슬롯 수를 초과할 수 없다
- 랭킹 데이터는 항상 최신 갱신 시각을 함께 제공해야 한다

#### 도메인 용어 사전

| 용어 | 영문 | 정의 |
|------|------|------|
| 랭킹 | Ranking | 판매량/인기도 기반 상품 순위 |
| 기획전 | Exhibition | 테마/이벤트별 상품 큐레이션 페이지 |
| 배너 | Banner | 광고/이벤트 이미지 링크 |
| 브랜드관 | Brand Shop | 브랜드 전용 전시 페이지 |
| 슬롯 | Slot | 배너가 배치될 수 있는 위치 |
| CTR | Click-Through Rate | 배너 노출 대비 클릭 비율 |

| 신상품 | New Arrival | 최근 등록된 상품 |
| 전시 순서 | Display Order | 상품/배너의 노출 순서 |
| 랭킹 스냅샷 | Ranking Snapshot | 특정 시점의 랭킹 데이터 저장본 |

#### 다른 도메인과의 관계

| 대상 도메인 | 관계 유형 | 방향 | 통신 방식 | 설명 |
|------------|----------|------|----------|------|
| Product | 데이터 참조 | Display → Product | Sync API | 전시 상품 정보 조회 |
| Search | 데이터 참조 | Display → Search | Sync API | 검색 기반 추천 |
| Order | 데이터 참조 | Order → Display | Kafka Event | 판매량 → 랭킹 업데이트 |
| Inventory | 상태 참조 | Display → Inventory | Sync API | 품절 상태 표시 |
| Review | 데이터 참조 | Display → Review | Sync API | 리뷰 수/평점 표시 |
| Content | 콘텐츠 배치 | Display → Content | Sync API | 메인에 콘텐츠 영역 배치 |

#### 핵심 유스케이스

1. **메인 페이지 구성**: 배너 + 랭킹 + 신상품 + 기획전 조합 → 캐시 기반 응답
2. **랭킹 갱신**: 판매/위시 이벤트 → Redis ZSET ZINCRBY → 주기적 스냅샷
3. **기획전 관리**: 관리자가 기획전 생성 → 상품 배치 → 전시 기간 설정
4. **배너 관리**: 배너 등록 → 기간/위치 설정 → 클릭 로그 수집
5. **브랜드관 구성**: 브랜드 정보 + 인기상품 + 전체상품 + 코디 연계

#### 엣지 케이스

- **품절 상품 전시**: 품절 상태 실시간 반영, 랭킹에서는 유지 (품절 뱃지 표시)
- **기획전 기간 중 상품 판매 중지**: 해당 상품만 기획전에서 자동 제외
- **배너 이미지 S3 장애**: CDN 캐시 폴백, 기본 이미지 표시

### 2.11 Settlement (정산) 도메인

#### 도메인 책임과 범위

정산 도메인은 **셀러에게 판매 대금을 정산하는 전체 프로세스를 관리**한다.

**범위 포함:**
- 구매확정 건 정산 대상 등록
- 수수료율 적용 (카테고리별)
- 정산 주기 관리 (주간)
- 정산서 생성
- 정산금 지급 처리

**범위 제외:**
- 결제 원장 (→ Payment)
- 셀러 계좌 정보 (→ Seller)

#### 핵심 비즈니스 규칙

1. **정산 대상 등록**
   - 구매확정(PurchaseConfirmedEvent) 시 정산 대상 자동 등록
   - 정산 금액 = 결제 금액 - 쿠폰 할인(플랫폼 부담분) - 수수료
   - 쿠폰 할인 중 플랫폼 부담분은 셀러 정산에서 차감하지 않음
   - 쿠폰 할인 중 셀러 부담분은 셀러 정산에서 차감

2. **수수료 규칙**
   - 카테고리별 수수료율 (Section 1.2 참조)
   - 신규 입점 브랜드: 3개월 프로모션 수수료 (기본 -5%p)
   - 월 매출 1억 이상: 수수료 1%p 할인 (볼륨 디스카운트)

3. **정산 주기**
   - 매주 월요일 전주 구매확정 건 일괄 집계
   - 정산서 생성 → 셀러 확인 → 수요일 지급
   - 최소 정산 금액: 10,000원 (미달 시 다음 주기로 이월)

4. **정산 상태**
   ```
   PENDING → CALCULATED → CONFIRMED → PAID
                │
                ▼
            DISPUTED (셀러 이의 제기)
                │
                ▼
            RE_CALCULATED → CONFIRMED → PAID
   ```

#### 불변 조건

- 정산 금액 = 판매 금액 × (1 - 수수료율) (소수점 이하 절사)
- 정산 상태는 역전이 불가 (DISPUTED 예외)
- 구매확정되지 않은 건은 정산 대상이 될 수 없다

#### 도메인 용어 사전

| 용어 | 영문 | 정의 |
|------|------|------|
| 정산 | Settlement | 셀러에게 판매 대금을 지급하는 과정 |
| 수수료 | Commission | 플랫폼이 매출에서 차감하는 서비스 이용료 |
| 정산 주기 | Settlement Cycle | 정산이 이루어지는 주기 (주간) |
| 정산서 | Settlement Statement | 정산 내역이 기재된 문서 |
| 구매확정 | Purchase Confirmation | 정산 기준이 되는 거래 완료 상태 |
| 이의 제기 | Dispute | 셀러가 정산 내역에 문제를 제기 |
| 이월 | Carry Over | 최소 금액 미달 시 다음 주기로 넘기는 것 |
| 볼륨 디스카운트 | Volume Discount | 매출 규모에 따른 수수료 할인 |
| 플랫폼 부담분 | Platform Share | 쿠폰 할인 중 플랫폼이 부담하는 금액 |
| 셀러 부담분 | Seller Share | 쿠폰 할인 중 셀러가 부담하는 금액 |

#### 다른 도메인과의 관계

| 대상 도메인 | 관계 유형 | 방향 | 통신 방식 | 설명 |
|------------|----------|------|----------|------|
| Shipping | 트리거 | Shipping → Settlement | Kafka Event | 구매확정 이벤트 → 정산 등록 |
| Payment | 데이터 참조 | Settlement → Payment | Sync API | 결제 정보 조회 |
| Seller | 정산 정보 | Settlement → Seller | Sync API | 셀러 정산 계좌, 수수료율 조회 |
| Product | 수수료율 | Settlement → Product | Sync API | 카테고리별 수수료율 조회 |
| Notification | 알림 | Settlement → Notification | Kafka Event | 정산서 생성/지급 완료 알림 |

#### 핵심 유스케이스

1. **정산 대상 등록**: 구매확정 이벤트 → 정산 레코드 생성 (PENDING)
2. **주간 정산 집계**: 월요일 배치 → 셀러별 합산 → 수수료 계산 → 정산서 생성
3. **정산금 지급**: 수요일 배치 → 은행 API → 지급 처리 → 결과 기록
4. **이의 제기**: 셀러 문의 → 관리자 검토 → 재계산 or 확정
5. **정산 내역 조회**: 셀러 파트너센터에서 기간별 정산 내역 확인

#### 엣지 케이스

- **구매확정 후 환불**: 이미 정산된 건 → 다음 정산에서 차감 (상계)
- **셀러 계좌 오류**: 지급 실패 → 재시도 3회 → 수동 처리 전환
- **수수료율 변경 시점**: 주문 시점 수수료율 적용 (정산 시점 아님)

### 2.12 Notification (알림) 도메인

#### 도메인 책임과 범위

알림 도메인은 **다양한 채널(이메일/SMS/푸시)을 통한 알림 발송을 관리**한다.

**범위 포함:**
- 이메일 발송 (주문확인, 배송알림, 정산)
- SMS/알림톡 발송 (배송, 인증번호)
- 앱 푸시 알림 (프로모션, 재입고)
- 재입고 알림 구독/발송
- 알림 템플릿 관리
- 마케팅 수신 동의 관리

**범위 제외:**
- 실시간 채팅 (→ CS)
- 알림 내용 결정 (각 도메인에서 이벤트와 함께 전달)

#### 핵심 비즈니스 규칙

1. **알림 채널 규칙**
   - 주문/배송: 알림톡(카카오) 우선, 실패 시 SMS 폴백
   - 마케팅: 수신 동의한 회원에게만 발송
   - 푸시: 기기 토큰 등록된 회원에게만
   - 야간 발송 제한: 마케팅 알림은 21:00~08:00 발송 금지

2. **재입고 알림 규칙**
   - SKU 단위 구독 (특정 사이즈/색상)
   - 재입고 시 구독자 전원에게 발송 (최대 1,000명)
   - 발송 후 구독 자동 해제
   - 동일 SKU 재구독 가능

3. **템플릿 규칙**
   - 채널별 템플릿 (이메일 HTML, SMS 텍스트, 푸시 JSON)
   - 변수 치환: `{{memberName}}`, `{{orderNumber}}`, `{{productName}}` 등
   - 템플릿 버전 관리

4. **발송 제한**
   - 동일 회원 동일 유형: 5분 내 중복 발송 방지
   - 일일 마케팅 알림: 회원당 최대 3건
   - 발송 실패 재시도: 최대 3회 (5분, 30분, 2시간 간격)

#### 불변 조건

- 마케팅 알림은 수신 동의 없이 발송할 수 없다
- 야간(21~08시) 마케팅 발송 불가
- 템플릿의 필수 변수가 모두 치환되어야 발송 가능

#### 도메인 용어 사전

| 용어 | 영문 | 정의 |
|------|------|------|
| 알림 | Notification | 사용자에게 전달되는 메시지 |
| 알림톡 | KakaoTalk Notification | 카카오톡 기반 알림 메시지 |
| 푸시 알림 | Push Notification | 모바일 앱 푸시 메시지 |
| 재입고 알림 | Restock Notification | 품절 상품 재입고 시 알림 |
| 수신 동의 | Opt-in | 마케팅 알림 수신에 대한 동의 |
| 템플릿 | Template | 알림 메시지의 정형화된 양식 |
| 변수 치환 | Variable Substitution | 템플릿 내 변수를 실제 값으로 교체 |
| 기기 토큰 | Device Token | 푸시 발송을 위한 기기 식별자 |
| 폴백 | Fallback | 우선 채널 실패 시 대체 채널로 발송 |
| 발송 이력 | Send History | 알림 발송 결과 기록 |

#### 다른 도메인과의 관계

| 대상 도메인 | 관계 유형 | 방향 | 통신 방식 | 설명 |
|------------|----------|------|----------|------|
| Order | 이벤트 수신 | Order → Notification | Kafka Event | 주문 상태 변경 알림 |
| Shipping | 이벤트 수신 | Shipping → Notification | Kafka Event | 배송 상태 변경 알림 |
| Promotion | 이벤트 수신 | Promotion → Notification | Kafka Event | 쿠폰 발급/만료 알림 |
| Inventory | 이벤트 수신 | Inventory → Notification | Kafka Event | 재입고 알림 |
| Settlement | 이벤트 수신 | Settlement → Notification | Kafka Event | 정산 완료 알림 |
| Member | 정보 참조 | Notification → Member | Sync API | 수신 동의, 연락처 조회 |

#### 핵심 유스케이스

1. **트랜잭셔널 알림**: 도메인 이벤트 수신 → 템플릿 렌더링 → 채널 발송
2. **재입고 알림**: 재입고 이벤트 → 구독자 조회 → 일괄 발송 → 구독 해제
3. **마케팅 알림**: 관리자 발송 요청 → 수신 동의 체크 → 시간대 체크 → 발송
4. **템플릿 관리**: CRUD + 변수 정의 + 미리보기
5. **발송 이력 조회**: 채널/상태/기간별 발송 결과 조회

#### 엣지 케이스

- **대량 발송 (10만건+)**: 배치 분할 + 큐 기반 비동기 처리
- **알림톡 실패 → SMS 폴백**: 알림톡 발송 실패 시 자동 SMS 전환
- **기기 토큰 만료**: 푸시 실패 → 토큰 무효화 → 재로그인 시 갱신

### 2.13 CS (고객서비스) 도메인

#### 도메인 책임과 범위

CS 도메인은 **고객 문의, FAQ, 반품/교환 접수 프로세스를 관리**한다.

**범위 포함:**
- 1:1 문의 (구매자 → 플랫폼/셀러)
- FAQ 관리
- 반품/교환 접수 (CS 측면)
- 셀러 답변 관리
- 문의 상태 추적

**범위 제외:**
- 반품 물류 처리 (→ Shipping)
- 환불 처리 (→ Payment)

#### 핵심 비즈니스 규칙

1. **문의 규칙**
   - 문의 유형: 상품문의, 배송문의, 반품/교환, 결제/환불, 기타
   - 셀러 답변 의무: 48시간 이내 (미응답 시 자동 알림 + 페널티)
   - 플랫폼 문의: 24시간 이내 답변 목표
   - 문의에 이미지 첨부 가능 (최대 5장)

2. **반품/교환 접수 규칙**
   - 배송 완료 후 7일 이내 접수 가능 (단순변심)
   - 상품 불량: 14일 이내
   - 사유 필수 입력 + 증빙 사진 (불량의 경우)
   - 접수 → 셀러 승인 → 수거 → 검수 → 환불/교환

3. **FAQ 규칙**
   - 카테고리별 분류 (배송, 결제, 반품, 회원 등)
   - 자주 조회되는 FAQ 상단 노출
   - 관리자만 등록/수정/삭제 가능

4. **문의 상태**
   ```
   SUBMITTED → ANSWERED → RESOLVED
        │          │
        ▼          ▼
   ESCALATED  RE_OPENED
        │
        ▼
   ADMIN_ANSWERED → RESOLVED
   ```

#### 불변 조건

- 답변이 없는 문의는 RESOLVED 될 수 없다
- 반품/교환 접수는 기한 내에서만 가능
- ESCALATED 문의는 관리자만 처리 가능

#### 도메인 용어 사전

| 용어 | 영문 | 정의 |
|------|------|------|
| 문의 | Inquiry | 고객이 제기하는 질문/요청 |
| 답변 | Reply | 셀러 또는 관리자의 문의 응답 |
| 에스컬레이션 | Escalation | 셀러가 해결 못 한 문의를 관리자에게 전달 |
| 반품 접수 | Return Request | 반품 의사를 공식적으로 접수 |
| 교환 접수 | Exchange Request | 교환 의사를 공식적으로 접수 |
| FAQ | Frequently Asked Questions | 자주 묻는 질문과 답변 |
| 응답 기한 | Response Deadline | 답변해야 하는 의무 시간 |
| 페널티 | Penalty | 셀러 미응답 등에 대한 불이익 |
| 증빙 | Evidence | 불량 등을 증명하는 사진/동영상 |
| 중재 | Mediation | 플랫폼이 구매자-셀러 간 분쟁을 조정 |

#### 다른 도메인과의 관계

| 대상 도메인 | 관계 유형 | 방향 | 통신 방식 | 설명 |
|------------|----------|------|----------|------|
| Order | 주문 참조 | CS → Order | Sync API | 문의 대상 주문 정보 조회 |
| Shipping | 반품 트리거 | CS → Shipping | Kafka Event | 반품 접수 승인 → 수거 요청 |
| Seller | 답변 요청 | CS → Seller | Kafka Event | 문의 접수 → 셀러 알림 |
| Member | 회원 참조 | CS → Member | Sync API | 문의자 정보 조회 |
| Notification | 알림 | CS → Notification | Kafka Event | 답변 완료 알림 |

#### 핵심 유스케이스

1. **문의 등록**: 구매자가 문의 작성 → 유형 분류 → 셀러/관리자 할당 → 알림 발송
2. **문의 답변**: 셀러가 답변 작성 → 구매자 알림 → 만족/재문의
3. **반품 접수**: 반품 사유 입력 → 기한 검증 → 셀러 승인 → 수거 이벤트
4. **에스컬레이션**: 셀러 미응답 or 구매자 불만족 → 관리자 전달
5. **FAQ 관리**: 관리자가 FAQ 등록/수정 → 카테고리 분류 → 노출 순서 설정

#### 엣지 케이스

- **48시간 셀러 미응답**: 자동 에스컬레이션 + 셀러 페널티 기록
- **반품 기한 경과 후 접수**: 불량 증빙 시 예외 처리 (관리자 판단)
- **동일 주문 중복 문의**: 기존 문의 연결 제안

### 2.14 Seller (셀러) 도메인

#### 도메인 책임과 범위

셀러 도메인은 **입점 브랜드/셀러의 계정, 심사, 파트너센터, 통계를 관리**한다.

**범위 포함:**
- 입점 신청 및 심사
- 셀러 계정 관리
- 파트너센터 (대시보드)
- 판매 통계 (매출, 전환율, 반품률)
- 셀러 등급 관리
- 정산 계좌 관리

**범위 제외:**
- 상품 등록/관리 (→ Product, 셀러가 Product API를 호출)
- 정산 계산 (→ Settlement)
- 배송 처리 (→ Shipping)

#### 핵심 비즈니스 규칙

1. **입점 심사 규칙**
   - 필수 서류: 사업자등록증, 브랜드 소개서, 대표 상품 이미지
   - 심사 기간: 영업일 기준 5일
   - 심사 기준: 브랜드 적합성, 상품 퀄리티, 가격 경쟁력
   - 결과: 승인/조건부 승인/거부
   - 거부 시 3개월 후 재신청 가능

2. **셀러 등급 규칙**
   | 등급 | 조건 (최근 3개월) | 혜택 |
   |------|-----------------|------|
   | 일반 | 입점 즉시 | 기본 수수료 |
   | 우수 | 월 매출 1,000만+ / 반품률 5% 이하 | 수수료 1%p 할인 |
   | 프리미엄 | 월 매출 5,000만+ / 반품률 3% 이하 | 수수료 2%p 할인 + 전용 CS |

3. **파트너센터 기능**
   - 대시보드: 실시간 매출, 주문현황, 정산현황
   - 상품 관리: 상품 등록/수정/중지
   - 주문 관리: 주문 확인/출고 처리
   - CS 관리: 문의 답변
   - 정산 관리: 정산 내역/정산서 확인
   - 통계: 매출/전환율/반품률/리뷰 분석

4. **셀러 계정 상태**
   ```
   PENDING_REVIEW → APPROVED → ACTIVE
         │                       │
         ▼                       ▼
      REJECTED              SUSPENDED (정책 위반)
                                │
                                ▼
                            TERMINATED
   ```

#### 불변 조건

- 사업자등록번호는 시스템 내 유일해야 한다
- 승인된 셀러만 상품 등록 가능
- 정산 계좌는 사업자등록증의 대표자 또는 법인 명의만 가능

#### 도메인 용어 사전

| 용어 | 영문 | 정의 |
|------|------|------|
| 셀러 | Seller | 플랫폼에 입점한 판매자 |
| 입점 | Onboarding | 셀러가 플랫폼에 등록되는 과정 |
| 심사 | Review/Screening | 입점 자격을 평가하는 과정 |
| 파트너센터 | Partner Center | 셀러 전용 관리 플랫폼 |
| 사업자등록번호 | Business Registration No. | 사업자를 식별하는 고유 번호 |
| 전환율 | Conversion Rate | 상품 조회 대비 구매 비율 |
| 반품률 | Return Rate | 판매 건수 대비 반품 비율 |
| 정산 계좌 | Settlement Account | 정산금을 입금받는 은행 계좌 |
| 정책 위반 | Policy Violation | 플랫폼 규정 위반 행위 |
| 제재 | Suspension | 정책 위반 시 계정 일시 정지 |

#### 다른 도메인과의 관계

| 대상 도메인 | 관계 유형 | 방향 | 통신 방식 | 설명 |
|------------|----------|------|----------|------|
| Product | 소유 관계 | Seller → Product | Sync API | 셀러가 상품 등록/관리 |
| Settlement | 정산 수신 | Settlement → Seller | Sync API | 정산 계좌/수수료율 조회 |
| CS | 답변 의무 | CS → Seller | Kafka Event | 문의 접수 알림 |
| Order | 주문 처리 | Order → Seller | Kafka Event | 주문 접수 알림 |
| Shipping | 출고 처리 | Shipping → Seller | Kafka Event | 출고 요청 알림 |

#### 핵심 유스케이스

1. **입점 신청**: 셀러 정보 입력 → 서류 제출 → 심사 요청
2. **입점 심사**: 관리자 검토 → 승인/거부 → 결과 알림
3. **파트너센터 대시보드**: 실시간 매출/주문/정산 현황 조회
4. **셀러 등급 재산정**: 분기별 배치 → 실적 평가 → 등급 변경
5. **정산 계좌 관리**: 계좌 등록/변경 → 본인 인증 → 확인

#### 엣지 케이스

- **심사 중 서류 보완 요청**: 심사 보류 → 셀러에게 보완 요청 → 재제출
- **정책 위반 제재**: 경고 3회 → 일시정지 → 해제 조건 안내
- **셀러 탈퇴 시 미완료 주문**: 탈퇴 불가, 모든 주문 완료 후 탈퇴 가능

### 2.15 Content (콘텐츠) 도메인

#### 도메인 책임과 범위

콘텐츠 도메인은 **매거진, 코디 추천, OOTD 스냅 등 패션 콘텐츠를 관리**한다.

**범위 포함:**
- 매거진 (에디터 기사)
- 코디 추천 (상품 조합 제안)
- OOTD 스냅 (사용자 착용 사진)
- 태그 시스템
- 콘텐츠-상품 연결

**범위 제외:**
- 상품 상세 (→ Product)
- 전시 배치 (→ Display)
- 리뷰 (→ Review)

#### 핵심 비즈니스 규칙

1. **매거진 규칙**
   - 에디터만 작성 가능 (EDITOR 권한)
   - 임시저장 → 검수 → 발행 → 비공개
   - 상품 태깅: 기사 내 언급 상품 자동/수동 링크
   - 발행 시간 예약 가능

2. **코디 추천 규칙**
   - 상품 2~8개 조합
   - 코디 테마 태그 (캠퍼스룩, 출근룩, 데이트룩 등)
   - 에디터 또는 인플루언서가 작성
   - 코디 내 상품 클릭 → 상품 상세 이동

3. **OOTD 스냅 규칙**
   - 회원 누구나 업로드 가능
   - 착용 상품 태깅 (상품명 or SKU)
   - 좋아요 / 댓글 기능
   - 부적절한 콘텐츠 신고 → 관리자 검토

4. **태그 시스템**
   - 스타일 태그: #스트릿, #미니멀, #캐주얼, #포멀
   - TPO 태그: #데일리, #출근, #여행, #운동
   - 계절 태그: #봄코디, #여름코디 등
   - 사용자 자유 태그

#### 불변 조건

- 발행된 매거진은 최소 1개 이상의 상품 태그를 포함해야 한다
- 코디 추천은 2개 이상 8개 이하의 상품을 포함해야 한다
- OOTD 스냅은 반드시 1장 이상의 이미지를 포함해야 한다

#### 도메인 용어 사전

| 용어 | 영문 | 정의 |
|------|------|------|
| 매거진 | Magazine | 에디터가 작성한 패션 기사/칼럼 |
| 코디 추천 | Coordination | 상품 조합으로 구성된 스타일 제안 |
| OOTD | Outfit Of The Day | 오늘의 착용 사진 (사용자 콘텐츠) |
| 스냅 | Snap | 착용 사진을 의미하는 줄임말 |
| 태그 | Tag | 콘텐츠를 분류/검색하기 위한 키워드 |
| 상품 태깅 | Product Tagging | 콘텐츠 내 상품을 연결하는 행위 |
| 좋아요 | Like | 콘텐츠에 대한 호감 표시 |
| 에디터 | Editor | 매거진/코디를 작성하는 콘텐츠 담당자 |
| 발행 | Publish | 콘텐츠를 공개 상태로 전환 |
| 커버 이미지 | Cover Image | 콘텐츠 목록에 표시되는 대표 이미지 |

#### 다른 도메인과의 관계

| 대상 도메인 | 관계 유형 | 방향 | 통신 방식 | 설명 |
|------------|----------|------|----------|------|
| Product | 상품 연결 | Content → Product | Sync API | 콘텐츠 내 상품 태깅 |
| Display | 전시 배치 | Display → Content | Sync API | 메인에 매거진/코디 노출 |
| Member | 작성자 | Content → Member | Sync API | OOTD 작성자 정보 |
| Search | 검색 인덱스 | Content → Search | Kafka Event | 매거진 검색 인덱싱 |

#### 핵심 유스케이스

1. **매거진 작성/발행**: 에디터 작성 → 상품 태깅 → 검수 → 발행 → 전시 배치
2. **코디 추천 등록**: 상품 선택 → 코디 이미지 업로드 → 태그 → 발행
3. **OOTD 업로드**: 회원이 사진 업로드 → 상품 태깅 → 태그 → 공개
4. **OOTD 피드**: 최신순/인기순 피드 → 좋아요/댓글 → 상품 클릭 구매 전환
5. **콘텐츠 신고**: 부적절 콘텐츠 신고 → 자동 숨김 → 관리자 검토

#### 엣지 케이스

- **태깅된 상품 판매 중지**: 콘텐츠 유지, 상품 링크 비활성화 표시
- **OOTD 어뷰징**: 광고/스팸 콘텐츠 → 자동 필터 + 신고 기반 관리
- **매거진 발행 후 수정**: 수정 가능하나 수정 이력 기록

### 3.2 통신 방식 요약

**동기 호출 (Sync API) — 즉시 데이터 필요:**
- Order → Product (상품 정보 스냅샷)
- Order → Member (배송지 조회)
- Order → Promotion (쿠폰 적용 가능 여부)
- Display → Product (전시 상품 정보)
- Review → Order (구매확정 확인)
- Settlement → Payment (결제 정보 조회)
- Settlement → Seller (정산 계좌, 수수료율)
- Notification → Member (연락처, 수신 동의)

**비동기 이벤트 (Kafka) — Eventually Consistent:**
- Product → Search (인덱스 업데이트)
- Product → Inventory (SKU 재고 초기화)
- Order → Inventory (재고 예약/복원)
- Order → Payment (결제 요청)
- Payment → Order (결제 결과)
- Shipping → Order (배송 상태 동기화)
- Shipping → Settlement (구매확정)
- Review → Member (포인트 적립)
- Inventory → Notification (재입고 알림)

## 4. 핵심 비즈니스 플로우 상세

### 4.1 주문-결제 Saga (성공/실패/보상 트랜잭션)

#### 정상 플로우 (Happy Path)

```
1. [Order] 주문 생성 (ORDER_CREATED)
   → OrderCreatedEvent 발행

2. [Inventory] 재고 예약
   → 각 SKU에 대해 비관적 락 획득
   → 가용재고 확인 → 예약재고 증가
   → StockReservedEvent 발행

3. [Order] 재고 예약 확인
   → 주문 상태: ORDER_CREATED → STOCK_RESERVED

4. [Payment] 결제 요청
   → PG사 승인 API 호출
   → 승인 결과 수신
   → PaymentCompletedEvent 발행

5. [Inventory] 재고 확정 차감
   → 예약재고 → 실물재고 차감
   → StockDeductedEvent 발행

6. [Promotion] 쿠폰 사용 확정
   → 쿠폰 상태: ISSUED → USED
   → CouponUsedEvent 발행

7. [Member] 포인트 차감
   → 사용 포인트 차감
   → PointsDeductedEvent 발행

8. [Order] 주문 확정
   → 주문 상태: STOCK_RESERVED → PAYMENT_COMPLETED
   → OrderConfirmedEvent 발행

9. [Notification] 주문 확인 알림
   → 구매자: 주문 확인 알림톡
   → 셀러: 신규 주문 알림
```

#### 실패 시나리오별 보상 트랜잭션

**Case 1: 재고 예약 실패**

```
1. [Order] OrderCreatedEvent 발행
2. [Inventory] 재고 부족 → StockReservationFailedEvent 발행
3. [Order] 주문 취소 (ORDER_CANCELLED)
4. [Notification] 구매자에게 재고 부족 알림
```

**Case 2: 결제 실패**

```
1~3. 재고 예약 성공
4. [Payment] PG 승인 실패 → PaymentFailedEvent 발행
5. [Inventory] 재고 복원
   → 예약재고 감소, 가용재고 증가
   → StockRestoredEvent 발행
6. [Order] 주문 취소 (ORDER_CANCELLED)
7. [Notification] 구매자에게 결제 실패 알림
```

**Case 3: 결제 승인 후 후처리 실패 (쿠폰/포인트)**

```
1~4. 결제 승인 성공
5. [Promotion] 쿠폰 사용 처리 실패
   → 결제는 유지, 쿠폰 적용 없이 전액 결제 처리
   → 또는: 결제 취소 → 재고 복원 → 주문 취소 (전체 롤백)
   → 관리자 알림 (수동 확인 필요)
```

**Case 4: PG 타임아웃 (결제 상태 불확실)**

```
1~3. 재고 예약 성공
4. [Payment] PG 승인 타임아웃 (30초)
5. [Payment] PG 상태 조회 API 호출 (3회 재시도, 10초 간격)
   Case 4a: 실제 승인됨 → 정상 플로우 5~9 진행
   Case 4b: 미승인 → 결제 실패 플로우 진행
   Case 4c: 조회도 실패 → PENDING 상태 유지 → 5분 후 재조회 스케줄 → 최종 10분 후 미확인 시 취소
```

### 4.2 반품-환불 플로우

#### 단순변심 반품

```
1. [구매자] 반품 요청 (배송완료 후 7일 이내)
   → 반품 사유: "단순변심"
   → 반품 배송비: 구매자 부담 (2,500원)

2. [CS] 반품 접수 자동 승인
   → ReturnRequestedEvent 발행

3. [Shipping] 수거 요청
   → 택배사 수거 API 호출
   → 수거 완료 대기

4. [Shipping] 수거 완료
   → PickupCompletedEvent 발행
   → 셀러에게 검수 요청

5. [Seller] 검수
   → 상품 상태 확인 (택/라벨 제거, 오염, 사용 흔적)
   → 검수 통과 → InspectionPassedEvent
   → 검수 거부 → InspectionRejectedEvent (사유 기재 + 증빙 사진)

6. [검수 통과 시]
   [Payment] 환불 처리
   → 환불 금액 = 결제금액 - 반품배송비(2,500원)
   → PG 부분 취소
   → RefundCompletedEvent 발행

   [Inventory] 재고 복원
   → 실물재고 증가

   [Order] 주문 상태 변경 → RETURN_COMPLETED

7. [검수 거부 시]
   [Shipping] 반송 처리
   → 구매자에게 상품 재발송
   → 구매자에게 검수 거부 사유 안내
```

#### 상품불량 반품

```
1. [구매자] 반품 요청
   → 반품 사유: "상품불량" + 증빙 사진 첨부
   → 반품 배송비: 셀러 부담

2. [CS] 반품 접수
   → 관리자 또는 셀러 승인 필요 (증빙 확인)
   → 48시간 이내 승인/거부

3. [승인 시] 이하 단순변심과 동일 플로우
   → 단, 환불 금액 = 전액 (반품배송비 구매자 부담 없음)
   → 셀러 페널티 기록 (불량률 집계)
```

### 4.3 정산 플로우

```
[일간 집계]
매일 00:00 배치
─────────────────────────────────────────────
1. 전일 구매확정 건 조회
2. 각 건의 정산 금액 계산
   - 판매 금액 (쿠폰 할인 반영)
   - 수수료 차감 (카테고리별 수수료율)
   - 셀러별 합산
3. 일간 정산 데이터 저장 (DAILY_SETTLEMENT)

[주간 정산]
매주 월요일 06:00 배치
─────────────────────────────────────────────
1. 전주(월~일) 일간 정산 합산
2. 셀러별 정산서 생성
   - 판매 건수, 총 판매 금액
   - 수수료 내역 (카테고리별)
   - 반품 상계 금액 (이전 정산 후 발생 반품)
   - 최종 정산 금액
3. 최소 정산 금액(10,000원) 미달 → 이월 처리
4. 정산서 상태: CALCULATED
5. 셀러에게 정산서 알림

[셀러 확인]
월~화요일
─────────────────────────────────────────────
1. 셀러가 파트너센터에서 정산서 확인
2. 이의 없음 → CONFIRMED
3. 이의 있음 → DISPUTED → 관리자 검토 → RE_CALCULATED

[지급]
매주 수요일 10:00 배치
─────────────────────────────────────────────
1. CONFIRMED 상태 정산서 조회
2. 셀러 정산 계좌로 이체 (은행 API)
3. 이체 결과 확인
4. 성공 → PAID, 실패 → 재시도 (수동 처리 전환)
5. 지급 완료 알림
```

### 4.4 선착순 쿠폰 발급 플로우

```
[사전 준비]
1. 관리자가 쿠폰 정책 생성
   - 총 발급 수량: 1,000장
   - 발급 시작 시각: 2026-03-22 10:00:00
   - 회원당 발급 제한: 1장

2. Redis에 카운터 초기화
   - Key: coupon:count:{couponPolicyId}
   - Value: 1000

[발급 플로우]
10:00:00 오픈
─────────────────────────────────────────────
1. [API Gateway] Rate Limiting 적용 (IP당 초당 10회)

2. [Promotion] 발급 요청 수신
   → 발급 시각 검증 (시작 시각 <= now <= 종료 시각)
   → 회원별 중복 발급 체크 (Redis SET: coupon:issued:{policyId})

3. [Redis] 원자적 수량 차감
   → DECR coupon:count:{couponPolicyId}
   → 결과 >= 0: 발급 성공
   → 결과 < 0: 수량 소진 → INCR로 복원 → 실패 응답

4. [Redis → Kafka] 발급 성공 이벤트
   → CouponIssuedEvent (비동기)

5. [Kafka Consumer] DB 영속화
   → coupon 테이블에 INSERT
   → 실패 시 DLQ → 수동 처리

6. [Notification] 발급 완료 푸시 알림

[동시성 보장]
- Redis DECR: 원자적 연산으로 정확히 1,000장만 발급
- SET 기반 중복 체크: SADD 원자 연산으로 회원당 1회 보장
- DB 영속화는 비동기: Kafka Consumer에서 순차 처리
```

### 4.5 재입고 알림 플로우

```
1. [구매자] 품절 상품 "재입고 알림 신청"
   → SKU 단위 구독 (사이즈/색상 특정)
   → notification_restock_subscription 테이블 저장
   → 동일 SKU 중복 구독 불가

2. [Seller] 재고 입고 처리
   → Inventory API: 실물재고 증가
   → 가용재고 0 → N으로 변경 감지

3. [Inventory] RestockedEvent 발행
   → SKU ID, 재입고 수량 포함

4. [Notification] 이벤트 수신
   → 해당 SKU 구독자 조회 (최대 1,000명)
   → 알림 발송 (푸시 + 알림톡)
   → 발송 완료 후 구독 자동 해제

5. [구매자] 알림 수신 → 상품 페이지 이동 → 구매
   → 재입고 수량 한정 시 선착순 구매 가능

[엣지 케이스]
- 1,000명 초과 구독: 선착순 1,000명에게만 발송 (FIFO)
- 재입고 후 즉시 품절: 알림은 발송되지만 구매 시 품절 안내
- 알림 발송 중 재품절: 이미 발송된 알림은 유지, 미발송분은 취소
```

## 5. 크로스커팅 관심사

### 5.1 인증/인가 (JWT + API Gateway)

```
[인증 플로우]
Client → API Gateway → Auth Server → 각 서비스

1. 로그인 요청 → Auth Server
2. 인증 성공 → Access Token (30분) + Refresh Token (14일) 발급
3. API 요청 시 Authorization: Bearer {accessToken}
4. API Gateway에서 JWT 검증 (서명 + 만료)
5. 유효 → 서비스로 라우팅 (X-Member-Id 헤더 추가)
6. 만료 → 401 → Client가 Refresh Token으로 재발급

[인가 구조]
- 역할(Role): MEMBER, SELLER, EDITOR, ADMIN
- 리소스별 권한 검사 (서비스 레벨)
- 셀러: 본인 상품/주문만 접근 가능
- 관리자: 전체 접근 + 심사/정산 관리

[API Gateway 역할]
- JWT 검증
- Rate Limiting (IP별, 회원별)
- 라우팅 (경로 기반)
- 요청 로깅
- CORS 처리
```

### 5.2 로깅/모니터링

```
[로깅 전략]
- 구조화된 로깅 (JSON format)
- 필수 필드: traceId, spanId, timestamp, level, service, message
- 분산 추적: Spring Cloud Sleuth / Micrometer Tracing
- 로그 레벨: ERROR(장애), WARN(주의), INFO(비즈니스), DEBUG(개발)
- 민감정보 마스킹: 카드번호, 주민번호, 전화번호

[모니터링]
- 메트릭: Prometheus + Grafana
  - 비즈니스: 주문수/분, 결제 성공률, 검색 QPS
  - 시스템: CPU, Memory, GC, 커넥션 풀
  - 인프라: 노드 상태, 파드 상태
- 알림: PagerDuty / Slack
  - P1 (즉시): 결제 실패율 > 5%, 서비스 다운
  - P2 (30분 내): 응답 지연 > 3초, 에러율 > 1%
  - P3 (당일): 디스크 > 80%, 메모리 > 70%
```

### 5.3 에러 핸들링 전략

```
[에러 분류]
1. 비즈니스 예외 (Business Exception)
   - 재고 부족, 쿠폰 만료, 중복 요청 등
   - HTTP 4xx 응답
   - 사용자에게 명확한 메시지 전달

2. 시스템 예외 (System Exception)
   - DB 연결 실패, 외부 API 타임아웃 등
   - HTTP 5xx 응답
   - 알림 + 자동 복구 시도

3. 외부 시스템 예외 (External Exception)
   - PG사, 택배사 API 장애
   - 서킷 브레이커 + 폴백

[에러 응답 표준 포맷]
{
  "code": "INVENTORY_INSUFFICIENT",
  "message": "재고가 부족합니다.",
  "detail": "SKU-001의 가용 재고: 0",
  "traceId": "abc-123"
}

[재시도 정책]
- 외부 API: 최대 3회, Exponential Backoff (1s, 2s, 4s)
- Kafka Consumer: 최대 3회 → DLQ
- DB 데드락: 최대 3회, 100ms 간격
```

### 5.4 데이터 일관성 전략

```
[Strong Consistency (즉시 일관성)]
적용 대상:
- 재고 차감/예약 (비관적 락)
- 결제 승인/취소 (동기 API)
- 포인트 잔액 변경 (비관적 락)
- 쿠폰 수량 차감 (Redis 원자 연산)

방법: DB 트랜잭션, Pessimistic Lock, Redis 원자 연산

[Eventually Consistent (최종 일관성)]
적용 대상:
- 검색 인덱스 업데이트 (5초 이내)
- 랭킹 갱신 (10분 주기)
- 정산 데이터 (일간 배치)
- 알림 발송 (비동기)
- 리뷰 평점 집계 (비동기)

방법: Kafka 이벤트 + Outbox 패턴

[Outbox 패턴]
1. 비즈니스 로직 + outbox 테이블 INSERT (같은 트랜잭션)
2. Debezium CDC 또는 폴링으로 outbox → Kafka 발행
3. 발행 완료 후 outbox 레코드 삭제
→ 이벤트 유실 방지
```

### 5.5 캐싱 전략

```
[Redis 캐싱 대상]

| 데이터 | 캐시 키 패턴 | TTL | 무효화 |
|--------|-------------|-----|--------|
| 상품 상세 | product:{id} | 10분 | 상품 수정 이벤트 시 삭제 |
| 카테고리 트리 | category:tree | 1시간 | 카테고리 변경 시 삭제 |
| 실시간 랭킹 | ranking:{type}:{category} | 10분 | 갱신 시 덮어쓰기 |
| 인기 검색어 | search:popular | 10분 | 갱신 시 덮어쓰기 |
| 회원 세션 | session:{token} | 30분 | 로그아웃 시 삭제 |
| 장바구니 | cart:{memberId} | 7일 | 변경 시 업데이트 |
| 쿠폰 수량 | coupon:count:{id} | 없음 | 발급 시 DECR |
| 재고 캐시 | stock:{skuId} | 1분 | 변경 시 삭제 (Write-Through) |

[캐시 전략 유형]
- Look-Aside: 상품, 카테고리 (읽기 많고 쓰기 적음)
- Write-Through: 재고 수량 (정확성 중요)
- Write-Behind: 검색 로그 (쓰기 많고 유실 허용)

[캐시 관련 문제 대응]
- Cache Stampede: 뮤텍스 락 (Redis SETNX) + 스태거 TTL
- Cache Penetration: 빈 결과도 캐시 (TTL 1분) + Bloom Filter
- Cache Avalanche: TTL 랜덤 분산 (기본 ± 20%)
```

### 5.6 이벤트 순서 보장 / 멱등성

```
[이벤트 순서 보장]
- Kafka Partition Key: 도메인별 핵심 식별자
  - 주문 이벤트: orderId
  - 재고 이벤트: skuId
  - 결제 이벤트: paymentId
  - 회원 이벤트: memberId
- 동일 키의 이벤트는 동일 파티션 → 순서 보장
- 컨슈머: 단일 파티션 내 순차 처리

[멱등성 보장]
- 모든 이벤트에 eventId(UUID) 포함
- Consumer: 처리 전 eventId 중복 체크 (idempotency 테이블)
  1. SELECT event_id FROM processed_events WHERE event_id = ?
  2. 존재하면 → 스킵
  3. 없으면 → 처리 + INSERT (같은 트랜잭션)
- HTTP API: Idempotency-Key 헤더 지원 (결제 등 중요 API)
- 멱등성 레코드 보관: 7일 후 자동 삭제

[이벤트 스키마]
{
  "eventId": "uuid",
  "eventType": "OrderCreated",
  "aggregateId": "order-123",
  "aggregateType": "Order",
  "timestamp": "2026-03-22T10:00:00.000000",
  "payload": { ... },
  "metadata": {
    "traceId": "trace-456",
    "version": 1
  }
}
```

### 5.7 장애 격리 (Circuit Breaker)

```
[서킷 브레이커 대상]
- PG사 API (Toss Payments)
- 택배사 API (CJ, 한진, 롯데)
- 소셜 로그인 API (카카오, 네이버)
- SMS/알림톡 API

[서킷 브레이커 설정]
- Resilience4j 사용
- 상태: CLOSED → OPEN → HALF_OPEN
- CLOSED → OPEN: 최근 10건 중 실패율 50% 초과
- OPEN 지속 시간: 30초
```
