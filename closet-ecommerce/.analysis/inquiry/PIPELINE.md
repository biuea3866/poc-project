# 문의 대응 파이프라인

> 버그 리포트, QA 이슈, CS 문의, 동작 원리 질문 시 사용

---

## 파이프라인 흐름

```
[INPUT] 문의/이슈/버그/질문
    │
    ▼
━━ Phase 1: 문의 분류 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    │  팀장이 문의를 읽고 유형 판별
    │  - 버그 (재현 가능한 오류)
    │  - 동작 원리 질문 (왜 이렇게 동작하는지)
    │  - 데이터 이슈 (특정 데이터가 이상한 경우)
    │  - 성능 이슈 (느림, 타임아웃)
    │  - 권한 이슈 (접근 불가, 403)
    │
    ▼
━━ Phase 2: 원인 탐색 (병렬) ━━━━━━━━━━━━━━━━━━━━━━━
    │  문의 유형에 따라 에이전트 할당
    │  ┌─ Agent a: 코드 흐름 추적
    │  └─ Agent b: 데이터/설정 확인
    │
    ▼
━━ Phase 3: 원인 특정 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    │  팀장이 에이전트 결과를 종합
    │  - 근본 원인 (Root Cause) 특정
    │  - 재현 경로 확인
    │  - 영향 범위 파악
    │
    ▼
━━ Phase 4: 해결 방안 제시 ━━━━━━━━━━━━━━━━━━━━━━━━━
    │  - 수정 대상 코드 위치 (파일:라인)
    │  - 수정 방향 제안
    │  - 사이드 이펙트 검토
    │  - 임시 조치 vs 근본 해결 구분
    │
    ▼
[OUTPUT] 문의 대응 보고서
```

## 문의 유형별 처리 전략

### Type 1: 버그 (Bug)
```
키워드: "안 돼요", "에러", "오류", "깨져요", "500", "404"

처리 전략:
1. 증상에서 관련 서비스 식별
2. API 엔드포인트 → Controller → Service → Domain 순 추적
3. 에러 핸들링 로직 확인
4. 엣지 케이스/유효성 검증 누락 확인
```

### Type 2: 동작 원리 질문 (How it works)
```
키워드: "왜 이렇게", "어떻게 동작", "로직이 뭐야", "흐름이"

처리 전략:
1. 관련 도메인/기능 식별
2. 전체 데이터 흐름 추적 (FE → Gateway → BE → DB)
3. 비즈니스 규칙 및 조건문 분석
4. 흐름도 형태로 설명
```

### Type 3: 데이터 이슈 (Data)
```
키워드: "데이터가 안 맞아", "값이 이상해", "누락", "중복", "재고 안 맞아"

처리 전략:
1. 관련 엔티티/테이블 식별
2. 데이터 생성 → 변환 → 저장 경로 추적
3. Kafka 이벤트 Consumer 확인 (비동기 처리 누락?)
4. DB 쿼리 및 조인 로직 확인
```

### Type 4: 성능 이슈 (Performance)
```
키워드: "느려요", "타임아웃", "로딩이 오래", "무한 로딩"

처리 전략:
1. 관련 API 엔드포인트 식별
2. DB 쿼리 분석 (N+1, 인덱스 누락)
3. 외부 API 호출 분석 (PG, 택배사 타임아웃 설정)
4. Kafka Consumer lag 가능성
5. Redis 캐시 미스 확인
6. Elasticsearch 쿼리 성능 확인
```

### Type 5: 권한 이슈 (Permission)
```
키워드: "접근 불가", "403", "권한 없음", "안 보여요"

처리 전략:
1. API Gateway 필터 확인 (회원 등급 기반 인가)
2. member-service 인증/인가 로직 확인
3. RBAC 역할/권한 매핑 확인 (구매자/판매자/관리자)
4. FE 조건부 렌더링 확인
```

## 에이전트 역할 정의

### 팀장 (메인 Claude)
- 문의 유형 분류
- 관련 서비스 사전 식별
- 에이전트 할당 및 탐색 범위 지정
- 결과 종합 및 최종 답변 작성

### Agent a: 코드 흐름 추적가
**목표**: 문제의 코드 경로를 정확히 추적

**분석 항목**:
1. 관련 API 엔드포인트 찾기 (URL 패턴 매칭)
2. Controller → Service → Domain 호출 체인 추적
3. 조건문/분기 로직 분석
4. 에러 핸들링 경로 확인
5. Kafka Consumer/Producer 흐름 확인

**탐색 전략**:
```
1단계: URL/키워드로 Controller 검색
    Grep: @GetMapping, @PostMapping, @RequestMapping + URL 패턴

2단계: Controller에서 호출하는 Service 추적
    해당 메서드의 의존성 주입된 서비스 확인

3단계: Service에서 호출하는 Domain/Repository 추적
    비즈니스 로직, 유효성 검증, 상태 전환 로직 확인

4단계: Repository에서 실행되는 쿼리 확인
    JPA, QueryDSL, Native Query 확인
```

### Agent b: 데이터/설정 확인가
**목표**: 설정, 데이터 모델, 인프라 관점에서 원인 탐색

**분석 항목**:
1. 관련 엔티티/테이블 스키마 확인
2. application.yml 설정 확인
3. Kafka 토픽 설정 확인
4. 권한/역할 설정 확인
5. 캐시 설정 확인
6. 환경별 설정 차이 확인

**탐색 대상**:
```
- 설정: application*.yml, application*.properties
- 엔티티: *Entity.kt
- 마이그레이션: 각 서비스의 src/main/resources/db/migration/
- 권한: closet-member/business/domain/
- 캐시: *CacheConfiguration*, adaptor/redis/
```

## 에이전트 동적 할당 규칙

| 문의 유형 | 할당 에이전트 | 이유 |
|----------|-------------|------|
| 버그 (단순) | Agent a만 | 코드 흐름만 추적하면 충분 |
| 버그 (복잡/서비스간) | Agent a + b | 코드 + 설정/데이터 모두 확인 필요 |
| 동작 원리 질문 | Agent a만 | 코드 흐름 추적이 핵심 |
| 데이터 이슈 | Agent a + b | 코드 흐름 + DB 스키마 확인 |
| 성능 이슈 | Agent a + b | 쿼리 + 설정 + 캐시 모두 확인 |
| 권한 이슈 | Agent b만 | 설정/권한 모델 확인이 핵심 |
| 간단한 질문 | 에이전트 없이 팀장만 | 보고서 참조로 즉답 가능 |

## 빠른 진단 체크리스트

문의 접수 시 팀장이 먼저 확인:

```
□ 어떤 화면/기능에서 발생? → 관련 FE 모듈 식별
□ 어떤 API 호출 관련?     → 관련 BE 서비스 식별 (Gateway 라우팅 참조)
□ 특정 사용자/셀러?       → 권한/등급 문제 가능성
□ 특정 환경에서만?         → 환경별 설정 차이 확인
□ 최근 배포 이후?          → git log로 최근 변경 확인
□ 간헐적 발생?            → 비동기 처리/경합 조건 의심
```

## 출력 어조

산출물은 **팀원과 공유하는 문서**입니다. 읽는 사람이 지치지 않도록 아래 원칙을 따릅니다.

- **핵심부터** — 결론·액션 아이템을 앞에 씁니다. 이유와 배경은 뒤에 써도 됩니다.
- **짧게** — 한 문장으로 쓸 수 있으면 세 문장으로 쓰지 않습니다.
- **구체적으로** — "여러 곳" 대신 "3곳", "느릴 수 있음" 대신 "products 테이블 full scan" 처럼 씁니다.
- **표·불릿 우선** — 비교·목록은 문장보다 표나 불릿으로 씁니다.
- **중립적으로** — 문제를 발견해도 단정짓지 않고 확인 사항으로 전달합니다.

## 출력 형식

분석 결과는 `.analysis/inquiry/results/` 디렉토리에 저장되며,
`common/templates/inquiry_analysis.md` 템플릿을 따릅니다.
