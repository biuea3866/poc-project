# 코드 리뷰 (PR Review) 파이프라인

> PR이 올라왔을 때 변경 영향 범위, 사이드 이펙트, 누락 검토

---

## 파이프라인 흐름

```
[INPUT] PR URL 또는 변경 파일 목록
    │
    ▼
━━ Phase 1: 변경 범위 파악 ━━━━━━━━━━━━━━━━━━━━━━━━
    │  팀장이 PR diff 확인
    │  - 변경 파일 목록 정리
    │  - 변경 유형 분류 (기능/버그픽스/리팩토링/설정)
    │  - 영향 받는 도메인 식별
    │
    ▼
━━ Phase 2: 심층 분석 (병렬) ━━━━━━━━━━━━━━━━━━━━━━
    │  ┌─ Agent 📐: 영향 범위 분석
    │  └─ Agent 🧪: 코드 품질 분석
    │
    ▼
━━ Phase 3: 리뷰 종합 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    │  팀장이 결과 종합
    │  - Must Fix (반드시 수정)
    │  - Should Fix (수정 권장)
    │  - Nice to Have (선택)
    │  - 누락된 변경 사항
    │
    ▼
[OUTPUT] 코드 리뷰 보고서
```

---

## 에이전트 역할 정의

### 팀장 (메인 Claude)
- PR diff 파싱 및 변경 파일 분류
- 에이전트 결과 종합
- 최종 리뷰 코멘트 작성 (심각도별 분류)

### Agent 📐: 영향 범위 분석가
**목표**: 변경된 코드가 다른 곳에 미치는 영향 추적

**분석 항목**:
1. 변경된 함수/클래스를 호출하는 곳 전부 추적
2. 변경된 API 엔드포인트를 사용하는 FE 코드 확인
3. 변경된 도메인 모델을 참조하는 다른 모듈 확인
4. 변경된 Kafka 메시지 포맷을 사용하는 Consumer 확인
5. 변경된 DB 엔티티의 다른 사용처 확인
6. 공유 라이브러리 변경 시 전체 소비자 확인

**탐색 전략**:
```
변경된 파일에서 public 메서드/클래스 추출
    ↓
Grep: 해당 메서드/클래스를 import/사용하는 파일
    ↓
동일 서비스 내 + 다른 서비스 간 영향 확인
    ↓
누락된 변경이 있는지 판단
```

**크로스 서비스 체크리스트**:
```
BE API 변경 → FE API 호출 코드 확인

공유 라이브러리 변경 → 소비 서비스 확인
    - closet-common → 모든 BE 서비스

Kafka 메시지 변경 → Producer/Consumer 양쪽 확인
    - closet-order (Producer) → closet-inventory (Consumer)
    - closet-payment (Producer) → closet-order (Consumer)
    - closet-order (Producer) → closet-shipping (Consumer)

DB 스키마 변경 → 관련 Entity 사용처
    - 각 서비스의 Flyway migration → JPA Entity 사용 서비스
```

### Agent 🧪: 코드 품질 분석가
**목표**: 변경된 코드의 품질, 패턴 준수, 잠재적 버그 검출

**분석 항목**:
1. **잠재적 버그**
   - Null 처리 누락 (Kotlin nullable 타입)
   - 경합 조건 (동시성 이슈, 재고 차감 등)
   - 리소스 누수 (스트림, 커넥션 미해제)
   - 에러 핸들링 누락 (try-catch, onError)

2. **패턴 준수**
   - Hexagonal Architecture 계층 위반 (domain → adaptor 직접 참조)
   - 트랜잭션 경계 적절성
   - Repository 패턴 준수 (Reader/Store 분리)

3. **성능**
   - N+1 쿼리 가능성
   - 불필요한 전체 조회 (findAll without pagination)
   - 대용량 데이터 메모리 로드
   - 재고 조회 시 Lock 전략

4. **보안**
   - SQL Injection 가능성
   - 권한 체크 누락
   - 민감 정보 노출 (로그, 응답)
   - 결제 정보 보안

5. **누락 체크**
   - 테스트 코드 추가 여부
   - 마이그레이션 스크립트 포함 여부
   - API 문서 업데이트 여부

## 에이전트 할당 규칙

| PR 유형 | 할당 에이전트 |
|---------|-------------|
| 단일 파일 수정 (간단) | 에이전트 없이 팀장만 |
| 기능 추가 (단일 서비스) | 📐 + 🧪 |
| 기능 추가 (크로스 서비스) | 📐 + 🧪 (📐 에 크로스 서비스 분석 강조) |
| 리팩토링 | 📐만 (영향 범위가 핵심) |
| 버그 픽스 | 🧪만 (수정 적절성 + 사이드 이펙트) |
| 공유 라이브러리 변경 | 📐 + 🧪 (전체 소비자 확인 필수) |
| 설정/인프라 변경 | 📐만 |

## 리뷰 심각도 분류

```
🔴 Must Fix (Blocker)
   - 런타임 에러 발생 가능
   - 데이터 손상 가능 (재고 불일치, 결제 이중 처리)
   - 보안 취약점
   - 다른 서비스 장애 유발 가능

🟡 Should Fix (Warning)
   - 성능 저하 가능
   - 패턴 위반
   - 에러 핸들링 부족
   - 테스트 누락

🟢 Nice to Have (Suggestion)
   - 코드 가독성 개선
   - 네이밍 개선
   - 중복 코드 제거
   - 주석 추가
```

## 출력 어조

산출물은 **팀원과 공유하는 문서**입니다. 읽는 사람이 지치지 않도록 아래 원칙을 따릅니다.

- **핵심부터** — 결론·액션 아이템을 앞에 씁니다. 이유와 배경은 뒤에 써도 됩니다.
- **짧게** — 한 문장으로 쓸 수 있으면 세 문장으로 쓰지 않습니다.
- **구체적으로** — "여러 곳" 대신 "3곳", "느릴 수 있음" 대신 "products 테이블 full scan" 처럼 씁니다.
- **표·불릿 우선** — 비교·목록은 문장보다 표나 불릿으로 씁니다.
- **중립적으로** — 문제를 발견해도 단정짓지 않고 확인 사항으로 전달합니다.

## verification 파이프라인과의 연계

PR 리뷰 시 verification 결과가 있으면 함께 참조한다.

```
verification 통과 여부 확인:
  .analysis/verification/results/{날짜}_{티켓번호}/verification_report.md
  → "위반 0건" 이면 설계 원칙 검증은 스킵 (이미 통과)
  → "위반 N건" 이면 해당 위반이 PR에서 수정되었는지 확인
```

| 시점 | 파이프라인 | 역할 |
|------|----------|------|
| PR 생성 전 | **verification** | 설계 원칙 + AC 검증 (자체 검증) |
| PR 생성 후 | **pr-review** (이 파이프라인) | 코드 품질 + 영향 범위 + 사이드 이펙트 (팀 리뷰) |

## 출력 형식

분석 결과는 `.analysis/pr-review/results/` 디렉토리에 저장
