# 기술 부채 백로그

## 개요
프로젝트 진행 중 인지한 기술 부채를 추적하고, 우선순위에 따라 해결 계획을 관리한다.

### 우선순위 기준
| 등급 | 기준 |
|------|------|
| P0 (Critical) | 보안/데이터 무결성 위험, 즉시 대응 필요 |
| P1 (High) | 개발 생산성 또는 성능에 큰 영향, Phase 1 내 해결 |
| P2 (Medium) | Phase 2에서 해결 가능, 현재 운영에 큰 지장 없음 |
| P3 (Low) | 개선 사항, 여유 있을 때 처리 |

---

## TD-001: Spring Security 미적용

| 항목 | 내용 |
|------|------|
| **우선순위** | P1 (High) |
| **영향도** | 높음 — 보안 취약점, 인증/인가 로직 분산 |
| **현재 상태** | 수동 JWT 필터(`JwtFilter`)로 인증 처리. `OncePerRequestFilter`를 상속한 커스텀 필터에서 토큰 검증. 인가는 컨트롤러 레벨에서 수동 체크. |
| **문제점** | - CSRF, CORS, Session Fixation 등 기본 보안 설정 누락 |
| | - `@PreAuthorize`, `@Secured` 등 선언적 인가 미사용 |
| | - Security Context 미활용으로 인해 현재 사용자 정보 조회가 비표준 |
| | - 보안 테스트 도구(`@WithMockUser` 등) 미활용 |
| **해결 방안** | 1. `spring-boot-starter-security` 의존성 추가 |
| | 2. `SecurityConfig` 작성 (SecurityFilterChain Bean) |
| | 3. 기존 `JwtFilter`를 `UsernamePasswordAuthenticationFilter` 앞에 등록 |
| | 4. `@PreAuthorize("hasRole('ADMIN')")` 등 선언적 인가 전환 |
| | 5. `SecurityContext`에서 현재 사용자 정보 주입 |
| **예상 소요** | 3~5일 |

## TD-002: QueryDSL 동적 쿼리 미구현

| 항목 | 내용 |
|------|------|
| **우선순위** | P1 (High) |
| **영향도** | 중간 — 조회 성능 및 코드 유지보수성 |
| **현재 상태** | JPA 기본 메서드(`findById`, `findAll`)와 단순 JPQL만 사용. 검색/필터링 시 여러 조건의 조합을 처리하지 못함. |
| **문제점** | - 상품 검색 시 카테고리, 가격 범위, 상태 등 복합 필터링 불가 |
| | - 주문 조회 시 기간, 상태 등 동적 조건 미지원 |
| | - N+1 문제 해결을 위한 fetchJoin 미적용 |
| | - 페이지네이션 최적화 미적용 (count 쿼리 분리 등) |
| **해결 방안** | 1. QueryDSL Q클래스 생성 설정 (kapt/KSP) |
| | 2. `{Entity}RepositoryCustom` + `{Entity}RepositoryImpl` 패턴 적용 |
| | 3. `BooleanExpression` 기반 동적 조건 조합 |
| | 4. `fetchJoin`으로 N+1 해결 |
| | 5. Projection DTO로 조회 최적화 |
| **예상 소요** | 3~4일 |

## TD-003: API 버저닝 전략 미정의

| 항목 | 내용 |
|------|------|
| **우선순위** | P3 (Low) |
| **영향도** | 낮음 — 현재 v1만 존재, 클라이언트 1개 |
| **현재 상태** | URL 경로에 `/api/v1/`을 하드코딩. 버전 업그레이드 시 전략이 정의되지 않음. |
| **문제점** | - v2 API 추가 시 기존 v1과의 공존 방안 미정 |
| | - 브레이킹 체인지 발생 시 클라이언트 마이그레이션 가이드 부재 |
| | - 버전별 Deprecation 정책 미수립 |
| **해결 방안** | 1. URL 기반 버저닝 유지 (`/api/v1/`, `/api/v2/`) |
| | 2. 버전별 Controller 분리 또는 Adapter 패턴 |
| | 3. Deprecation 헤더(`Sunset`, `Deprecation`) 표준 적용 |
| | 4. API Changelog 문서화 |
| **예상 소요** | 1~2일 (정책 수립) |

## TD-004: 분산 트랜잭션(Saga) 미구현

| 항목 | 내용 |
|------|------|
| **우선순위** | P2 (Medium) |
| **영향도** | 높음 — MSA 전환 시 필수, 현재 모놀리스에서는 영향 없음 |
| **현재 상태** | 모놀리스 구조에서 `@Transactional`로 단일 트랜잭션 관리. 주문 생성 시 재고 차감, 결제 승인이 하나의 트랜잭션으로 처리됨. |
| **문제점** | - 서비스 분리 시 분산 트랜잭션 필요 |
| | - 현재 구조는 강결합 (주문 → 상품 재고 차감이 같은 트랜잭션) |
| | - 보상 트랜잭션(Compensation) 로직 미구현 |
| | - 장애 시 부분 실패 복구 메커니즘 부재 |
| **해결 방안** | 1. Phase 2에서 Saga 패턴 (Choreography) 도입 |
| | 2. 도메인 이벤트 발행 (`OrderCreatedEvent` → 재고 차감 → 결제 승인) |
| | 3. 보상 이벤트 정의 (`PaymentFailedEvent` → 재고 복원) |
| | 4. Outbox 패턴으로 이벤트 발행 보장 |
| | 5. Kafka를 이벤트 브로커로 사용 |
| **예상 소요** | 2~3주 |

## TD-005: 캐싱 전략 미구현

| 항목 | 내용 |
|------|------|
| **우선순위** | P2 (Medium) |
| **영향도** | 중간 — 성능 최적화, 현재 트래픽 수준에서는 영향 미미 |
| **현재 상태** | 모든 조회가 DB 직접 접근. 캐시 레이어 없음. |
| **문제점** | - 상품 상세 조회 등 빈번한 읽기 요청이 매번 DB 접근 |
| | - 카테고리 목록 등 변경 빈도 낮은 데이터도 매번 조회 |
| | - JWT 블랙리스트만 Redis 사용, 그 외 캐시 미활용 |
| | - 트래픽 증가 시 DB 부하 급증 예상 |
| **해결 방안** | 1. Redis를 캐시 레이어로 도입 (`spring-boot-starter-data-redis`) |
| | 2. `@Cacheable`, `@CacheEvict` 적용 |
| | 3. 캐시 대상 식별: 상품 상세, 카테고리 목록, 회원 등급 |
| | 4. 캐시 무효화 전략: TTL + 이벤트 기반 무효화 |
| | 5. Cache-Aside 패턴 적용 |
| | 6. 캐시 키 네이밍 컨벤션: `{service}:{entity}:{id}` |
| **예상 소요** | 3~4일 |
