# ADR-008: 에러 코드 체계

## 상태: 승인

## 컨텍스트
API 에러 응답에서 클라이언트가 에러 유형을 식별하고 적절히 대응할 수 있는 체계적인 에러 코드가 필요하다.
- HTTP 상태 코드만으로는 세부 에러 구분 불가
- 서비스별 에러를 네임스페이스로 분리 필요
- 에러 메시지의 다국어 처리 확장 가능성
- 로그 추적과 디버깅 편의성

## 결정
서비스별 prefix + 숫자 번호로 에러 코드 체계를 정의한다.

| Prefix | 서비스 | 범위 |
|--------|--------|------|
| C | 공통 (Common) | C001 ~ C099 |
| M | 회원 (Member) | M001 ~ M099 |
| P | 상품 (Product) | P001 ~ P099 |
| O | 주문 (Order) | O001 ~ O099 |
| PAY | 결제 (Payment) | PAY001 ~ PAY099 |

- 에러 코드는 `enum class`로 관리 (각 모듈별)
- API 응답: `{ "success": false, "data": null, "error": { "code": "M001", "message": "이미 등록된 이메일입니다" } }`

## 이유
- 서비스별 prefix로 에러 발생 지점을 즉시 식별
- 숫자 범위 예약으로 향후 에러 추가에 유연
- 클라이언트에서 에러 코드 기반 분기 처리 가능
- 로그 검색 시 에러 코드로 빠른 필터링

## 결과
- 각 모듈의 `exception` 패키지에 에러 코드 enum 정의
- `BusinessException` 추상 클래스: 에러 코드 + HTTP 상태 코드 매핑
- `GlobalExceptionHandler`에서 일관된 에러 응답 포맷 생성
- 에러 코드 레지스트리 문서(`docs/error-codes.md`)로 전체 목록 관리
- 에러 메시지는 코드 내 기본값 + MessageSource 확장 가능

## 대안 (검토했으나 선택하지 않은 것)

### HTTP 상태 코드만 사용
- 장점: 표준적, 추가 구현 불필요
- 단점: 400 Bad Request 하나로 다양한 에러를 구분 불가
- 기각 사유: 비즈니스 에러의 세밀한 구분이 불가

### 전역 단일 에러 코드 체계 (E001, E002...)
- 장점: 단순한 번호 체계
- 단점: 서비스 규모 증가 시 번호 충돌, 소속 서비스 식별 불가
- 기각 사유: 서비스별 분리가 없으면 에러 추적 및 관리 어려움

### 계층형 에러 코드 (MEMBER.AUTH.DUPLICATE_EMAIL)
- 장점: 매우 명시적, 자기 설명적
- 단점: 코드가 길어짐, 클라이언트 파싱 복잡
- 기각 사유: 간결한 코드 + 메시지 조합이 API 응답에 더 적합
