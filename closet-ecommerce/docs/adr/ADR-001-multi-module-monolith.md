# ADR-001: 멀티 모듈 모놀리스 → 점진적 MSA 전환

## 상태: 승인

## 컨텍스트
이커머스 학습 프로젝트에서 처음부터 MSA로 시작할지, 모놀리스로 시작할지 결정 필요.
- 4개 도메인(회원, 상품, 주문, 결제)이 존재
- 1인 개발 환경에서 운영 복잡성 최소화 필요
- 향후 서비스 분리 가능성을 열어두어야 함

## 결정
Phase 1은 멀티 모듈 모놀리스로 시작하고, Phase 2부터 서비스 분리.
- 모듈 경계 = 향후 서비스 경계
- 모듈 간 통신은 인터페이스로 추상화 (향후 HTTP/Kafka로 교체 가능)
- 각 모듈은 자체 도메인 패키지 구조를 유지

## 이유
- 1인 개발에서 MSA 운영 오버헤드(서비스 디스커버리, 분산 트레이싱, 컨테이너 오케스트레이션)가 학습 효과를 저해
- 모듈 경계만 잘 나누면 분리는 나중에 가능 (Modular Monolith → MSA는 검증된 전환 패턴)
- 트랜잭션 관리가 단순 (Saga 불필요)
- 단일 배포 단위로 CI/CD 파이프라인이 단순

## 결과
- 각 모듈은 독립 패키지로 분리 (`com.closet.{module}`)
- 모듈 간 의존은 인터페이스(Port)로만 허용
- 직접적인 엔티티 참조 금지 — ID 참조만 허용
- Phase 2에서 Kafka 이벤트 기반 분리 시작
- 모듈별 독립 테스트 가능한 구조 유지

## 대안 (검토했으나 선택하지 않은 것)

### 처음부터 MSA
- 장점: 독립 배포, 기술 스택 다양성
- 단점: 1인 개발에 과도한 인프라 비용, 분산 트랜잭션 복잡성
- 기각 사유: 학습 프로젝트에서 운영 오버헤드가 도메인 학습을 방해

### 단순 모놀리스 (모듈 분리 없이)
- 장점: 가장 단순한 구조
- 단점: 도메인 간 결합도 증가, 향후 분리 비용 급증
- 기각 사유: 모듈 경계 없이 개발하면 빅볼 오브 머드(Big Ball of Mud)로 귀결
