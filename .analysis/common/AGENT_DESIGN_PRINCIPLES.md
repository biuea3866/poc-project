# 에이전트 설계 원칙

> 모든 파이프라인에 공통 적용되는 에이전트 구성 원칙

---

## 핵심 원칙

### 1. 계층적 에이전트 구조 (Hierarchical Agent Structure)

```
[사용자 요청]
    │
    ▼
[팀장 (메인 Claude)]
    │  - 요청 파싱, Phase 계획, 최종 산출물 종합
    │
    ├─▶ [리드 A] ─── spawn ──┬─ 팀원 A-1
    │                        ├─ 팀원 A-2
    │                        └─ 팀원 A-3
    │
    ├─▶ [리드 B] ─── spawn ──┬─ 팀원 B-1
    │                        └─ 팀원 B-2
    │
    └─▶ [리드 C] ─── spawn ──┬─ 팀원 C-1
                             ├─ 팀원 C-2
                             └─ 팀원 C-3
```

- **팀장**: 사용자 커뮤니케이션, Phase 전환, 리드 할당, 최종 종합
- **리드**: Phase 하나를 책임. 팀원을 spawn하여 작업 분배, 결과 수집 후 팀장에게 반환
- **팀원**: 단일 분석/설계 단위를 수행. 결과를 리드에게 반환

### 2. 리드 에이전트 동작 규칙

리드 에이전트는 다음 프롬프트 패턴을 따른다:

```
당신은 [Phase명]의 리드입니다.

## 당신의 역할
1. 아래 작업을 팀원 에이전트를 spawn하여 병렬로 수행하세요
2. 팀원들의 결과를 수집하여 하나의 산출물로 종합하세요
3. 종합 결과를 .md 파일로 Write하세요

## 팀원 할당 계획
- 팀원 1: [역할] - [담당 범위]
- 팀원 2: [역할] - [담당 범위]
- ...

## 입력 컨텍스트
[이전 Phase 결과물 요약]

## 출력
[파일 경로 + 형식]
```

### 3. 최대 병렬화 (Max Parallelism)
- 리드 에이전트들은 **팀장이 병렬로 spawn**
- 리드 내부에서 팀원들도 **병렬로 spawn**
- 결과: 2-depth 병렬화로 처리량 극대화

### 4. 단일 책임 (Single Responsibility)
- 각 팀원 에이전트는 **하나의 명확한 질문**에 답한다
- 리드 에이전트는 **하나의 Phase 산출물**을 책임진다
- 팀장은 **전체 파이프라인 오케스트레이션**만 한다

### 5. 입출력 명시 (Explicit I/O)
- 팀장 → 리드: 이전 Phase 결과 요약 + 출력 파일 경로 지정
- 리드 → 팀원: 구체적 분석 범위 + 출력 형식 지정
- 팀원 → 리드: 마크다운 텍스트 반환
- 리드 → 파일: Write tool로 종합 산출물 직접 작성

### 6. 적응적 스케일링 (Adaptive Scaling)
- PRD 규모/복잡도에 따라 에이전트 수를 동적 조정
- 소규모: 리드 1~2명 (팀원 없이 리드가 직접 수행)
- 중규모: 리드 2~3명 × 팀원 2~3명
- 대규모: 리드 3~5명 × 팀원 3~5명

## Phase별 리드-팀원 구조

### Phase 1: 분석 (Analysis)

```
리드: 분석 리드
├── 팀원 A: PRD 요구사항 Gap 분석
├── 팀원 B: 핵심 도메인 코드 분석
├── 팀원 C: DB 스키마 & 데이터 흐름 분석
├── 팀원 D: Kafka/이벤트 & 외부 연동 분석
└── 팀원 E: 디자인 ↔ PRD 정합성 (Optional)
산출물: gap_analysis.md + tech_analysis.md
```

### Phase 2: 설계 (Design)

```
리드: 설계 리드
├── 팀원 F: ERD + 도메인 모델 설계
├── 팀원 G: API 스펙 설계
├── 팀원 H: 이벤트 + 스케줄러 설계
└── 팀원 I: 시퀀스 다이어그램 (Mermaid)
산출물: tdd.md
```

### Phase 3: 티켓 (Tickets)

```
리드: 티켓 리드
├── 팀원 J: 인프라 티켓 (DB, Kafka, 설정)
├── 팀원 K: 도메인/서비스 티켓
├── 팀원 L: API/이벤트 티켓
└── 팀원 M: 테스트 케이스
산출물: tickets/*.md + _overview.md
```

## 리드 프롬프트 템플릿

### 분석 리드 템플릿

```
당신은 [PRD명] 분석 Phase의 리드입니다.

## 역할
1. 아래 팀원 에이전트를 Agent tool로 병렬 spawn하세요
2. 모든 팀원 결과를 수집한 후 종합하세요
3. Write tool로 산출물을 파일에 저장하세요

## 팀원 할당
- Agent "Gap 분석": PRD에서 BE 관점 애매/누락/고려사항 도출
- Agent "도메인 코드 분석": [레포]에서 관련 도메인 모델/서비스/API 탐색
- Agent "DB/인프라 분석": DB 스키마, Kafka 토픽, 캐시 구조 탐색

## 입력
[PRD 요약 내용]

## 출력
파일: [경로]/gap_analysis.md + tech_analysis.md
```

### 설계 리드 템플릿

```
당신은 [PRD명] TDD 작성 Phase의 리드입니다.

## 역할
1. 이전 Phase(분석) 결과를 기반으로 팀원을 spawn하세요
2. 각 팀원이 TDD의 한 섹션을 작성하도록 지시하세요
3. 결과를 종합하여 하나의 tdd.md로 Write하세요

## 팀원 할당
- Agent "ERD+도메인": 도메인 모델 변경, Flyway DDL, 인덱스
- Agent "API 스펙": REST API 엔드포인트, Request/Response
- Agent "이벤트 설계": Spring Event, Kafka 토픽, 스케줄러
- Agent "다이어그램": Mermaid 시퀀스 다이어그램 N개

## 입력
[이전 Phase 결과 요약]

## 출력
파일: [경로]/tdd.md
```

### 티켓 리드 템플릿

```
당신은 [PRD명] 구현 티켓 작성 Phase의 리드입니다.

## 역할
1. TDD를 기반으로 팀원을 spawn하여 레이어별 티켓을 작성하세요
2. 각 팀원이 담당 레이어의 티켓을 .md 파일로 직접 Write하게 하세요
3. 전체 요약(_overview.md)을 직접 작성하세요

## 팀원 할당
- Agent "인프라 티켓": DB migration, Kafka topic, 프로젝트 설정
- Agent "도메인 티켓": 도메인 모델, 서비스, UseCase
- Agent "API/이벤트 티켓": Controller, EventHandler, 스케줄러
- Agent "테스트 티켓": 단위/통합/E2E 테스트

## 입력
[TDD 요약]

## 출력
파일: [경로]/tickets/*.md + _overview.md
```

## 파이프라인별 권장 구성

| 파이프라인 | 리드 수 | 총 팀원 수 | 총 에이전트 |
|-----------|:------:|:---------:|:---------:|
| PRD 분석 | 1 | 3~4 | 4~5 |
| 문의 대응 | 1 | 2~3 | 3~4 |
| 장애 대응 | 2 | 4~5 | 6~7 |
| PR 리뷰 | 1 | 2~3 | 3~4 |
| 배포 영향 | 1 | 3~4 | 4~5 |
| 리팩토링 | 2 | 4~6 | 6~8 |
| 온보딩 | 1 | 2~3 | 3~4 |
| API 변경 | 1 | 3~4 | 4~5 |
| **BE 구현 설계** | **3** | **9~12** | **12~15** |

---

## 팀장(메인 Claude) 역할

1. **요청 파싱**: 사용자 요청을 분석하여 파이프라인/Phase 결정
2. **리드 할당**: Phase별 리드 에이전트를 병렬 spawn
3. **컨텍스트 전달**: 이전 Phase 결과를 다음 Phase 리드에게 전달
4. **품질 게이트**: Phase 완료 시 결과물 확인
5. **사용자 커뮤니케이션**: 진행 상황 보고, 의사결정 요청
6. **결과 종합은 리드가 한다**: 팀장은 리드의 산출물을 그대로 사용

---

## 에러 처리

| 상황 | 대응 |
|------|------|
| 팀원 실패 | 리드가 해당 팀원을 재spawn 또는 직접 수행 |
| 리드 실패 | 팀장이 리드를 재spawn |
| Rate Limit | 팀장이 대기 후 재시도. 사용자에게 상황 공유 |
| 컨텍스트 초과 | 팀장이 세션을 분리하여 다음 대화에서 이어감 |
