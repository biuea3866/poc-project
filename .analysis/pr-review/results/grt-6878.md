## 요구사항 충족 분석

**요구사항**: 평가지가 할당되어 있으면, 열람 범위 설정과 무관하게 면접뷰에서 평가 가능해야 한다.

### `getAccessibleProcessesConditionForOneDayInterview()` 로직 검토

```kotlin
// isOnIpp = true (열람 범위 = 평가 대상 지원자만)
// isOnReadOtherEvaluationContent = false (다른 평가 내용 열람 불가)
// → 아래 조건으로 접근 판단

evaluationModules.isEvaluator  // 평가 모듈의 평가자
|| isManager                   // 매니저
|| evaluation.isEvaluator      // ← 해당 평가에 배정된 평가자
```

**분석**: `evaluation.isEvaluator` 조건이 있으므로 요구사항의 핵심("할당된 평가지가 있으면 접근 가능")은 **충족**됨.

## 리뷰 결과

### 🔴 Must Fix

**1. `/internal/evaluations/contents` 권한 체크 제거**
`checkApplicantPermissionUseCase`가 주석 처리된 채로 배포될 경우, 일반 평가 목록 조회 API에서 권한 검증이 완전히 누락됨. greeting-aggregator의 `@Authorization`이 선행 방어선이지만, `/internal/` API가 다른 서비스에서도 호출될 경우 무방비 상태가 됨.

→ **확인 필요**: 이 주석 처리가 의도적인 것인지, 아니면 임시 조치인지. 의도적이라면 왜 제거해야 하는지 명확한 근거가 필요.

**2. FE 미대응 — 403 발생 위험**
`FAKE_ONE_DAY_INTERVIEW` 활성화 워크스페이스에서 FE는 여전히 `/app/candidates/detail`만 호출함. 해당 엔드포인트는 `@Authorization(READ_APPLICANT)` 복원 상태이므로, 열람 범위 제한 대상 평가자가 면접뷰 접근 시 **403** 발생 가능.

```
영향 레포: greeting_front, greeting_front_mobile-next, next-greeting
사용 파일: src/@api/candidate/apis.ts
신규 엔드포인트: /app/candidates/detail/one-day-interview (FE에서 아직 미사용)
```

→ **FE 작업이 함께 배포되어야 이 티켓이 완성**됨. 현재 상태로는 원데이 면접 워크스페이스에서 버그가 해결되지 않음.

---

### 🟡 Should Fix

**3. `evaluations.associateBy(applicantId)` 중복 덮어쓰기 위험**
지원자 한 명이 동일 공고에서 복수의 평가를 가질 경우, `associateBy`는 마지막 항목만 유지함. `groupBy`로 변경 또는 중복 가능 여부 도메인 확인 필요.

**4. aggregator `@Authorization` 제거 → new-back 권한 체크 우회 가능성**
greeting-aggregator에서 `@Authorization(READ_APPLICANT)`을 제거한 API가 greeting-new-back 내부 API를 호출할 때, new-back 쪽에 `privateBetweenProcess` 권한 체크가 포함된 경로를 타는지 확인 필요.
`CandidateAdaptor` / `CandidateNewBackFeignClient`가 호출하는 new-back API 엔드포인트 확인 권장.

**5. 사내추천 케이스 수동 검증**
"사내추천 권한은 원복" 커밋이 트리 SHA 분석 결과 실질적 코드 변경이 없는 커밋일 가능성 있음. 원데이 면접 활성화 워크스페이스에서 사내추천 지원자의 면접뷰 접근이 요구사항대로 동작하는지 수동 검증 필요.

**6. `evaluations.associateBy { it.applicantId }` 덮어쓰기 위험**
동일 지원자에 다중 Evaluation이 존재하면 마지막 값만 남아 권한 판단 왜곡 가능. 정상 케이스에서는 발생하지 않지만 데이터 정합성 이슈 시 보안 판단에 영향.

**7. `CheckApplicantPermissionService` — Hexagonal 위반**
`CheckApplicantPermissionService`(domain)가 `EvaluationJpaRepository`(adaptor)를 직접 참조. 기존 코드지만 이번 수정과 연관된 경로이므로 확인 권장.

---

### 🟢 Nice to Have

**6. 테스트 커버리지 보강**
`getAccessibleProcessesConditionForOneDayInterview()`의 핵심 분기에 대한 테스트 케이스가 미확인.
추가 권장 케이스:
- `isOnIpp=true, isOnReadOtherEvaluationContent=false, evaluation.isEvaluator=true` → 접근 허용
- `isOnIpp=true, isOnReadOtherEvaluationContent=false, evaluation.isEvaluator=false` → 접근 차단
- `isOnIpp=false` → 항상 접근 허용

**7. Revert 커밋 메시지**
"api 모으기 Revert" 이유가 커밋 메시지에 없어 히스토리 파악이 어려움.

## 결론

작업 방향은 올바름 — `evaluation.isEvaluator` 조건으로 요구사항 핵심(할당된 평가지가 있으면 접근 가능)을 구현하고 있음.

**단, 현재 상태로는 배포 불가**: FE에서 신규 엔드포인트(`/candidates/detail/one-day-interview`)를 호출하도록 대응 작업이 완료되어야 버그가 실제로 해결됨. FE 티켓이 별도로 존재하거나 함께 작업 중인지 확인 필요.
