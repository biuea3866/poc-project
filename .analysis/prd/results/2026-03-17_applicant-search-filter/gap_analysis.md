## 1. 애매한 요구사항 (의사결정 필요)

### 1-1. 검색

| # | 항목 | PRD 기술 | 애매한 점 | 필요한 결정 |
|---|------|----------|-----------|-------------|
| 1 | 전문 텍스트 검색 범위 | "지원서 내 모든 텍스트 검색" + "업로드파일 파싱된 텍스트" + "태그" | 현재 BE(`filterContact`)는 name/phone/email/examNumber만 LIKE 검색. "모든 텍스트"의 구체적 범위 불명확 | 사전질문 응답, 자기소개서 텍스트, 추가질문 응답도 포함하는지 확인 필요 |
| 2 | 검색 엔진 | PRD에 명시 없음 | 현재 ATS 지원자 검색은 MySQL LIKE, TRM은 OpenSearch(`offercent-search`). ATS 쪽 전문검색을 OpenSearch로 전환할지, DB fulltext로 할지 | ATS 지원자용 OpenSearch 인덱스 신규 생성 여부 결정 필요 |
| 3 | 검색 vs 필터 조합 | PRD에 검색+필터 동시 적용 여부 미기술 | 검색 결과에 필터를 중첩 적용 가능해야 하는지, 독립 동작인지 | UX 확인 필요 |

### 1-2. 필터

| # | 항목 | PRD 기술 | 애매한 점 | 필요한 결정 |
|---|------|----------|-----------|-------------|
| 4 | 패싯 검색 결과 수 | "속성별 검색결과 수가 출력되어야 함" | 필터 적용 전 전체 기준 카운트인지, 현재 적용된 필터 기준 카운트인지 불명확 | 패싯 카운트 계산 시점/기준 확인 필요 |
| 5 | NOT 연산 | "속성별로 not을 적용할 수 있어야 함" | 모든 필터 항목에 NOT 적용 가능한지, 특정 항목만인지 | 범위 확정 필요 |
| 6 | 이직빈도 필터 | "이직빈도" 필터 항목 존재 | 이직빈도 계산 로직 미정의 (경력 수 기준? 평균 근속 기간 기준?) | 계산 공식 확정 필요 |
| 7 | 외국어시험 점수 필터 | "외국어시험(종류/시험명/점수)" | 점수 범위 필터인지, 특정 시험 + 점수 조합 필터인지 | 필터 UI/로직 상세 설계 필요 |
| 8 | 필터 저장 | "자주 사용하는 필터 저장 가능" | 공고별 저장인지 워크스페이스 전체인지, 개인별인지 공유인지 | 저장 단위/공유 범위 확정 필요 |
| 9 | 값 없는 필터 | "값이 없는 필터 유형은 비활성화 표시" | 공고별 지원서 양식이 다를 때, 해당 공고에 수집하지 않는 항목은 필터 자체를 숨기는지, 비활성만 하는지 | 동작 방식 확인 필요 |

## 2. 누락된 요구사항 (PRD에 없지만 BE가 처리해야 하는 항목)

| # | 항목 | 상세 | 우선순위 |
|---|------|------|----------|
| 1 | ATS 지원자용 OpenSearch 인덱스 설계 | 현재 `offercent-search`의 `candidate_embedding` 인덱스는 TRM용. ATS 지원자 40+ 필터를 지원하려면 별도 인덱스 또는 기존 확장 필요 | 높음 |
| 2 | 패싯 카운트 API | 필터별 결과 수를 반환하는 Aggregation 쿼리 설계. 현재 BE에 해당 기능 없음 | 높음 |
| 3 | 필터 저장 테이블/API | `saved_filters` 또는 유사 테이블 신규 생성 필요 | 중간 |
| 4 | 리스트뷰 커스텀 설정 저장 | 사용자별/공고별 노출 항목 설정 저장 테이블/API 필요 | 중간 |
| 5 | 스크리닝 규칙/점수 저장 모델 | 공고별 스크리닝 조건, 가점/감점 규칙, 커트라인 저장 테이블 필요 | 높음 |
| 6 | 스크리닝 결과 저장 | 지원자별 스크리닝 점수/판정 결과 저장 테이블 필요 | 높음 |
| 7 | 데이터 마이그레이션 | 새싹 Phase 1에서 정규화된 데이터를 새 OpenSearch 인덱스에 초기 색인 | 높음 |
| 8 | 필터 적용 항목 리스트뷰 표시 API | 필터 적용 시 해당 항목을 리스트 응답에 동적 포함하는 로직 | 중간 |
| 9 | CDC 파이프라인 확장 | ATS 지원자 데이터 변경 시 OpenSearch 인덱스 실시간 업데이트 파이프라인 | 높음 |
| 10 | 권한 체계 연동 | 필터/스크리닝 기능에 대한 플랜별 접근 제어 (Business 이상만 스크리닝 사용) | 중간 |

## 3. 추가 고려사항

### 3-1. 성능

| 항목 | 리스크 | 대응 방안 |
|------|--------|-----------|
| 패싯 카운트 쿼리 비용 | 40+ 필터 항목의 Aggregation은 데이터량에 따라 수 초 소요 가능. 5000+ 지원자 공고 존재 | OpenSearch Aggregation 활용, 캐싱 전략 수립 |
| 전문 텍스트 검색 | MySQL LIKE 검색은 풀스캔 유발. 지원서 전체 텍스트 검색은 OpenSearch 필수 | ATS 지원자용 OpenSearch 인덱스 도입 |
| 리스트뷰 동적 컬럼 | 필터 적용 항목에 따라 응답 필드가 변동되면 API 복잡도 증가 | 별도 필드 조회 API vs 일괄 반환 방식 결정 필요 |
| 스크리닝 일괄 계산 | 수천 명 지원자에 대한 점수 계산/태그 부여 시 트랜잭션 크기 | 비동기 처리(이벤트 기반) 또는 배치 처리 검토 |

### 3-2. 동시성

| 항목 | 리스크 | 대응 방안 |
|------|--------|-----------|
| 스크리닝 규칙 변경 중 지원자 유입 | 규칙 변경과 신규 지원자 자동 스크리닝 간 race condition | 규칙 버전 관리 또는 낙관적 잠금 |
| 다중 채용담당자 필터 저장 | 같은 공고에서 여러 담당자가 동시에 필터 저장 시 충돌 가능 | 개인별 필터 저장으로 분리 |

### 3-3. 보안

| 항목 | 리스크 | 대응 방안 |
|------|--------|-----------|
| 민감 정보 필터 | 성별/나이/장애/보훈 필터는 채용 차별 이슈 가능 | 필터 사용 로그 기록, 감사 추적 기능 |
| 블라인드 처리 연동 | 블라인드 설정된 항목이 필터에 노출되면 안 됨 | 블라인드 설정과 필터 항목 연동 로직 필수 |

## 4. PO/기획자에게 확인할 질문

### 검색 관련
1. "지원서 내 모든 텍스트"에 사전질문 응답, 자기소개서 본문, 추가질문 응답이 모두 포함되는지?
2. 검색어 자동완성 또는 최근 검색어 기능이 필요한지?
3. 검색과 필터를 동시에 적용할 수 있어야 하는지?

### 필터 관련
4. 패싯 카운트는 다른 필터 적용 후의 결과 기준인지(교차 카운트), 필터 미적용 전체 기준인지?
5. NOT 연산은 모든 필터 항목에 적용 가능한지, 특정 항목만인지?
6. 필터 저장은 개인별인지, 공고별 공유인지? 최대 저장 개수 제한이 있는지?
7. 이직빈도 필터의 계산 기준은? (경력 수 기반? 평균 근속 기간 기반?)
8. 외국어시험 필터에서 "종류+시험명+점수"를 조합 필터로 제공하는지, 각각 독립 필터인지?

### 리스트뷰 관련
9. 리스트뷰 커스텀 설정은 개인별 저장인지, 공고별 공유인지?
10. 필터 적용 시 리스트뷰에 자동 추가되는 컬럼은 필터 해제 시 자동 제거되는지?

### 스크리닝 관련
11. 스크리닝 점수 범위와 총점 계산 방식은? (단순 합산? 가중치?)
12. "자동 합격/불합격 태그 부여"의 "태그"는 실제 지원자 상태 변경(합격/불합격)인지, 별도 라벨인지?
13. 스크리닝 설정 후 기존 지원자에게도 소급 적용하는지?
14. 스크리닝 결과를 되돌리기(undo) 할 수 있어야 하는지?
15. 스크리닝 기능은 Business 플랜 이상만 사용 가능한지? 다른 플랜 제한이 있는지?

### 기타
16. "전체 지원자 리스트"에서도 검색/필터/스크리닝이 동일하게 적용되는지, 기능 차이가 있는지?
17. 모바일 환경에서의 필터 UX는 별도 기획이 있는지?
