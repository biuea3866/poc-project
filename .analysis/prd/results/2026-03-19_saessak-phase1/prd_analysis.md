## 1. 프로젝트 전체 구조

### 1.1 핵심 목표
1. **지원자 정보 정규화**: 비정형 지원서 항목을 Category > Component > Attribute 3계층으로 구조화

## 2. PRD별 BE 영향 분석

### PRD #1: 지원자정보 수집 항목 정의

**핵심**: 지원서에서 수집하는 모든 항목과 필드타입을 정의

| 항목 | 영향도 | 상세 |
|------|--------|------|
| 필드타입 정의 (10종) | High | 텍스트(단답/장문), 토글, 단일선택(옵션/검색), 날짜, 기간, 이미지첨부, 파일첨부, 숫자, URL |
| 수험번호 신규 도입 | High | 공고ID-6자리 일련번호, 동시성 제어 필수 |
| 지원분야 저장 | Medium | 1~3순위 + 표준 직군/직무 코드 매핑 |
| 날짜 수집 단위 | Medium | YMD/YM 선택, 변경 시 기존 데이터 호환 |

**필요 엔티티**: ExamNumber, ApplicationFieldType(enum), DateCollectionUnit(enum)

### PRD #2: 지원서 설정 정보구조 및 Template 관리

**핵심**: 3계층 구조 + 템플릿 CRUD

| 항목 | 영향도 | 상세 |
|------|--------|------|
| 3계층 구조 | Critical | Category(11종) > Component(81종) > Attribute(231종) |
| 템플릿 관리 | High | 워크스페이스 단위, 간편이력서/상세이력서 기본 제공 |
| 항목 편집 | High | 사용여부, 필수여부, 노출순서, 타이틀 변경 |
| Dataset 관리 | Medium | 기본항목/커스텀항목 구분, 커스텀 항목 CRUD |

**필요 엔티티**: ApplicationForm, ApplicationFormCategory, ApplicationFormComponent, ApplicationFormAttribute, ApplicationFormTemplate

### PRD #3: 열람권한(블라인드) 설정

**핵심**: Attribute 단위로 Role 기반 열람 제어

| 항목 | 영향도 | 상세 |
|------|--------|------|
| 열람권한 설정 | Critical | Attribute별 열람 가능 Role 목록 (기본: ALL) |
| 일괄 변경 | High | Category/Component 단위 일괄 적용 |
| PDF 자동생성 제거 | Medium | 신규 공고에서 제거, 기존 공고는 Beta 기간 유지 |

**필요 엔티티**: AttributeVisibility(scope: ALL/SELECTED), AttributeVisibilityRole

### PRD #4: 지원자상세, 면접뷰

**핵심**: 정규화된 데이터 기반 view 리뉴얼

| 항목 | 영향도 | 상세 |
|------|--------|------|
| LNB 메뉴 통합 | High | 지원서류/추가정보/사전질문 → 지원정보/제출서류 |
| 지원자 정보 수정 | High | 채용담당자가 ATS에서 직접 수정 + diff 저장 |
| PDF 다운로드 | Medium | 블라인드 반영, 역할 기반 필터링 |
| default 메뉴 설정 | Low | 공고 단위 설정 |

### PRD #5: B2C 지원서 등록/수정

**핵심**: 채용홈페이지 지원서 작성 화면 대응

## 3. Cross-cutting Concerns

### 3.1 데이터 마이그레이션
- 기존 공고의 지원서 설정 → 새 3계층 구조로 변환
- 기존 지원자 데이터 → 정규화된 구조로 변환 (배치)
- 모집분야: 공고 단위 → 워크스페이스 단위

### 3.2 하위 호환성
- OPEN API: 기존 v1 유지 + v2 신규 구조 병행
- 기존 공고: PDF 자동생성 유지, 블라인드 미적용
- 데이터 다운로드: 신/구 공고 모두 지원

### 3.3 성능
- 지원자 목록 조회 시 블라인드 체크 (N개 attribute × M명)
- 엑셀 대용량 다운로드 (비동기 + 이메일)
- 수험번호 동시 발급 (동시성 제어)

### 3.4 보안
- 블라인드 미적용 영역 명확화 (면접요청, 메일/문자 발송)
- 내부 관리 API 접근 제어
