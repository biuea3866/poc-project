# 장애 대응 (Incident Response) 파이프라인

> 프로덕션 장애 발생 시 긴급 원인 분석 + 핫픽스 위치 특정

---

## 파이프라인 흐름

```
[INPUT] 장애 보고 (에러 로그, 스크린샷, 증상)
    │
    ▼
━━ Phase 1: 긴급 분류 (30초 이내) ━━━━━━━━━━━━━━━━━━
    │  팀장이 즉시 판단
    │  - 장애 등급 결정 (P0~P3)
    │  - 영향 서비스 즉시 식별
    │  - 최근 배포 여부 확인 (git log)
    │
    ▼
━━ Phase 2: 동시 긴급 분석 (병렬) ━━━━━━━━━━━━━━━━━━
    │  ┌─ Agent 🔥: 장애 지점 추적 (에러 로그 → 코드)
    │  ├─ Agent 🔍: 최근 변경 분석 (git diff, 배포 이력)
    │  └─ Agent 🛡️: 임시 조치 방안 탐색
    │
    ▼
━━ Phase 3: 원인 특정 + 핫픽스 ━━━━━━━━━━━━━━━━━━━━
    │  팀장이 결과 종합
    │  - 근본 원인 1줄 요약
    │  - 핫픽스 코드 제시 (파일:라인 + 수정 내용)
    │  - 롤백 필요 여부 판단
    │
    ▼
━━ Phase 4: 사후 분석 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    │  - 재발 방지 대책
    │  - 모니터링/알림 추가 필요 항목
    │  - 테스트 추가 필요 항목
    │
    ▼
[OUTPUT] 장애 대응 보고서
```

---

## 장애 등급 분류

| 등급 | 기준 | 대응 속도 | 예시 |
|------|------|----------|------|
| **P0** | 전체 서비스 불가 | 즉시 | API Gateway 다운, DB 접속 불가 |
| **P1** | 핵심 기능 장애 | 15분 내 | 지원 불가, 로그인 불가, 결제 실패 |
| **P2** | 주요 기능 부분 장애 | 1시간 내 | 특정 공고 에러, 엑셀 다운로드 실패 |
| **P3** | 경미한 이슈 | 당일 내 | UI 깨짐, 특정 환경 오류 |

## 에이전트 역할 정의

### 팀장 (메인 Claude)
- 장애 등급 즉시 판단
- 영향 서비스 사전 식별 (아키텍처 보고서 기반)
- 에이전트 결과 종합 및 핫픽스 제시
- 롤백 vs 핫픽스 판단

### Agent 🔥: 장애 지점 추적가
**목표**: 에러 발생 코드 위치를 최단 시간에 특정

**분석 순서**:
1. 에러 메시지/스택트레이스에서 클래스명, 메서드명 추출
2. 해당 코드 파일 직접 탐색 (Grep → Read)
3. 호출 체인 역추적 (Controller → Service → Domain)
4. 에러 조건/분기 로직 확인
5. 핫픽스 가능 지점 특정

**탐색 전략**:
```
에러 로그에서 키워드 추출
    ↓
Grep: 클래스명, 메서드명, 에러 코드
    ↓
Read: 해당 파일의 에러 발생 로직
    ↓
호출자 역추적 (해당 메서드를 호출하는 곳)
    ↓
핫픽스 포인트 특정
```

### Agent 🔍: 최근 변경 분석가
**목표**: 장애 원인이 최근 배포에 있는지 확인

**분석 항목**:
1. 관련 레포의 최근 커밋 (git log --oneline -20)
2. 최근 변경된 파일과 장애 지점 교차 분석
3. 최근 머지된 PR 내용 확인
4. 설정 파일 변경 여부 (application.yml, 환경변수)
5. DB 마이그레이션 최근 실행 여부

**탐색 명령**:
```bash
git log --oneline -20
git diff HEAD~5..HEAD --name-only
git log --since="24 hours ago" --oneline
```

### Agent 🛡️: 임시 조치 탐색가
**목표**: 근본 해결 전 즉시 적용 가능한 임시 조치 탐색

**분석 항목**:
1. 롤백 가능 여부 (이전 버전으로 되돌리기)
2. Feature flag로 기능 비활성화 가능 여부
3. Kafka Consumer 일시 중지로 영향 차단 가능 여부
4. 캐시 무효화로 해결 가능 여부
5. 설정 변경만으로 우회 가능 여부

## 에이전트 할당 규칙

| 장애 유형 | 할당 에이전트 |
|----------|-------------|
| 500 에러 (스택트레이스 있음) | 🔥만 (코드 직접 추적) |
| 배포 직후 장애 | 🔥 + 🔍 (코드 + 최근 변경) |
| 간헐적/재현 어려운 장애 | 🔥 + 🔍 + 🛡️ (전체 동원) |
| 설정/인프라 장애 | 🔍 + 🛡️ (변경 + 임시 조치) |
| 외부 서비스 장애 | 🛡️만 (우회 방안 탐색) |

## 빠른 진단 맵

```
[에러 500] → Gateway 라우팅 확인 → 어떤 서비스? → 해당 서비스 로그/코드
[에러 403] → authz-server 권한 체크 → 역할/플랜 확인
[에러 401] → authn-server 토큰 검증 → JWT 만료/무효
[Timeout]  → 외부 API 호출 확인 → DB 쿼리 확인 → Kafka lag 확인
[데이터 불일치] → Kafka Consumer 확인 → 비동기 처리 실패 여부
[메시지 미발송] → communication → doodlin-communication → 외부 API
```

## 출력 어조

산출물은 **팀원과 공유하는 문서**입니다. 읽는 사람이 지치지 않도록 아래 원칙을 따릅니다.

- **핵심부터** — 결론·액션 아이템을 앞에 씁니다. 이유와 배경은 뒤에 써도 됩니다.
- **짧게** — 한 문장으로 쓸 수 있으면 세 문장으로 쓰지 않습니다.
- **구체적으로** — "여러 곳" 대신 "3곳", "느릴 수 있음" 대신 "users 테이블 full scan" 처럼 씁니다.
- **표·불릿 우선** — 비교·목록은 문장보다 표나 불릿으로 씁니다.
- **중립적으로** — 문제를 발견해도 단정짓지 않고 확인 사항으로 전달합니다.
