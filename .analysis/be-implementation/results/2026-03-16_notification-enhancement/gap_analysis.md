## 1. 애매한 요구사항

| # | 항목 | PRD 내용 | 애매한 부분 | BE 영향 |
|---|------|----------|-------------|---------|
| A-1 | "전체 평가자 평가 완료 시 알림" | P0 | "전체 평가자"의 범위가 불명확. 배정 전원인지, 최소 필수 인원 충족인지 구분 불가 | 완료 판정 로직 설계가 달라짐 |
| A-2 | "개별 평가자 평가 등록 시 알림" | P1 | 알림 수신자가 누구인지 미정의. 채용 담당자만? 같은 면접의 다른 평가자? | 알림 라우팅, 수신자 조회 쿼리 설계 |
| A-3 | "특정 전형 진입 시 알림" | P0 | 누가 설정하는지, 전형별/공고별/워크스페이스별 설정인지 범위 불명 | 설정 도메인 모델 granularity |
| A-4 | "그리팅 내 푸시 알림 On/Off" | P2 | 웹 푸시인지 모바일 푸시인지, In-App 노티피케이션인지 미정의 | 알림 채널 추상화 계층 설계 |
| A-5 | "발송 시점 커스텀" | P1 | 프리셋(n시간 전)인지, 완전 자유 입력인지, 허용 범위가 있는지 | 스케줄러 설계 복잡도 |
| A-6 | "발송 내용 커스텀" | P2 | 템플릿 변수 목록, 에디터 수준(plain text vs HTML), 미리보기 필요 여부 미정의 | 템플릿 엔진 선택, 변수 바인딩 |
| A-7 | "면접 완료 후 n일/n시간 뒤" | P0 | "면접 완료" 시점 정의 없음. 면접 종료 시각? 상태 '완료' 변경 시점? | 트리거 이벤트 정의, 스케줄 등록 시점 |
| A-8 | "On/Off 여부 템플릿 저장" | P0 | 공고별인지, 워크스페이스 전역 기본값인지, 계층 구조인지 불명 | 설정 상속/오버라이드 모델 |

## 2. 누락된 요구사항

| # | 항목 | 필요 이유 | 심각도 |
|---|------|----------|--------|
| M-1 | **알림 채널 정의** | 이메일/인앱/Slack 등 어떤 채널로 발송하는지 정의 없음 | 높음 |
| M-2 | **알림 수신자 매핑 규칙** | 역할(채용담당자/평가자/관리자)별 수신 가능 알림 미정의 | 높음 |
| M-3 | **알림 설정 권한 모델** | 관리자가 멤버 알림을 제어하는 구체적 방식 없음 | 높음 |
| M-4 | **멱등성 및 중복 발송 방지** | 이벤트 재처리 시 중복 알림 방지 전략 없음 | 높음 |
| M-5 | **알림 이력 및 감사 로그** | 발송 이력 조회, 실패 추적 요구사항 없음 | 중간 |
| M-6 | **비정규화 전형 대응 전략** | 25% 그룹핑 전형에서 "전형 진입" 알림 처리 방안 없음 | 높음 |
| M-7 | **스케줄 취소/갱신 시나리오** | 면접 일정 변경/취소 시 예약된 리마인드 처리 방안 없음 | 높음 |
| M-8 | **대량 발송 제한(Rate Limit)** | 동시 다수 알림 트리거 시 처리 전략 없음 | 중간 |
| M-9 | **알림 설정 기본값(Default)** | 신규 공고/워크스페이스 알림 기본값 미정의 | 중간 |
| M-10 | **하위 호환성** | 기존 알림 시스템과 신규 시스템 공존 전략 없음 | 중간 |
| M-11 | **알림 빈도 제한(Throttling)** | 단기간 다수 알림 발생 시 digest 여부 미정의 | 낮음 |
| M-12 | **타임존 처리** | 발송 시점 기준 (서버/사용자/KST) 미정의 | 중간 |

## 3. 추가 고려사항

| # | 영역 | 고려 사항 | 상세 |
|---|------|----------|------|
| T-1 | 동시성 | 전체 평가 완료 판정 Race Condition | 복수 평가자 동시 제출 시 분산 락 or 원자적 카운트 비교 필요 |
| T-2 | 동시성 | 알림 설정 동시 변경 | 관리자/멤버 동시 변경 시 충돌 전략 |
| T-3 | 성능 | 스케줄된 알림 폴링 부하 | 대량 스케줄 조회 시 DB 부하, Job Queue 도입 검토 |
| T-4 | 성능 | 수신자 목록 조회 비용 | 구독 테이블 비정규화 또는 캐시 검토 |
| T-5 | 캐시 | 알림 설정 캐시 전략 | Read-through 캐시 + 변경 시 Invalidation |
| T-6 | 에러 | 메일 발송 실패 재시도 | 재시도 횟수, DLQ 처리, 사용자 피드백 |
| T-7 | 에러 | 스케줄 누락 복구 | 서버 장애 시 과거 스케줄 소급 발송 여부 |
| T-8 | 확장성 | 알림 채널 추상화 | 슬랙/카카오톡/SMS 채널 확장 대비 인터페이스 설계 |
| T-9 | 확장성 | 이벤트 기반 아키텍처 | 도메인 이벤트 발행 → 알림 서비스 구독 구조 |
| T-10 | 정합성 | 템플릿 버전 관리 | 저장 시점 스냅샷 vs 발송 시점 최신 |
| T-11 | 보안 | 알림 내 개인정보 | 지원자 정보 포함 시 개인정보보호법 준수 |
| T-12 | 모니터링 | 알림 발송 메트릭 | 성공/실패율, 지연시간, 큐 적체량 |
