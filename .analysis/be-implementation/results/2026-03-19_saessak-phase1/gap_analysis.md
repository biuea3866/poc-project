| # | PRD | 누락 항목 | 필요 이유 | 영향도 | 제안 |

| M-4 | #5 | 지원서 제출 실패 시 재시도 | 네트워크 오류 등으로 제출 실패 시 멱등성 보장 | High | 멱등키(idempotency key) 기반 중복 제출 방지 |
| M-5 | #5 | 스냅샷 저장 전략 | 지원서 양식 스냅샷의 저장 위치/형식 (JSON? 별도 테이블?) | High | JSON으로 직렬화하여 별도 스냅샷 테이블에 저장 |
| M-6 | #6 | 엑셀 다운로드 대용량 처리 | 수만 명 지원자의 엑셀 다운로드 시 OOM/타임아웃 | High | 스트리밍 방식(SXSSF) + 비동기 처리 + 이메일 전송 |
| M-7 | #7 | 마이그레이션 롤백 전략 | 모집분야 레벨 변경 마이그레이션 실패 시 | High | 배치 단위 + 실패 건 별도 추적 + 수동 보정 도구 |
| M-8 | #1~#6 | 감사 로그 정책 | 지원서 설정 변경, 블라인드 변경, 지원자 정보 수정의 감사 기록 | Medium | 각 변경 이벤트별 활동기록 정의 |
| M-9 | #1 | 필드타입별 유효성 검증 상세 | 서버사이드 validation 규칙 (프론트 검증만으로 불충분) | High | 필드타입별 Validator 인터페이스 설계 |

## 3. 추가 고려사항

| # | 고려사항 | 카테고리 | 상세 | 제안 |
|---|---------|---------|------|------|
| C-1 | 3계층 구조의 확장성 | 아키텍처 | 향후 Phase2에서 항목 추가가 쉬워야 함 | enum 기반이 아닌 테이블 기반 관리 고려 |
| C-2 | 블라인드 체크 성능 | 성능 | 지원자 리스트 조회 시 N명 × M개 attribute 블라인드 체크 | 역할별 열람 가능 attribute 목록을 캐싱 |
| C-3 | 스냅샷 저장 용량 | 인프라 | 지원서 제출마다 양식 스냅샷 저장 시 DB 크기 증가 | 동일 양식 → 동일 스냅샷 키 참조 (중복 제거) |
| C-4 | 국제화 (다국어) | 기능 | 항목명의 한/영 지원, 채용홈페이지 영문 | 모든 텍스트 필드에 ko/en 다국어 구조 |
| C-5 | 동시 편집 | 동시성 | 지원서 설정을 여러 관리자가 동시에 편집 | Optimistic Lock (version 필드) |
| C-6 | 커스텀 항목 제한 | 비즈니스 | 커스텀 항목 무한 생성 방지 | 플랜별 최대 개수 제한 |
| C-7 | 데이터 정합성 | 트랜잭션 | 지원서 설정 변경 + 블라인드 설정이 하나의 트랜잭션인지 | 각각 독립 트랜잭션 (eventual consistency) |
| C-8 | 검색 인덱스 업데이트 | 검색 | 지원자 정보 변경 시 검색 DB 반영 타이밍 | 이벤트 기반 비동기 인덱싱 (Kafka) |
