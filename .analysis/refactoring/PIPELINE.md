## 일반적인 리팩토링/마이그레이션 시나리오

### Scenario 1: 라이브러리 버전 업그레이드
```
예: React Query v3 → v5, React 18 → 19, Spring Boot 2.x → 3.x

분석 필요:
1. Breaking Changes 목록 확인
2. 현재 사용 패턴 중 영향 받는 것
3. 수정 필요 파일/코드 전수 조사
4. 단계별 마이그레이션 순서
```

### Scenario 2: 패턴/아키텍처 변경
```
예: Recoil → Zustand, Pages Router → App Router, Layered → Hexagonal

분석 필요:
1. 현재 패턴 사용처 전수 조사
2. 새 패턴으로의 변환 규칙 정의
3. 공존 기간 전략 (점진적 마이그레이션)
4. 완료 기준 정의
```

### Scenario 3: 공유 라이브러리 메이저 업데이트
```
예: doodlin-ui 0.x → 1.0, greet-components 0.x → 1.x

분석 필요:
1. 변경된 API/Props 목록
2. 모든 소비 레포에서의 사용 현황
3. 레포별 수정 필요 코드
4. 배포 순서 (라이브러리 먼저 → 소비자 순차)
```

### Scenario 4: 서비스 분리/통합
```
예: greeting-new-back에서 도메인 분리, 모노레포 통합

분석 필요:
1. 분리/통합 대상 코드 범위
2. 서비스 간 인터페이스 정의
3. DB 분리 필요 여부
4. Kafka 토픽 변경 필요 여부
```

## 에이전트 역할 정의

### Agent 🔬: 현재 사용 현황 분석가
**목표**: 변경 대상의 현재 사용 현황 완전 파악

**분석 항목**:
1. 변경 대상 라이브러리/패턴의 모든 사용처 검색
2. 사용 패턴 분류 (표준 사용 / 비표준 사용 / 커스텀)
3. 레포별 사용량 통계
4. 가장 복잡한 사용 사례 식별

**탐색 전략**:
```
Grep: import/require 문으로 사용처 전수 검색
    ↓
패턴 분류: 표준 vs 비표준 사용
    ↓
레포별/파일별 수정 난이도 평가
    ↓
우선순위 결정 (쉬운 것부터 or 영향 큰 것부터)
```

### Agent 🗺️: 의존성 그래프 분석가
**목표**: 변경 시 연쇄적으로 영향 받는 의존성 체인 파악

**분석 항목**:
1. 직접 의존성 (이 라이브러리를 직접 import)
2. 간접 의존성 (직접 의존하는 코드를 다시 import)
3. 순환 의존성 여부
4. 변경 전파 경로 (A → B → C)

**탐색 전략**:
```
대상 모듈의 export 목록
    ↓
각 export를 import하는 파일 검색
    ↓
그 파일들이 다시 export하는 것 확인
    ↓
전체 의존성 트리 구성
```

## 마이그레이션 계획 템플릿

```
Phase 0: 준비
  □ Breaking Changes 목록 정리
  □ 영향 범위 확정
  □ 테스트 환경 준비

Phase 1: 공존 준비
  □ 새 버전/패턴 도입 (기존과 공존)
  □ 어댑터/호환 레이어 추가 (필요 시)

Phase 2: 점진적 마이그레이션
  □ 가장 단순한 사용처부터 변환
  □ 각 변환 후 테스트
  □ 중간 배포 (공존 상태)

Phase 3: 완료
  □ 모든 사용처 변환 완료
  □ 구버전/구패턴 제거
  □ 호환 레이어 제거
  □ 최종 테스트

Phase 4: 정리
  □ 문서 업데이트
  □ 팀 공유
```

## 출력 어조

산출물은 **팀원과 공유하는 문서**입니다. 읽는 사람이 지치지 않도록 아래 원칙을 따릅니다.

- **핵심부터** — 결론·액션 아이템을 앞에 씁니다. 이유와 배경은 뒤에 써도 됩니다.
- **짧게** — 한 문장으로 쓸 수 있으면 세 문장으로 쓰지 않습니다.
- **구체적으로** — "여러 곳" 대신 "3곳", "느릴 수 있음" 대신 "users 테이블 full scan" 처럼 씁니다.
- **표·불릿 우선** — 비교·목록은 문장보다 표나 불릿으로 씁니다.
- **중립적으로** — 문제를 발견해도 단정짓지 않고 확인 사항으로 전달합니다.
