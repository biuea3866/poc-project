# 기술 설계 문서 템플릿

## 개요

- 이 문서가 다루는 기술 주제를 요약합니다.
- 어떤 구현 결정 또는 기술 선택을 다루는지 적습니다.

## 목표

- 이 문서로 해결하려는 문제를 적습니다.
- 기대하는 결과와 비목표를 함께 적습니다.

## 설계

### 데이터 흐름

- 요청/이벤트/데이터가 어떻게 흐르는지 설명합니다.
- 반드시 시각 자료를 포함합니다.
- 권장 형식: `draw.io` 원본 또는 `svg/png` 이미지
- 예시:

```md
![데이터 흐름 다이어그램](./pinpoint-data-flow.svg)
```

### ERD

- 필요한 엔티티와 관계를 정리합니다.
- DB 변경이 없다면 해당 없음으로 명시합니다.
- DB 변경이 없더라도 "변경 없음"을 시각적으로 표현한 다이어그램을 넣습니다.

### 컴포넌트 다이어그램

- 모듈, 컴포넌트, 외부 시스템 간 경계를 설명합니다.
- 서비스 경계와 연동 포인트가 한눈에 보이도록 시각 자료를 포함합니다.

### 플로우 다이어그램

- 주요 시나리오의 처리 순서를 단계별로 설명합니다.
- 정상 흐름과 실패 시점이 보이도록 시각 자료를 포함합니다.

## 다이어그램 규칙

- `데이터 흐름`, `ERD`, `컴포넌트 다이어그램`, `플로우 다이어그램`은 텍스트만으로 끝내지 않습니다.
- 각 섹션마다 최소 1개의 시각 자료를 포함합니다.
- 가급적 문서와 함께 원본 편집 파일을 남깁니다.
  - `draw.io`를 쓰면 `.drawio`
  - 이미지를 쓰면 `.svg` 우선
- Confluence에 올릴 때도 동일한 시각 자료를 함께 첨부하거나 본문에 포함합니다.

## 결정 사항

- 어떤 선택을 했는지와 이유를 적습니다.

## 트레이드오프

- 대안과 선택하지 않은 이유를 적습니다.

## 미결 사항

- 아직 결정되지 않은 항목과 확인이 필요한 내용을 적습니다.
